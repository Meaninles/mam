# 统一文件管理系统标签管理整体方案

## 文档信息

- 文档名称：统一文件管理系统标签管理整体方案
- 文档用途：用于标签管理功能的业务规划、技术架构设计、实现拆解与后续演进对齐
- 当前阶段：MVP / V1 方案设计
- 适用范围：Windows 桌面端当前前端原型与本地 SQLite mock 实现

## 1. 背景与目标

### 1.1 背景

当前系统已经具备基础的资产标记能力：

- 文件中心支持单资产标签编辑。
- 标签以 `entry_tags` 中 `kind = 'tag'` 的自由文本形式存储。
- 文件中心列表、详情和标签建议已经可以展示与使用标签。

这套实现足以支撑“给文件打标签”，但不足以支撑“标签管理”：

- 标签不是独立业务对象，无法稳定承载改名、合并、分组、置顶和排序。
- 标签与资产库的适用范围没有建模。
- 标签没有正式进入检索体系，只是局部建议词。
- 文件中心中的标签输入仍以自由文本为主，容易产生同义词、脏数据和命名漂移。

因此，本次不是补一个小弹窗，而是补齐“标签域”的业务与技术基础设施。

### 1.2 目标

本方案目标是在不改变当前“前端 + sql.js + localStorage mock”验证模式的前提下，建立一套可长期演进的标签管理架构，使标签成为统一文件管理系统中正式的组织与检索能力之一。

### 1.3 本次明确范围

- 提供独立标签库管理。
- 支持标签新增、改名、合并、删除、排序、置顶、分组。
- 支持标签配置可作用的资产库范围。
- 在设置页中新增独立“标签管理”子页面。
- 保持文件中心中的资产打标签入口，但改为基于标签库选择。
- 为未来模糊检索接入标签命中能力预留架构。

### 1.4 本次明确不做

- 不做标签颜色，避免与文件中心色标混淆。
- 不做标签筛选器，不在文件中心增加“按标签筛选”。
- 不做复杂权限模型、多用户协同和审计审批。
- 不做完整后端化改造，当前仍以本地 SQLite mock 为准。

## 2. 需求结论

### 2.1 业务结论

- 标签库全系统共用。
- 单个标签可以配置作用于 0 个、1 个或多个资产库。
- 标签作用域影响“可被选择与可被推荐”，不改变标签本身的全局身份。
- 标签将作为未来检索体系的重要组成部分，但本期不扩展为筛选项。
- 设置页是标签管理的正式入口；文件中心是标签使用入口。

### 2.2 架构结论

- 标签必须从“自由文本字段”升级为“独立领域实体”。
- 资产与标签之间必须通过稳定主键关联，不能继续仅靠标签名进行业务操作。
- 现有 `entry_tags` 不再承担业务标签主存储职责，应收缩为非业务标签用途，如 `badge`、`risk`。
- 标签域要显式建模分组、作用域、排序与引用关系。

## 3. 设计原则

- 标签优先实体化：标签是正式业务对象，不再只是字符串。
- 全局唯一身份：标签的改名、合并、删除必须以标签主键为中心执行。
- 使用与管理分离：设置页负责管理标签库，文件中心负责消费标签库。
- 作用域限制选择，不回写历史：标签作用域主要影响后续可选范围，不默认回收历史关联。
- 本期简单、未来可演进：当前为单机 mock 设计，但模型需兼容未来服务化。
- 检索优先于筛选：先把标签接入搜索与召回，再考虑复杂筛选面板。
- 降低脏数据：尽量减少自由输入、同义词扩散和重复标签。

## 4. 标签域业务定位

### 4.1 在系统中的角色

标签域是统一文件资产体系中的“人工组织层”。它不替代目录、不替代星级、不替代色标，而是承担以下职责：

- 作为跨目录、跨端点的语义组织工具。
- 作为未来模糊检索、推荐检索的重要召回字段。
- 作为批量整理、人工归类和资产治理的辅助结构。

### 4.2 与其他组织方式的边界

- 目录：表达文件在资产库中的结构位置。
- 标签：表达用户赋予文件的业务语义和组织语义。
- 星级：表达优先级或主观质量判断。
- 色标：表达即时状态标记或轻量工作流状态。
- 风险标签 / 徽标：表达系统生成的状态与提醒，不属于业务标签库。

### 4.3 典型应用场景

- 摄影项目中标记“客户已选”“待精修”“封面候选”“品牌主视觉”。
- 视频项目中标记“口播素材”“B-roll”“高优先级返修”。
- 跨资产库建立共享语义标签，如“年度精选”“宣传素材”“合同归档关联”。
- 在未来搜索中通过标签召回并定位文件，而不依赖路径记忆。

## 5. 业务对象与关系

### 5.1 核心业务对象

#### 5.1.1 标签组

标签组用于对标签库进行人工管理分区，主要服务于管理页展示与标签选择体验。

职责：

- 承载标签集合的业务分类。
- 提供稳定排序。
- 为未来标签模板、默认展开策略预留结构。

建议默认存在一个系统保留组：`未分组`。

#### 5.1.2 标签

标签是正式业务对象，具备稳定身份和可配置属性。

职责：

- 保存标签名称与规范化名称。
- 保存所在分组。
- 保存排序与置顶信息。
- 保存适用资产库范围。
- 保存使用统计与变更语义。

#### 5.1.3 标签作用域

标签作用域表示某个标签可作用于哪些资产库。

语义：

- 0 个资产库：标签存在但当前不可在任何资产库中新选。
- 1 个资产库：标签仅在该资产库可选。
- 多个资产库：标签在指定资产库范围内可选。

#### 5.1.4 资产标签关联

资产标签关联表示某个统一文件资产绑定了哪些标签。

职责：

- 记录稳定的标签引用。
- 记录展示顺序。
- 支撑未来搜索召回与统计分析。

### 5.2 关键关系

```text
标签组 1 ---- N 标签
标签 N ---- N 资产库
标签 N ---- N 文件资产
```

### 5.3 关系解释

- 标签属于一个标签组。
- 标签可作用于多个资产库；一个资产库也可允许多个标签。
- 一个文件资产可绑定多个标签；同一标签可被多个文件资产复用。

## 6. 核心业务规则

### 6.1 标签命名规则

- 标签名称在全局范围内唯一。
- 名称去首尾空格后参与唯一性判定。
- 建议增加规范化字段用于大小写统一、全角半角统一和空白折叠。
- 改名不能与现有规范化标签名冲突。

### 6.2 合并规则

- 合并以“目标标签”为主身份。
- 源标签的所有资产关联、统计引用、作用域配置迁移到目标标签。
- 若迁移后出现重复资产标签关联，应自动去重。
- 合并后源标签应被删除或转入历史状态。

本期建议直接执行“迁移并删除源标签”，不保留复杂历史链路。

### 6.3 删除规则

- 删除标签会解除该标签与所有资产的关联。
- 删除前必须展示影响范围，如命中资产数、涉及资产库数。
- 若标签正在被大量使用，应优先引导用户使用“合并”而不是直接删除。

### 6.4 作用域规则

- 作用域只限制“未来可选范围”和“推荐范围”。
- 已绑定资产若因作用域调整而超出范围，本期不自动移除，避免破坏历史整理结果。
- 对超出范围的历史绑定，后续可在管理页提供“超范围引用”提示。

### 6.5 排序与置顶规则

- 标签组之间按组排序展示。
- 组内先按置顶状态排序，再按手动排序字段排序。
- 同组内排序应稳定，避免由名称排序导致用户手工顺序失效。

### 6.6 文件中心使用规则

- 文件中心标签选择器默认只展示当前资产库可用的标签。
- 文件中心支持按名称搜索已有标签。
- 文件中心可保留“快速新建标签”，但新建后应立即写入标签库，而不是只写资产关联。
- 文件中心本期不支持按标签筛选。

### 6.7 搜索接入规则

- 标签应作为未来统一搜索索引的一部分。
- 搜索命中标签后，返回对应资产。
- 本期不需要独立搜索页，但数据层要支持基于标签名命中的资产召回。

## 7. 信息架构与页面规划

### 7.1 导航位置

在设置页 `Settings` 中新增子页面：

- 通用
- 文件总览
- 标签管理
- 校验恢复
- 性能
- 外观

“标签管理”位于“文件总览”之后、“校验恢复”之前。

### 7.2 页面角色划分

#### 7.2.1 设置页中的标签管理子页面

职责：

- 管理标签组。
- 管理标签主数据。
- 管理作用域。
- 管理合并、删除等危险操作。
- 展示标签使用情况和影响范围。

#### 7.2.2 文件中心标签选择弹层

职责：

- 搜索已有标签。
- 选择/取消当前资产标签。
- 可选快速新建标签。
- 展示标签所属组、适用资产库提示、置顶优先展示。

### 7.3 标签管理页建议结构

页面建议拆为三栏或双栏工作区：

- 左侧：标签组列表与组排序。
- 中间：当前组标签列表，支持搜索、置顶、拖拽排序、批量选择。
- 右侧：标签详情面板，配置名称、作用资产库、使用统计、危险操作。

如果当前原型复杂度控制优先，可先落双栏：

- 左栏：分组 + 标签列表。
- 右栏：详情与操作。

### 7.4 标签管理页核心功能块

- 标签总览区：总标签数、使用中标签数、未分组标签数、跨库标签数。
- 分组管理区：新增组、改名、排序。
- 标签列表区：搜索、置顶、排序、重命名入口。
- 标签详情区：名称、所属分组、作用资产库、使用统计。
- 危险操作区：合并标签、删除标签。

### 7.5 文件中心标签选择体验建议

- 顶部先展示已选标签。
- 其次展示“置顶标签”和“当前资产库常用标签”。
- 搜索结果按“名称匹配优先、当前库可用优先、置顶优先”排序。
- 如果某标签存在但当前库不可用，不应出现在默认可选列表。

## 8. 业务流程设计

### 8.1 新增标签流程

```text
用户创建标签
-> 校验名称合法性与唯一性
-> 选择分组
-> 配置作用资产库
-> 写入标签主表
-> 刷新管理页与文件中心标签源
```

### 8.2 文件打标签流程

```text
用户在文件中心打开标签编辑
-> 加载当前资产库可用标签
-> 搜索/选择标签
-> 提交资产标签关联
-> 更新资产标签展示
-> 更新标签使用统计
```

### 8.3 标签改名流程

```text
用户发起改名
-> 校验重名
-> 更新标签主记录
-> 所有关联资产自动感知新名称
-> 搜索索引命中内容同步更新
```

### 8.4 标签合并流程

```text
用户选择源标签与目标标签
-> 预估影响资产数量
-> 确认合并
-> 将源标签关联迁移到目标标签
-> 去重重复关联
-> 合并作用域与统计
-> 删除源标签
```

### 8.5 标签删除流程

```text
用户发起删除
-> 展示影响范围
-> 用户确认
-> 删除资产关联
-> 删除标签主记录与作用域记录
-> 刷新管理页与文件中心
```

## 9. 技术架构设计

### 9.1 当前阶段总体方案

当前标签管理功能继续落在前端本地 mock 架构中：

- UI 层：React 页面与弹层。
- 应用层：`App.tsx` 负责页面路由与状态编排。
- 领域访问层：`fileCenterApi.ts` 作为标签域与文件中心的本地数据服务。
- 存储层：`sql.js` SQLite + `localStorage` 持久化。

### 9.2 建议的前端分层

#### 9.2.1 页面层

负责页面组织和用户交互。

建议文件：

- `client/src/pages/TagManagementPage.tsx`
- `client/src/pages/SettingsPage.tsx`
- `client/src/App.tsx`

#### 9.2.2 领域服务层

负责标签域数据读写、规则收口和 mock 查询封装。

建议文件：

- `client/src/lib/fileCenterApi.ts`
- 可选拆分：`client/src/lib/tagManagementApi.ts`

本期为了降低改动成本，可以继续放在 `fileCenterApi.ts`，但内部应按标签域函数分段组织。后续若标签逻辑显著增大，再拆为独立 `tagManagementApi.ts`。

#### 9.2.3 共享组件层

负责管理页表单、确认弹窗、标签选择组件等复用 UI。

建议文件：

- `client/src/components/Shared.tsx`
- 可选新增：`client/src/components/TagScopeSelector.tsx`
- 可选新增：`client/src/components/TagPickerDialog.tsx`

### 9.3 当前代码结构下的模块职责调整

#### 9.3.1 `App.tsx`

新增职责：

- 管理设置页 `tag-management` 子页切换。
- 管理标签管理页所需的局部状态与反馈。
- 协调文件中心标签编辑与标签库数据刷新。

应避免新增职责：

- 不要把标签领域规则继续堆在 `App.tsx`。
- 不要让 `App.tsx` 直接操作 SQL 细节。

#### 9.3.2 `fileCenterApi.ts`

新增职责：

- 提供标签组、标签、作用域、资产标签关联的完整读写能力。
- 提供文件中心标签选择器的数据装配。
- 提供未来基于标签的搜索命中入口。

保留职责：

- 继续管理文件中心 mock 数据。

应收缩职责：

- 不再用 `entry_tags.kind='tag'` 作为业务标签主存储。

#### 9.3.3 `data.ts` 与 `clientState.ts`

新增职责：

- 增加设置页 `SettingsTab` 枚举项。
- 更新设置页标签顺序。
- 确保设置内容克隆函数兼容新子页。

## 10. 数据模型设计

### 10.1 本期推荐表结构

#### 10.1.1 `tag_groups`

- `id`
- `name`
- `order_index`
- `created_at`
- `updated_at`

#### 10.1.2 `tags`

- `id`
- `name`
- `normalized_name`
- `group_id`
- `order_index`
- `is_pinned`
- `usage_count`
- `created_at`
- `updated_at`

#### 10.1.3 `tag_library_scopes`

- `tag_id`
- `library_id`

#### 10.1.4 `entry_tag_links`

- `entry_id`
- `tag_id`
- `order_index`
- `created_at`

### 10.2 与现有结构的关系

#### 10.2.1 保留

- `entries`
- `entry_endpoints`
- `entry_metadata`
- `entry_tags` 中 `badge` 与 `risk` 用途

#### 10.2.2 收缩

- `entry_tags.kind='tag'` 不再作为正式业务标签表。

#### 10.2.3 新的标签读取方式

文件中心在装配 `FileCenterEntry.tags` 时，应通过：

- `entry_tag_links`
- `tags`

进行关联查询，而不是从 `entry_tags` 直接取自由文本。

### 10.3 规范化字段建议

`normalized_name` 建议至少处理：

- 去首尾空格
- 多空格压缩为单空格
- 英文大小写归一
- 可选的全角半角归一

这样可以在不改变显示名的前提下，提高重名判定稳定性。

### 10.4 统计字段建议

`usage_count` 本期可作为冗余统计字段维护，也可每次聚合查询。

建议：

- 本期允许通过聚合查询得到真实值。
- 若页面性能或复杂度要求提升，再引入稳定的冗余统计更新机制。

## 11. 检索架构预留

### 11.1 本期必须

- 标签具备独立实体与稳定主键。
- 标签名可被搜索。
- 资产可通过标签关联被召回。

### 11.2 本期可选

- 将文件中心现有搜索框直接接入标签命中。

如果本期接入，建议在当前 `buildEntryFilter` 中增加：

- 名称命中
- 路径命中
- 来源命中
- 标签命中

其中标签命中通过 `EXISTS` 子查询实现，而不是将标签反写到资产主表字符串字段。

### 11.3 为未来预留但暂不实现

- 标签别名。
- 标签同义词。
- 标签搜索权重。
- 搜索历史与搜索建议。
- 跨标签组合查询。
- 标签模板与自动推荐。

## 12. 状态与约束

### 12.1 操作约束

当前为单机场景，默认不存在多用户并发写入，但仍需保证单次操作的一致性：

- 合并标签时应放在单次事务中完成。
- 删除标签时应保证“先处理关联，后删主记录”。
- 调整排序时应保证组内顺序连续。

### 12.2 数据一致性约束

- `tags.normalized_name` 全局唯一。
- `entry_tag_links(entry_id, tag_id)` 唯一。
- `tag_library_scopes(tag_id, library_id)` 唯一。
- 标签删除后不能残留悬空关联。

### 12.3 交互约束

- 危险操作必须二次确认。
- 合并操作必须显式选择目标标签。
- 删除操作必须显示影响资产数量。
- 不允许在文件中心直接编辑组、排序或作用域。

## 15. 风险与取舍

### 15.1 核心风险

- 自由文本迁移到实体标签后，旧数据兼容与展示可能出错。
- 合并与删除容易误伤资产标签关联。
- 作用域规则若处理不清，会让用户误以为“限制作用域会删除历史标签”。
- 当前所有逻辑继续堆在 `fileCenterApi.ts`，后续会快速膨胀。

### 15.2 主要取舍

#### 取舍一：本期不做颜色

优点：

- 避免与现有色标概念冲突。
- 降低用户认知成本。

代价：

- 标签视觉识别力较弱。

结论：

- 本期不做颜色是正确取舍。

#### 取舍二：本期不做标签筛选

优点：

- 避免在文件中心筛选面板中引入复杂度。
- 先把标签域模型做正确。

代价：

- 标签的用户价值暂时更多体现在组织和未来检索，而非即时筛选。

结论：

- 先接搜索，不接筛选，符合当前阶段。

#### 取舍三：继续沿用本地 SQLite mock

优点：

- 与当前项目现状一致，能快速验证交互与模型。
- 避免在功能尚未稳定时提前服务化。

代价：

- API 形态、事务模型和真正服务层边界暂时不完整。

结论：

- 本期继续 mock，但接口命名和读写模型应尽量贴近未来服务化。

#### 取舍四：标签域是否单独拆 API 文件

优点：

- 单独拆分更清晰。

代价：

- 当前项目规模下会增加额外接线成本。

结论：

- 本期可以先不拆文件，但必须按标签域分段组织函数；达到一定体量后再抽离。

## 16. 未来服务化演进建议

### 16.1 目标演进方向

未来进入真正后端化后，标签域建议形成独立服务边界或至少独立模块：

- 标签主数据管理
- 标签作用域管理
- 资产标签关联管理
- 标签搜索与推荐

### 16.2 推荐服务接口方向

可演进为如下资源模型：

- `GET /tags`
- `POST /tags`
- `PATCH /tags/:id`
- `POST /tags/:id/merge`
- `DELETE /tags/:id`
- `GET /tag-groups`
- `POST /tag-groups`
- `PATCH /entries/:id/tags`

### 16.3 服务化时的额外建议

- 使用显式事务处理合并与删除。
- 将标签统计与搜索索引更新解耦为异步任务。
- 为标签搜索引入 `normalized_name`、别名与权重字段。
- 视规模考虑引入专门搜索索引或 PostgreSQL 模糊检索增强。

## 18. 架构决策摘要

1. 标签升级为独立领域实体，而不是继续沿用自由文本标签。
原因：只有实体化才能稳定支持改名、合并、作用域和检索。

2. 标签库全局共享，作用域单独建模为“标签可作用资产库”关系。
原因：共享身份与局部适用性是两个不同维度，不能混在一起。

3. 设置页负责标签管理，文件中心负责标签消费。
原因：管理与使用分层清晰，避免文件中心承载过重配置职责。

4. 本期不做标签颜色。
原因：与色标概念冲突，收益低于认知成本。

5. 本期不做标签筛选，但必须为搜索接入预留模型。
原因：先保障标签进入检索体系，再扩展复杂筛选。

6. `entry_tags` 继续保留给 `badge`、`risk`，业务标签迁移到正式关联表。
原因：不同语义的数据不能长期混存于同一弱结构表。

7. 文件中心标签编辑器改为“从标签库选择”，允许有限快速新建。
原因：既控制脏数据，又保留整理效率。

8. 本期继续使用本地 SQLite mock，但接口设计要贴近未来服务层。
原因：当前阶段以快速验证为主，同时避免未来重构成本过高。
