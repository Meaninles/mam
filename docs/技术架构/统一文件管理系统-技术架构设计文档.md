# 统一文件管理系统技术架构设计文档

## 文档信息

- 文档名称：统一文件管理系统技术架构设计文档
- 文档用途：用于后续详细设计、数据库设计、接口设计、模块拆分、开发实施与 AI 协同开发对齐
- 当前阶段：V1 技术架构基线
- 首发平台：Windows 桌面端
- 目标形态：桌面客户端 + 本地常驻核心服务

## 1. 文档目标与范围

本文档基于已确认的产品功能规划，给出本项目的首版技术架构方案，重点解决以下问题：

- 如何围绕“统一文件资产表”设计核心领域模型
- 如何支撑本地、移动盘、SMB/NAS、网盘等多端文件统一管理
- 如何落地导入、复制、同步、扫描、校验、删除等核心任务
- 如何实现统一任务体验与断点续传
- 如何设计适合大规模文件管理的数据存储、索引与搜索能力
- 如何在首版先做本地部署，同时为后续迁移到 NAS 保留演进空间

本文档为架构设计文档，不展开到所有接口字段和数据库 DDL 级别，但会明确实现边界、模块职责、核心状态机与关键技术决策。

## 2. 架构设计前提

以下前提为当前版本已确认的业务和架构约束：

- 产品以“文件资产”为核心，而不是以存储端点为核心。
- 单个资产库中，同一文件资产只保留一条统一资产主记录。
- 同文件判断规则固定，不允许用户自定义规则。
- 同文件判定采用固定语义：相对路径 + 文件大小 + 修改时间 + 时间容差。
- 路径不同即视为不同资产，即使内容相同也不自动归并。
- 导入、复制、同步是三种不同业务任务，不合并。
- 删除采用逻辑删除加后台清理模型；当所有副本都不存在时，资产进入逻辑删除状态，等待后台清理。
- 扫描是统一资产表的写入入口之一，但不是唯一入口。
- 扫描结果必须完整成功后才能提交正式状态，不允许边扫边提交正式状态。
- 媒体文件需要增强元数据管理，但普通文件也必须完整支持基础管理。
- 首版部署在本地 Windows 电脑，核心服务、CloudDrive2 与 aria2 均部署在本机。
- 网盘相关上传任务通过 CloudDrive2 API 集成，不能仅通过 SMB 黑盒方式集成。
- 下载能力统一通过 aria2 集成实现，复用 aria2 的断点下载、暂停、恢复、取消能力。

## 3. 总体架构概览

### 3.1 总体分层

系统采用四层架构：

1. 客户端表示层
2. 本地核心服务层
3. 存储端适配与执行层
4. 本地数据与索引层

### 3.2 架构拓扑

```text
+--------------------------------------------------------------+
| Tauri 桌面客户端                                             |
| - 文件中心 / 任务中心 / 异常中心 / 存储节点 / 设置（顶层多标签）|
| - 面包屑导航 / 搜索筛选 / 标签 / 元数据展示 / 操作入口       |
+-----------------------------+--------------------------------+
                              |

                              v
+--------------------------------------------------------------+
| Go 本地核心服务                                               |
| - 资产库管理                                                  |
| - 存储端管理                                                  |
| - 统一资产服务                                                |
| - 目录服务                                                    |
| - 扫描与对账服务                                              |
| - 任务调度与执行编排                                          |
| - 导入 / 复制 / 同步 / 下载 / 校验 / 删除                     |

         +------------------+   +----------------------------+
```

### 3.3 架构原则

- 单一权威：核心服务是唯一权威状态源。
- 统一账本：统一资产表是业务主账本。
- 多端展开：多端状态通过副本表显式展开，不塞 JSON 黑盒。
- 任务驱动：所有重操作都以任务形式执行并持久化。
- 正式状态与过程状态分离：任务过程不直接污染正式资产状态。
- 先可用后增强：基础文件管理和多端同步优先，复杂智能能力逐步扩展。
- 统一体验、分层实现：业务任务分开，底层执行器可复用。

## 4. 技术栈选型

### 4.1 客户端

- 桌面框架：`Tauri`
- 前端职责：
  - 页面与交互
  - 文件中心列表、目录导航、筛选、详情页
  - 任务中心、异常中心、设置页
  - 只调用核心服务，不直接操作数据库或存储端

#### 当前客户端原型实现

- 当前客户端已在页头中部落地“顶层多标签工作区”，承载 `文件中心 / 任务中心 / 异常中心 / 存储节点 / 设置` 五个一级页面并行切换。
- 顶层标签遵循“单页面单实例”：左侧导航点击时，已打开则聚焦，未打开则新开并插入当前激活标签右侧。
- 导入中心仍为页头右侧独立入口，不进入顶层标签体系；设备插入绿色标签与通知中心 `Sheet` 同样属于页头信号层，不进入顶层标签集合。
- 顶层标签关闭通过右键菜单执行，不使用可见 `X` 按钮；排序支持右键“向左/向右移动”以及直接拖拽整个标签框体。
- 最近关闭标签栈当前保留 8 条记录；关闭后无标签时回到文件中心。
- 顶层多标签为纯客户端能力，不与后端耦合，也不要求重启恢复标签。
- 当前保活策略为“会话内已访问一级页面持续挂载”，上限受一级页面总数约束（当前 5 个）；非激活页通过脱离主舞台容器保留挂载与上下文。
- 当前客户端“文件中心”已经在前端实现目录加载、面包屑导航、前进 / 后退、搜索、筛选、排序、分页、详情抽屉、标签编辑、批量同步、批量删除和新建目录。
- 当前客户端同时已在设置页实现独立“标签管理”子页面，页面包含顶部统计小徽标、分组侧栏、标签列表、详情面板，以及合并 / 删除确认弹层。
- 标签管理页当前使用客户端内 `sql.js` SQLite mock 直接驱动，已引入 `tag_groups`、`tags`、`tag_library_scopes`、`entry_tag_links` 等结构用于分组、作用域、排序与资产标签关联。
- 当前页面中的排序交互采用“上移 / 下移”按钮，不是拖拽排序；页面也尚未实现批量选择与批量操作。
- 当前原型尚未接入 Go 核心服务；文件中心数据由客户端内 `sql.js` 本地 SQLite mock 提供，并持久化到 `localStorage`。
- 当前任务中心中的“传输任务”原型已将任务模型与文件数据关联：一级任务使用 `fileNodeIds`，二级任务使用 `fileNodeId`；名称、路径、大小优先从 `fileNodes` 实时解析，导入或未入库场景再兜底旧字段。
- “其它任务”当前仅纳入 `SCAN / METADATA_EXTRACT / VERIFY / DELETE_CLEANUP` 四类页面任务类型；`REPAIR` 等历史占位任务不会在其它任务子页面展示。
- “其它任务”当前采用单层任务列表，不在主列表中区分一二级任务；对象级文件、目录、资产与步骤信息统一在详情抽屉和异常联动中承接。
- “其它任务”工具条当前提供任务类型筛选、状态筛选、排序与搜索，不提供独立资产库筛选，因为大量后台任务并不天然绑定单一资产库。
- 删除清理任务当前在 `运行中` 与 `等待清理` 两类状态下都允许暂停；暂停后状态切换为 `已暂停`，并在界面上显示“继续”。
- 删除文件前会先检查关联的运行中传输任务；若存在冲突，当前原型会提示，并支持跳转查看任务或取消任务后再删除。
- 客户端启动时会用最新任务 mock 覆盖任务相关持久化状态，保证任务中心展示与当前原型数据一致。
- 任务状态筛选已新增“活跃中”默认项。
- 任务行动作遵循互斥规则：暂停 / 继续 / 重试三者互斥渲染；已完成任务显示“完成”占位按钮。
- 已删除关联文件的任务子项会进入失效展示态，表现为文件已删除、删除线样式且整行不可点击。
- 删除清理任务当前明确展示“等待清理 / 清理中”阶段，并允许在 `运行中` 与 `等待清理` 状态下执行暂停，暂停后显示“继续”。
- 当前客户端“异常中心”已完成正式工作台化改造，具备一级分类、问题性质、来源域、资产库、状态、严重级别、搜索、排序等筛选能力，并支持详情抽屉、批量治理与历史清理入口。
- 异常中心当前使用稳定的前端异常模型：`IssueCategory / IssueNature / IssueSourceDomain / IssueStatus / IssueSource / IssueCapabilities / IssueHistoryRecord / IssueImpactSummary` 已在客户端数据层落地。
- 异常中心与任务中心、存储节点之间已形成稳定跳转契约，聚焦参数已覆盖 `issueId / taskId / taskItemId / libraryId / endpointId / fileNodeId / path` 等键。
- 当前客户端“通知中心”已落地为页头右侧 `Sheet`，与导入中心共同形成页头右侧入口层，不进入顶层多标签体系。
- 页头信号已分层：设备插入 `HeaderSignal`（绿色标签）独立于通知流，不计入铃铛角标；通知铃铛数字角标仅统计未消费通知。
- 通知模型已在前端稳定落地：`HeaderSignal / NoticeRecord / NoticeSource / NoticeCapabilities / NoticeJumpParams`。
- 通知消费规则已在客户端实现：打开通知中心自动消费提醒类；处置类仅在手动已读或点击跳转后退出角标统计。
- 通知中心跳转当前复用异常中心既有跨页定位键集合（`issueId / taskId / taskItemId / libraryId / endpointId / fileNodeId / path`）；当请求中携带 `issueId` 时，当前客户端会自动聚焦对应异常并打开详情抽屉，缺少 `issueId` 时则保持列表聚焦上下文。
- 当前通知 mock 数据已覆盖设备插入、提醒类、处置类、已读、已跳转、已失效等组合，服务化阶段可直接替换数据源与命令实现。
- 异常处理当前仍为客户端 mock 状态变更，但 UI/UX 及能力边界已定型，后续服务化时应仅替换查询与命令实现，不重构页面结构。
- 因此，本章描述的是目标架构基线；客户端现状属于目标架构落地前的交互验证实现。
- 与标签管理相关的检索接入、文件中心改造成纯标签库选择器、后端服务化模型，仍属于目标架构预留，不代表当前标签管理页面已实现。

### 4.2 核心服务

- 语言：`Go`
- 选型原因：
  - 适合常驻服务与并发任务调度
  - 标准库和生态对文件系统、网络、gRPC、数据库支持成熟
  - 适合实现任务执行器、扫描器、哈希校验与服务端 API
  - 部署简单，后续迁移到 NAS 也相对顺滑

### 4.3 本地数据库

- 首选：`PostgreSQL 18`
- 选择原因：
  - 适合复杂关系模型与高并发后台任务写入
  - 适合大表、分区、结构化筛选、模糊搜索
  - 支持 `BTREE / BRIN / GIN / pg_trgm`
  - 更适合承载统一账本、副本状态、目录、任务、校验记录等多表联动

### 4.4 搜索与索引

- 首版搜索引擎：`PostgreSQL 自带索引能力`
- 首版不额外引入独立搜索引擎
- 原则：
  - 结构化筛选优先走 PostgreSQL
  - 文件名/路径/标签模糊搜索优先走 `pg_trgm`
  - 超大表范围筛选使用 `BRIN`
  - JSON/数组类字段适度使用 `GIN`

### 4.5 网盘上传执行器

- 首版网盘上传执行器：`CloudDrive2`
- 集成方式：`CloudDrive2 gRPC API`
- 集成目标：
  - 创建真实远程上传任务
  - 读取真实上传进度与速度
  - 暂停/恢复/取消 CloudDrive2 中对应任务
  - 将 CD2 任务状态同步到本系统任务中心

### 4.6 下载执行器

- 首版下载执行器：`aria2`
- 集成方式：将 `aria2c` 可执行文件作为项目内置二进制集成，并在安装时随产品一起部署
- 集成目标：
  - 复用 aria2 的断点下载能力
  - 复用 aria2 的暂停、恢复、取消能力
  - 将 aria2 下载任务统一封装进本系统任务中心
  - 使下载任务在用户视角上与导入、复制、同步保持一致的任务体验

## 5. 部署架构

### 5.1 当前阶段部署方式

- Tauri 客户端部署在 Windows 本机
- Go 核心服务部署在 Windows 本机
- PostgreSQL 部署在 Windows 本机（开发阶段部署在本机 Docker 环境）
- CloudDrive2 部署在 Windows 本机（开发阶段部署在本机 Docker 环境）
- aria2 部署在 Windows 本机（以项目内置二进制方式随开发环境和安装包提供）

### 5.2 当前阶段支持的存储端类型

- 本地磁盘目录
- 移动硬盘/U 盘目录（可选择作为导入源或存储介质）
- SMB/NAS 共享目录
- 通过 CloudDrive2 接入的网盘目录与网盘上传能力

### 5.3 后续演进方向

后续可迁移为：

- 客户端仍在 Windows
- 核心服务迁移到 NAS
- PostgreSQL 迁移到 NAS
- CloudDrive2 迁移到 NAS
- aria2 迁移到 NAS

当前设计需保持接口和模块边界清晰，但不在首版实现跨设备桥接逻辑。

## 6. 核心领域模型

### 6.1 资产库

`asset_library`

职责：

- 资产隔离边界
- 绑定多个存储端
- 作为统一资产、目录、任务、异常的归属范围

### 6.2 存储端

`storage_endpoint`

职责：

- 表示可读写、可扫描、可执行任务的受管存储位置
- 同一物理端点在不同任务中可扮演来源或目标
- 首版不单独区分 `ingest_source` 实体

关键属性：

- 端点类型：`LOCAL / REMOVABLE / SMB / CLOUD`
- 地址信息
- 凭证与鉴权配置
- 可用性状态
- 能力矩阵

### 6.3 文件资产主表

`file_asset`

职责：

- 统一文件资产主账本
- 对外一行代表一个文件资产
- 承载基础属性、路径语义、主展示字段与摘要状态

主字段建议：

- `id`
- `library_id`
- `relative_path`
- `parent_path`
- `file_name`
- `asset_kind`
- `asset_size`
- `asset_mtime`
- `lifecycle_state`
- `created_at`
- `updated_at`

其中：

- `lifecycle_state` 当前建议至少支持：`ACTIVE / PENDING_DELETE`

### 6.4 文件副本表

`file_replica`

职责：

- 显式表示每个资产在每个存储端上的状态
- 是“资产库 x 存储端”的状态矩阵

设计原则：

- 对每个 `file_asset`，为每个受管 `storage_endpoint` 显式维护一条状态记录
- 以空间换查询效率与状态明确性

主字段建议：

- `asset_id`
- `endpoint_id`
- `state`
- `observed_size`
- `observed_mtime`
- `last_confirmed_at`
- `verify_state`
- `created_at`
- `updated_at`

当前建议状态：

- `UNCONFIRMED`
- `PRESENT`
- `MISSING`

校验摘要状态建议：

- `UNVERIFIED`
- `BASIC_VERIFIED`
- `STRONG_VERIFIED`
- `VERIFY_FAILED`

### 6.5 目录表

`directory_entry`

职责：

- 全量显式维护目录节点
- 支撑目录树、目录浏览、目录级操作
- 不与 `file_asset` 混表

关键原则：

- 文件与目录不共用主表
- 目录是正式数据对象，但不是统一资产主对象

### 6.6 媒体增强模型

所有文件先统一纳入 `file_asset`。  
照片、视频、音频再进入媒体增强层。

#### 媒体总表

`media_asset`

职责：

- 表示资产是否是媒体文件
- 表示媒体类型与元数据解析状态
- 保存原始元数据 JSON

#### 子表

- `photo_metadata`
- `video_metadata`
- `audio_metadata`

设计原则：

- 高频筛选字段结构化
- 长尾字段保存在 `raw_metadata_json`
- 普通文件不进入媒体增强表

### 6.7 任务模型

- `job`
- `job_item`

`job` 表示任务主单，`job_item` 表示任务子项。

### 6.8 异常模型

`issue`

职责：

- 承接路径冲突、外部占用、传输失败、鉴权失效、空间不足、临时文件等异常

当前客户端字段对齐（2026-04-07）：

- 基础字段：`id / libraryId / taskId / taskItemId / category / type / nature / sourceDomain / severity / title / summary / detail / status / createdAt / updatedAt / resolvedAt / archivedAt`
- 来源模型：`IssueSource` 已落地 `taskId / taskItemId / taskTitle / taskItemTitle / assetId / fileNodeId / endpointId / endpointLabel / path / sourceLabel / routeLabel`
- 能力模型：`IssueCapabilities` 已落地 `canRetry / canConfirm / canPostpone / canIgnore / canRefreshCheck / canArchive / canClearHistory / canOpenTaskCenter / canOpenFileCenter / canOpenStorageNodes`
- 历史模型：`IssueHistoryRecord` 已落地 `id / issueId / action / operatorLabel / result / createdAt`
- 影响范围模型：`IssueImpactSummary` 已落地 `assetCount / replicaCount / folderCount / endpointCount / blocksStatusCommit / blocksTaskExecution`

当前前端已经按“查询 / 命令分离”思路组织 UI：页面只依赖异常记录与能力字段决定展示和按钮可用性，后续后端接入时应复用同一字段契约。

## 7. 文件身份与资产归并规则

### 7.1 固定规则

同一资产判断规则固定采用：

- 相对路径
- 文件大小
- 修改时间
- 时间容差

不支持用户配置规则。

### 7.2 业务含义

- 相对路径不同：不同资产
- 路径相同但大小/时间不一致：不是同一状态文件
- 内容相同但路径不同：仍视为不同资产

### 7.3 使用范围

该规则用于：

- 扫描归并
- 副本识别
- 资产构建

不用于强校验。

## 8. 目录模型设计

### 8.1 设计原则

- 文件和目录不放在同一张表
- 目录采用全量显式维护
- 文件通过 `parent_path` 或目录关系归属到目录节点

### 8.2 目录包含文件的表达方式

- `directory_entry` 表示目录节点
- `file_asset.parent_path` 表示资产所属目录
- 目录下文件通过查询 `file_asset.parent_path = directory.relative_path` 获得

### 8.3 目录操作

首版应支持：

- 在文件中心内浏览目录层级
- 新建目录
- 重命名目录
- 移动目录
- 查看目录下文件

当前客户端已经实现的是“面包屑 + 前进 / 后退 + 双击目录进入”的目录导航方式，尚未提供独立目录树。

目录移动与重命名需要联动更新受影响资产路径与副本目标路径；该能力当前仍属于后续实现项。

## 9. 扫描、对账与状态提交模型

### 9.1 扫描定位

扫描是统一资产表的写入入口之一，但不是主更新手段。

主更新来源仍包括：

- 导入
- 复制
- 同步
- 下载
- 删除
- 元数据解析

扫描主要用于：

- 初始化建库
- 对账
- 恢复
- 异常修复
- 外部变更回收

### 9.2 扫描提交原则

- 一次扫描必须完整成功后才能提交正式状态
- 不允许边扫边更新正式资产/副本状态
- 不允许按中间批次提交正式结果

### 9.3 扫描流程

1. 创建 `scan_session`
2. 将扫描结果写入 staging 或临时结构
3. 扫描完整成功
4. 统一归并更新 `file_asset` 与 `file_replica`
5. 更新目录节点
6. 清理临时数据

### 9.4 失败处理

若扫描失败：

- 不更新正式状态
- 保留旧状态
- 记录失败原因
- 允许重试

## 10. 统一任务系统设计

### 10.1 任务类型

首版建议支持：

- `SCAN`
- `RECONCILE_SCAN`
- `IMPORT`
- `COPY`
- `SYNC`
- `DOWNLOAD`
- `VERIFY`
- `DELETE`
- `METADATA_EXTRACT`
- `ASSET_CLEANUP`

### 10.2 任务状态

统一任务状态建议：

- `PENDING`
- `RUNNING`
- `PAUSED`
- `COMPLETED`
- `FAILED`
- `CANCELED`
- `PARTIAL_SUCCESS`

### 10.3 设计原则

- 任务必须持久化
- 客户端关闭不影响任务继续运行
- 任务必须支持子项
- 正式状态与任务过程状态分离
- 核心服务重启后可恢复任务

### 10.4 统一传输体验

虽然 `IMPORT / COPY / SYNC / DOWNLOAD` 是不同业务任务，但在任务中心对用户应呈现统一传输体验：

- 当前文件
- 总进度
- 当前文件进度
- 速度
- 剩余时间
- 暂停/继续/取消
- 校验状态
- 错误原因

## 11. 导入、复制、同步、下载的业务语义

### 11.1 导入

定义：

- 从来源端文件进入资产库
- 根据导入规则分发到一个或多个目标端
- 结果是生成新资产并建立副本状态

### 11.2 复制

定义：

- 从一个相对路径复制到另一个相对路径
- 或同路径复制为不同名称
- 结果是生成新的文件资产

元数据继承规则：

- 原生元数据继承
- 非原生元数据不继承

原生元数据包括：

- 拍摄时间
- 设备信息
- 分辨率
- 时长
- 编码参数
- EXIF 等

非原生元数据包括：

- 标签
- 评分
- 色标
- 备注

### 11.3 同步

定义：

- 在同一资产的多个存储端点之间补齐副本
- 不生成新资产
- 只更新已有资产的副本状态

源端选择规则：

- 优先级：`NAS > 本地 > 网盘`
- 同类型多个端点时，优先修改时间更晚者

### 11.4 下载

定义：

- 将远程可下载资源或云端文件拉取到本地受管存储端
- 由 aria2 执行真实下载过程
- 下载完成后更新本系统资产与副本状态

### 11.5 同步的限制规则

- 若目标端已存在当前资产自己的受管副本，则 UI 层面直接不支持再同步
- 不做同一资产旧版本覆盖
- 不做双向合并
- 不做复杂主副本协商

## 12. 路径占用与冲突处理

### 12.1 占用类型

目标路径写入前，需要统一判断占用类型：

- `NONE`
- `UNMANAGED_FILE`
- `MANAGED_REPLICA_SELF`
- `MANAGED_REPLICA_OTHER`

### 12.2 处理策略

#### 1. 目标路径空闲

- 直接执行

#### 2. 目标路径为未纳管文件

- 默认允许覆盖
- 但必须通过安全前置检查
- 覆盖行为进入任务日志

#### 3. 目标路径为当前资产自己的受管副本

- UI 层直接禁止发起该同步

#### 4. 目标路径为其他资产的受管副本

- 禁止自动覆盖
- 进入异常中心

### 12.3 路径类型说明

目录与文件名共存不构成冲突，例如：

- `a/`
- `a.mov`

只有同一完整路径位置要求文件而实际为目录，或反之，才构成真正路径类型冲突。

## 13. 删除与清理模型

### 13.1 用户操作

用户只看到两类删除：

- 删除资产
- 从指定端点删除

### 13.2 状态模型

删除后的正式状态：

- 被删除副本最终表现为 `MISSING`
- 只要仍有任一副本为 `PRESENT`，资产仍存在
- 当所有副本都不再存在时，资产进入逻辑删除标记

当前客户端原型中的文件中心实现略有简化：

- 从指定端点删除副本时，若删除的是最后一个受管副本，资产会直接从文件中心列表中移除
- 直接执行“删除资产”时，才会将条目标记为 `PENDING_DELETE`

### 13.3 逻辑删除与后台清理

当所有副本都不存在后：

1. `file_asset.lifecycle_state = PENDING_DELETE`
2. 推入后台清理队列
3. 由 `ASSET_CLEANUP` 任务在系统空闲时清理：
   - `file_replica`
   - `media_asset`
   - 媒体元数据子表
   - 标签关联
   - 目录引用关系
   - 其他扩展数据
4. 最后删除 `file_asset`

### 13.4 取消删除条件

若资产在 `PENDING_DELETE` 阶段尚未清理时，扫描或其他任务重新发现有效副本：

- 取消删除
- 恢复为 `ACTIVE`

## 14. 断点续传与传输执行架构

### 14.1 目标

所有跨端上传、下载、同步都必须支持断点续传。

### 14.2 统一分层

断点续传分三层：

#### 1. 任务层断点

- 任务做到哪一步
- 哪些子项完成
- 哪些子项失败
- 哪些子项待恢复

#### 2. 文件层断点

- 单文件已完成字节
- 当前偏移
- 已完成 chunk

#### 3. 端点层能力

- 是否支持偏移读取
- 是否支持偏移写入
- 是否支持分块上传
- 是否支持真实暂停/恢复

### 14.3 执行器模型

建议内部抽象统一执行器接口，再按实现区分：

- `NativeTransferExecutor`
  - 本地磁盘
  - 移动盘/U盘
  - SMB/NAS

- `Aria2DownloadExecutor`
  - HTTP/HTTPS 下载
  - 网盘可下载资源拉取
  - 真实下载任务管理
  - 断点下载
  - 暂停/恢复/取消

- `CloudDrive2TransferExecutor`
  - 网盘上传任务
  - 上传任务状态读取
  - 上传任务暂停/恢复/取消

### 14.4 当前阶段的真实网盘上传要求

不能仅将 CloudDrive2 当作 SMB 使用。

必须：

- 通过 CD2 API 创建真实远程上传任务
- 读取真实上传进度与速度
- 在本系统中暂停、恢复、取消对应任务
- 同步 CD2 中任务状态变更

### 14.5 当前阶段的下载实现要求

系统中的下载能力统一由 aria2 实现。

必须：

- 将 `aria2c` 可执行文件集成到项目中
- 后续安装包中随产品一起打包安装
- 通过核心服务创建真实 aria2 下载任务
- 读取真实下载进度、速度、剩余时间等状态
- 在本系统中暂停、恢复、取消对应下载任务
- 核心服务重启后可重新关联和恢复 aria2 下载任务状态

### 14.6 任务映射

每个使用外部执行器的 `job_item` 都应保存统一外部任务映射信息。

#### 使用 CloudDrive2 执行上传时，应保存：

- `executor_type = CLOUDDRIVE2`
- `external_task_id = cd2_task_id`
- `last_external_state`
- `last_progress`
- `last_speed`
- `last_error`

#### 使用 aria2 执行下载时，应保存：

- `executor_type = ARIA2`
- `external_task_id = aria2_gid`
- `last_external_state`
- `last_progress`
- `last_speed`
- `last_error`

### 14.7 统一体验原则

用户在任务中心看到的仍然是一套统一任务体验，而不是直接暴露 CD2 或 aria2 的原始任务模型。

## 15. CloudDrive2 集成架构

### 15.1 集成定位

CloudDrive2 是：

- 网盘存储端适配器
- 真实上传执行器
- 外部任务执行后端

CloudDrive2 不是：

- 系统权威任务中心
- 权威资产状态源

### 15.2 核心服务职责

核心服务负责：

- 创建业务任务
- 调用 CD2 API 创建真实网盘上传任务
- 保存任务映射关系
- 读取和映射 CD2 状态
- 更新 `file_replica`
- 在最终完成后更新正式资产状态

### 15.3 CD2 状态同步

核心服务需要支持：

- 主动轮询 CD2 任务状态
- 将 CD2 暂停/恢复/取消结果同步到本地任务中心
- 识别 CD2 端人工变更任务状态并同步回来

### 15.4 当前版本实现要求

后续需专项验证 CD2 API 细节，但当前架构已明确以下实现要求：

- 必须接入真实远程上传任务
- 必须支持任务控制
- 必须获取真实上传进度与速度
- 必须支持任务 ID 映射

## 16. aria2 下载集成架构

### 16.1 集成定位

aria2 是：

- 系统统一下载执行器
- 外部下载任务执行后端
- 断点下载能力提供者

aria2 不是：

- 系统权威任务中心
- 权威资产状态源

### 16.2 集成方式

- 将 `aria2c` 可执行文件集成到项目目录中
- 开发阶段直接随项目运行
- 发布阶段随产品安装包一起安装
- 由核心服务负责启动、停止、调度和状态同步

### 16.3 核心服务职责

核心服务负责：

- 创建 `DOWNLOAD` 任务与任务子项
- 调用 aria2 创建真实下载任务
- 保存 aria2 任务映射关系
- 读取 aria2 真实下载进度、速度、错误信息
- 将暂停、恢复、取消操作转发给 aria2
- 在下载完成后执行默认校验并更新副本状态

### 16.4 任务控制要求

用户在本系统中的以下操作必须真实映射到 aria2：

- 暂停
- 恢复
- 取消

系统还应支持：

- 核心服务重启后重新关联未完成的 aria2 任务
- 统一任务中心展示 aria2 任务状态

### 16.5 适用范围

当前架构中，凡是下载语义的任务，统一优先通过 aria2 执行，包括但不限于：

- 从网盘下载到本地受管存储端
- 从远程可下载资源拉取到本地受管存储端
- 未来可抽象为下载链路的同步子项

### 16.6 统一任务体验要求

虽然下载底层由 aria2 执行，但在任务中心必须保持与导入、复制、同步一致的统一体验：

- 当前文件
- 总进度
- 当前文件进度
- 速度
- 剩余时间
- 暂停/继续/取消
- 校验状态
- 错误原因

## 17. 校验架构

### 17.1 默认传输校验

导入、复制、同步、下载完成后的默认成功判定：

- 文件大小一致
- 修改时间符合预期

这是首版默认校验策略。

### 17.2 强校验

系统应支持强校验任务：

- 任务类型：`VERIFY`
- 校验方式：全文件哈希

### 17.3 强校验触发方式

- 用户手动发起
- 用户配置时间窗自动发起

### 17.4 时间窗规则

用户可配置夜间校验时间段。

规则：

- 到达时间窗自动开始
- 到达结束时间自动暂停
- 下一次时间窗继续上一次进度
- 不从头重来

### 17.5 强校验结果缓存

建议单独建表保存强校验结果，而不是将哈希值放入 `file_asset` 或 `file_replica` 主表。

建议字段：

- `asset_id`
- `endpoint_id`
- `algorithm`
- `hash_value`
- `verified_at`
- `observed_size`
- `observed_mtime`

- 所有文件先进入统一资产模型

## 19. 搜索与索引架构

### 19.1 首版搜索边界

首版搜索主要覆盖：

- 文件名
- 路径
- 标签
- 基础属性
- 媒体结构化字段

#### BTREE

用于：

- 主键

- 排序高频字段

用于：

## 20. 数据库与分区策略

### 20.1 重点大表

- `file_asset`
- `file_replica`
- `job_item`
- `issue`
- `verify_result`

### 20.2 分区原则

首版建议预留按 `library_id` 或时间维度分区的能力，但不强制所有表首版立即分区。

优先考虑的分区候选表：

- `file_replica`

- 高频筛选字段单独建索引

- 超大状态表保持“窄表化”

## 21. 统一状态写入链路

### 21.1 设计原则

所有会影响资产状态的行为，不允许各模块直接自由写账。

必须走统一写账链路：

1. 上游模块产出事实
2. 转换为标准资产变更动作
3. 统一写入 `file_asset` / `file_replica`
4. 重新计算资产摘要状态

### 21.2 上游来源

- 扫描
- 导入
- 复制
- 同步
- 下载
- 删除
- 元数据解析
- 校验

### 21.3 摘要更新原则

- 资产固有字段直接维护在 `file_asset`
- 多端存在性与可用性摘要由 `file_replica` 汇总驱动

## 22. 异常中心设计

### 22.1 首版应纳入的异常类型

- 目标路径被其他受管资产占用
- 外部未纳管文件占用
- 文件锁定/权限不足
- 鉴权失效
- 存储空间不足
- 上传失败
- 下载失败
- 校验失败
- 扫描失败
- 临时文件待处理

### 22.2 设计原则

- 异常是统一收敛入口
- 异常与任务失败相关联
- 异常与资产/副本可回溯关联

### 22.3 当前客户端原型对齐（2026-04-07）

- 异常中心已按 `冲突 / 传输 / 校验 / 节点与权限 / 容量与资源 / 扫描与解析 / 清理与治理` 七类一级分类落地展示。
- 问题性质 `BLOCKING / RISK` 已独立于状态建模，并在页面中作为单独筛选维度暴露。
- 页面已支持异常详情抽屉、批量治理、历史清理和跨页回查；轻动作在异常中心执行，强动作回原业务页面完成。
- 与任务中心和存储节点的聚焦跳转已稳定，后续服务化时应保持 `issueId / taskId / taskItemId / libraryId / endpointId / fileNodeId / path` 这组跨页定位键不变。

### 22.4 通知中心与异常中心的当前架构边界（2026-04-07）

- 通知中心为异常中心上游的“关注入口”，不替代异常中心治理能力。
- 通知中心承接的通知记录来自异常模型映射快照，模型层使用 `NoticeRecord`，不是 `IssueRecord` 直接复用。
- 设备插入信号使用 `HeaderSignal` 独立建模，不进入通知记录集合，不进入通知角标统计。
- 通知中心状态机当前包含 `UNREAD / READ / JUMPED / STALE`，其中 `STALE` 用于表示关联异常已清理或失效后的通知状态。
- 角标聚合当前口径为“未消费通知数”，即 `status = UNREAD` 的通知总量。
- 异常中心状态与通知中心状态分离维护：通知消费不等价于异常已治理，治理结果仍以异常中心状态机为准。

## 23. 非功能要求

### 23.1 性能目标

架构必须面向以下长期目标设计：

- 上亿级文件资产
- 数亿级副本状态记录
- 数 TB 单文件
- 百 TB 级以上总容量

### 23.2 首版性能原则

- 列表与目录查询必须分页化
- 大量状态筛选优先走副本表索引
- 元数据解析异步化
- 删除清理异步化
- 校验任务低优先级调度

### 23.3 可靠性要求

- 核心任务持久化
- 核心服务重启后可恢复
- 默认校验不可省略
- 强校验可持续调度
- CloudDrive2 任务状态必须可同步
- aria2 任务状态必须可同步

## 24. 安全与运维要求

### 24.1 凭证管理

- 存储端凭证不得硬编码
- CloudDrive2 鉴权信息只由核心服务安全保存与访问
- 客户端不直接持有敏感凭证

### 24.2 审计与日志

必须记录：

- 任务创建与结束
- 删除行为
- 覆盖未纳管文件行为
- CloudDrive2 外部任务映射关系
- aria2 外部任务映射关系
- 失败原因与重试历史

### 24.3 崩溃恢复

- 核心服务启动时重建未完成任务状态
- 检查 `PENDING_DELETE` 清理队列
- 检查正在运行的 CD2 外部任务映射
- 检查正在运行的 aria2 外部任务映射

## 27. 架构结论

本项目首版技术架构明确采用：

- `Tauri` 桌面客户端
- `Go` 本地常驻核心服务
- `PostgreSQL` 本地权威数据库
- `CloudDrive2 API` 作为网盘上传执行器
- `aria2` 作为统一下载执行器
- `file_asset` 作为统一资产主对象
- `file_replica` 作为多端显式状态矩阵
- `directory_entry` 作为全量显式维护目录表
- `job / job_item` 作为统一任务引擎基础
- 默认大小/时间校验 + 独立强校验任务

该架构满足当前业务规划，也为后续迁移到 NAS、扩展更多网盘、引入图像语义搜索保留了清晰的演进路径。
