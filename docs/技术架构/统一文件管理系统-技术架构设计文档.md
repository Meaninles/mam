# 统一文件管理系统技术架构设计文档

## 文档说明

- 更新时间：2026-04-11
- 对照代码：`client/src`、`client/src-tauri/src`
- 目标：把技术架构文档刷新为当前仓库里的真实实现，不再混写未来服务化架构

## 1. 当前架构结论

当前项目已经不再是“只有 Tauri 原型客户端”的状态，而是处于“原型客户端 + 切片 0、1、2、3 已落地 + 切片 4 作业内核已落地”的阶段。

截至当前，仓库中的真实架构由以下几层组成：

1. React + Vite + Tauri 的桌面客户端
2. Go 中心服务
3. Go 本地执行器
4. PostgreSQL 开发态主库
5. 仍然保留中的本地原型数据层

当前最重要的架构事实是：**三段式系统底座已经出现，存储节点、文件中心主资产链路、标签域和作业内核已经迁移到中心服务与 PostgreSQL；但任务中心、异常中心、通知中心与导入中心的客户端消费层仍主要处于本地原型 / 局部落地状态。**

- Rust 侧 SQLite：Tauri 环境下仍保留的存储节点本地桥接数据

## 3. 当前总体分层

## 3.1 表现层

页面与组件主要位于：

- `client/src/App.tsx`
- `client/src/pages/*`
- `client/src/components/*`

表现层负责：

- 页面渲染
- 交互事件
- 标签页工作区管理
- 抽屉、弹层、上下文菜单
- 页面间跳转

## 3.2 编排层

当前真正的应用编排中心是 `client/src/App.tsx`。它负责：

- 顶层工作区标签管理
- 左侧导航与页头入口
- 文件中心状态
- 任务中心状态
- 异常中心上下文聚焦
- 通知跳转路由
- 设置草稿与保存
- 文件中心动作触发后对任务 / 异常 / 通知的联动写入

换句话说，当前并没有独立的全局状态框架，`App.tsx` 是主要的业务 orchestrator。

## 3.3 领域工具层

当前项目中已有一批轻量领域模块：

- `clientState.ts`：全局持久化状态的装载、克隆、设置值解析
- `workspaceTabs.ts`：多标签工作区的插入、关闭、移动、恢复
- `importCenter.ts`：导入中心筛选、排序、提交可用性判断
- `noticeCenter.ts`：通知排序、消费、跳转后的状态更新
- `fileCenterApi.ts`：文件中心的中心服务适配层，以及端点动作 / `test` fallback 的本地补充层
- `jobsApi.ts`：作业查询、控制与 SSE 订阅的客户端基础适配层
- `storageNodesApi.ts`：存储节点的中心服务调用、浏览器 fallback 与少量 Tauri 桥接

## 3.4 中心服务层

当前仓库已新增 `services/center`，并落地以下最小底座能力：

- 配置读取

- 本地执行器注册与心跳接收
- 系统运行状态汇总
- 面向客户端运行状态轮询的最小 CORS 支持
- 统一作业模型、状态流转、调度与事件写入
- 作业 SSE 基础能力
- 资产库、目录树、文件条目浏览
- 文件详情查询
- 新建目录、扫描、上传、文件级标记更新与条目删除

## 3.7 数据层

当前数据层并不是单一来源，而是三块：

### A. 文件中心 / 标签管理数据

- 前端入口：`client/src/lib/fileCenterApi.ts`
- 中心服务主链路：`services/center/internal/assets/*` + `services/center/internal/tags/*` + PostgreSQL
- 本地补充链路：`sql.js` + `localStorage`
- key：`mare-file-center-sqlite-v4`

中心服务主链路当前负责：

- 资产库列表
- 目录树与文件条目
- 资产副本状态聚合
- 文件详情
- 新建目录、扫描、上传、文件级星级 / 色标、资产删除
- 标签管理（分组、标签、作用域、置顶/排序、合并/删除）
- 标签建议（文件中心与批量标签编辑使用）
- 文件条目与目录标签保存（通过 `PATCH /api/file-entries/{id}/annotations` 写入 PostgreSQL）

本地补充链路当前负责：

- 同步到端点 / 删除端点副本的原型态状态流转
- `test` 运行时的文件中心 fallback

### B. 全局客户端状态

- 文件：`client/src/lib/clientState.ts`
- 存储：`localStorage`
- key：`mare-client-state-v4`

这套数据负责：

- 导入中心 mock 数据
- 任务中心 mock 数据
- 异常中心 mock 数据
- 通知中心数据
- 设置项
- 头部信号
- 资产库列表

需要特别注意：

- `loadPersistedState()` 会重新注入导入、任务、异常等种子数据
- 因此这部分状态不是完全的“全量持久化快照”
- 这不影响中心服务中的真实作业内核已经成立；只是任务中心 UI 还未改接到真实作业数据

### C. 作业数据

- 前端入口：`client/src/lib/jobsApi.ts`
- 中心服务主链路：`services/center/internal/jobs/*` + PostgreSQL
- 当前后端真实模型：
  - `jobs`
  - `job_items`
  - `job_attempts`
  - `job_events`
  - `job_object_links`

中心服务当前负责：

- 创建扫描作业
- 作业调度执行
- 作业查询、详情、事件读取
- 暂停 / 继续 / 取消 / 重试 / 优先级更新
- SSE 推送作业事件

当前客户端现实边界：

- 文件中心与存储节点已经会触发真实作业创建
- 任务中心页面尚未消费这套真实作业数据

### D. 存储节点数据

- 前端入口：`client/src/lib/storageNodesApi.ts`
- Tauri 命令：`client/src-tauri/src/lib.rs`
- Rust 侧数据库文件名：`storage_nodes_mock.sqlite3`（当前仅作为桌面桥接遗留实现存在，不代表中心服务权威数据源）
- 浏览器 fallback key：`mare-storage-nodes-browser-v3`

这套数据当前已经分裂为两层：

- 真实层：`services/center` 的存储域接口和 PostgreSQL 表
- 过渡层：`storageNodesApi.ts` 的浏览器 fallback / Tauri 桥接

真实层负责：

- 挂载文件夹列表
- NAS 列表
- 网盘列表
- 扫描历史
- 连接测试结果相关状态

过渡层仍负责：

- 浏览器离线读写
- 本地目录选择
- 少量尚未完全收口的桌面桥接

## 4. 当前架构拓扑

```text
React 页面/组件
  -> App.tsx 顶层编排
    -> systemRuntimeApi / runtimeConfig
    -> workspaceTabs / importCenter / noticeCenter / clientState
    -> fileCenterApi
         -> 优先 center API
         -> 扫描动作创建真实作业
         -> 端点动作 / test fallback 走 sql.js + localStorage
    -> jobsApi
         -> /api/jobs/*
         -> /api/events/stream
    -> storageNodesApi
         -> 优先 center API
         -> 挂载扫描创建真实作业
         -> 失败时浏览器 fallback -> localStorage
         -> 少量 Tauri invoke -> Rust -> 本地目录桥接
    -> data.ts 种子数据
  -> 中心服务 (Go)
       -> PostgreSQL
       -> assets/tags/storage/jobs
       -> agent/register, agent/heartbeat, runtime status, SSE
  -> 本地执行器 (Go)
       -> 注册到中心服务
       -> 周期性心跳
```

## 5. 关键架构边界

### 5.0 切片 0 已实现但业务迁移未完成

当前需要特别承认一个过渡态事实：

- 底座层已经三段式化
- 业务页数据源大多仍是本地原型态

因此不能把当前仓库表述为“已经完成服务化”，也不能再表述为“只有桌面原型客户端”。

## 5.1 文件中心与任务中心并未共用一个真实数据库

当前文件中心主资产数据已经通过 `fileCenterApi` 接到中心服务，扫描动作也已经切到真实作业；但任务中心、异常中心仍依赖 `clientState` 中的 `fileNodes`、`taskRecords`、`issueRecords` 等种子数据。

因此当前存在以下边界事实：

- 文件中心的资产、目录、详情、新建目录、上传、星级 / 色标、资产删除优先落到中心服务
- 文件中心目录扫描已经改为“创建真实作业 -> 中心服务调度执行”
- 文件中心的标签、标签建议与标签保存已走中心服务与 PostgreSQL；端点状态原型逻辑仍落到 `fileCenterApi` 本地层
- 任务中心与异常中心的文件关联仍然主要依赖 `PersistedState.fileNodes`
- 两者尚未统一为单一权威资产模型

文档、设计和后续改造都必须承认这一事实。

## 5.2 存储节点已经具备“中心服务 + 少量桌面桥接”链路

存储节点页面不是纯前端 mock。当前已经具备以下混合链路：

- 读取 / 保存 / 测试 / 删除大部分走中心服务 HTTP 接口
- 浏览目录仍通过 Tauri 本地桥接
- 浏览器环境仍保留 fallback
- 少量历史实现仍保留 Rust 侧 SQLite 桥接数据作为过渡

当前实现的 Tauri 命令包括：

- `storage_nodes_load_dashboard`
- `storage_nodes_save_mount_folder`
- `storage_nodes_save_nas_node`
- `storage_nodes_save_cloud_node`
- `storage_nodes_run_mount_scan`
- `storage_nodes_run_mount_connection_test`
- `storage_nodes_run_nas_connection_test`
- `storage_nodes_run_cloud_connection_test`
- `storage_nodes_update_mount_heartbeat`
- `storage_nodes_load_mount_scan_history`
- `storage_nodes_delete_mount_folder`
- `storage_nodes_delete_nas_node`
- `storage_nodes_delete_cloud_node`
- `storage_nodes_browse_local_folder`

## 5.3 通知、异常、任务、文件中心通过 `App.tsx` 直接耦合

当前跨页面联动主要不是事件总线，也不是完整的服务消费层，而是 `App.tsx` 中的直接函数编排，例如：

- 文件中心同步 -> 创建传输任务
- 文件中心删除 -> 创建删除清理任务
- 导入中心提交 -> 创建导入任务 + 导入报告
- 异常处理 -> 关联通知失效
- 通知跳转 -> 写入聚焦请求，再切换工作区

需要补充一条当前已经成立的新边界：

- 文件中心扫描、存储节点扫描不再由 `App.tsx` 本地编排任务，而是直接调用中心服务创建真实作业

这个设计适合当前原型阶段，但不适合作为生产最终形态。

## 7. 当前运行时状态模型

## 7.1 工作区状态

顶层工作区状态包含：

- `openWorkspaceViews`
- `mountedWorkspaceViews`
- `activeWorkspaceView`
- `recentlyClosedWorkspaceViews`
- `workspaceRefreshTokens`

设计结果是：

- 打开过的页面会保活
- 关闭标签不等于卸载页面
- 刷新是通过增加 `workspaceRefreshTokens` 强制重建对应页面

## 7.2 页面局部状态

各页面大量使用局部状态管理：

- 文件中心：目录、筛选、分页、选中项、待确认动作
- 导入中心：当前设备、筛选、抽屉、提交确认
- 任务中心：筛选、展开状态、选中任务、异常浮窗
- 异常中心：筛选、上下文聚焦、选中项、详情抽屉
- 存储节点：分页、表单、菜单、测试结果、历史、心跳编辑器

## 7.3 设置状态

设置页采用“双层模型”：

- `persisted.settings`：已保存配置
- `settingsDraft`：编辑中的草稿配置

只有点击“保存设置”后，草稿才会写回持久状态。

## 8. 当前实际数据模型

### 8.1 文件中心数据模型

当前文件中心围绕以下对象运转：

- 服务端主模型：
  - `libraries`
  - `library_directories`
  - `assets`
  - `asset_replicas`
  - `directory_presences`
  - `asset_metadata`
- 服务端标签模型：
  - `tag_groups`
  - `tags`
  - `tag_library_scopes`
  - `asset_tag_links`
  - `directory_tag_links`
- 本地补充模型：
  - 端点同步 / 删除副本的 `sql.js` 原型态状态流转
  - `test` 运行时 fallback

这意味着文件中心已经不再是单一的本地 `sql.js` 模型，而是“后端资产读写 + 后端标签域 + 后端扫描作业 + 本地端点原型动作”并存的结构。

### 8.2 任务模型

任务主模型是 `TaskRecord`，子项模型是 `TaskItemRecord`。

关键字段包括：

- `kind`
- `businessType`
- `syncLinkType`
- `otherTaskType`
- `status`
- `priority`
- `fileNodeIds`
- `sourcePath`
- `targetPath`
- `issueIds`

### 8.3 异常模型

异常主模型是 `IssueRecord`，包含：

- 分类
- 严重级别
- 性质
- 来源对象
- 影响范围
- 能力开关
- 处理历史

### 8.4 通知模型

通知主模型是 `NoticeRecord`，包含：

- 通知类型
- 来源类型
- 关联异常
- 状态
- 跳转参数
- 能力开关

### 8.5 存储节点模型

存储节点侧模型以三类记录为核心：

- `MountFolderRecord`
- `NasRecord`
- `CloudRecord`

并辅以：

- `StorageConnectionTestResult`
- `StorageScanHistoryItem`

## 10. 当前架构限制

### 10.1 数据源尚未统一

当前最大限制是多个本地数据源并行存在：

- 文件中心标签 / 端点原型逻辑一套
- 任务 / 导入 / 异常 / 通知 / 设置一套
- 存储节点一套

同时还存在一个新的“后端已先行、前端未消费”的不对称状态：

- 作业内核已经在中心服务落地
- 但任务中心 UI 还没有完成切片 5 所需的真实消费层迁移

### 10.2 `App.tsx` 过重

`App.tsx` 同时承担：

- 路由
- 编排
- 跨页面跳转
- 部分状态变更
- 任务生成
- 通知联动

这会增加后续生产化改造成本。

### 10.3 大部分业务仍是高保真 mock

除了切片 0 已落地的中心服务、本地执行器和客户端状态接线，以及已经先行落地的存储节点真实服务、文件中心主资产链路、标签域与作业内核之外，其余大多数业务消费层仍是本地模拟，不应在文档中表述成已全部接入真实业务服务。

## 11. 当前版本结论

截至 2026-04-11，仓库中真正已经成立的技术架构可以概括为：

- 一个 React/Tauri 桌面工作台
- 一个 Go 中心服务底座
- 一个 Go 本地执行器底座
- 一个 PostgreSQL 开发态主库
