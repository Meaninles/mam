# 统一文件管理系统技术架构设计文档

## 文档说明

- 更新时间：2026-04-08
- 对照代码：`client/src`、`client/src-tauri/src`
- 目标：把技术架构文档刷新为当前仓库里的真实实现，不再混写未来服务化架构

## 1. 当前架构结论

当前项目已经不再是“只有 Tauri 原型客户端”的状态，而是处于“原型客户端 + 切片 0 底座已落地”的阶段。

截至当前，仓库中的真实架构由以下几层组成：

1. React + Vite + Tauri 的桌面客户端
2. Go 中心服务
3. Go 本地执行器
4. PostgreSQL 开发态主库
5. 仍然保留中的本地原型数据层

当前最重要的架构事实是：**三段式系统底座已经出现，但业务主源尚未全面从本地原型数据迁移到中心服务。**

- Rust 侧 SQLite：Tauri 环境下的存储节点 mock 数据库

## 3. 当前总体分层

## 3.1 表现层

页面与组件主要位于：

- `client/src/App.tsx`
- `client/src/pages/*`
- `client/src/components/*`

表现层负责：

- 页面渲染
- 交互事件
- 标签页工作区管理
- 抽屉、弹层、上下文菜单
- 页面间跳转

## 3.2 编排层

当前真正的应用编排中心是 `client/src/App.tsx`。它负责：

- 顶层工作区标签管理
- 左侧导航与页头入口
- 文件中心状态
- 任务中心状态
- 异常中心上下文聚焦
- 通知跳转路由
- 设置草稿与保存
- 文件中心动作触发后对任务 / 异常 / 通知的联动写入

换句话说，当前并没有独立的全局状态框架，`App.tsx` 是主要的业务 orchestrator。

## 3.3 领域工具层

当前项目中已有一批轻量领域模块：

- `clientState.ts`：全局持久化状态的装载、克隆、设置值解析
- `workspaceTabs.ts`：多标签工作区的插入、关闭、移动、恢复
- `importCenter.ts`：导入中心筛选、排序、提交可用性判断
- `noticeCenter.ts`：通知排序、消费、跳转后的状态更新
- `fileCenterApi.ts`：文件中心与标签管理的本地数据库 API
- `storageNodesApi.ts`：存储节点的 Tauri 调用与浏览器 fallback

## 3.4 中心服务层

当前仓库已新增 `services/center`，并落地以下最小底座能力：

- 配置读取
- 结构化日志
- 统一错误与响应 envelope
- PostgreSQL 连接
- baseline migration
- 健康检查与就绪检查
- 本地执行器注册与心跳接收
- 系统运行状态汇总
- 面向客户端运行状态轮询的最小 CORS 支持

已实现接口包括：

- `GET /healthz`
- `GET /readyz`
- `GET /api/runtime/status`
- `POST /agent/register`
- `POST /agent/heartbeat`

## 3.5 本地执行器层

当前仓库已新增 `services/agent`，并落地以下最小底座能力：

- 配置读取
- 身份持久化
- 最小运行摘要
- 注册
- 周期性心跳
- 基础重试

## 3.6 共享契约层

当前仓库已新增 `shared/contracts`，用于承载切片 0 所需的：

- 健康检查 DTO
- 运行状态 DTO
- 执行器注册与心跳 DTO
- 最小运行态 OpenAPI

## 3.7 数据层

当前数据层并不是单一来源，而是三块：

### A. 文件中心 / 标签管理数据库

- 文件：`client/src/lib/fileCenterApi.ts`
- 存储：`sql.js` + `localStorage`
- key：`mare-file-center-sqlite-v4`

这套数据负责：

- 目录树与文件条目
- 端点副本状态
- 文件元数据
- 星级、色标
- 标签建议
- 标签分组、标签作用域、标签关联

### B. 全局客户端状态

- 文件：`client/src/lib/clientState.ts`
- 存储：`localStorage`
- key：`mare-client-state-v4`

这套数据负责：

- 导入中心 mock 数据
- 任务中心 mock 数据
- 异常中心 mock 数据
- 通知中心数据
- 设置项
- 头部信号
- 资产库列表

需要特别注意：

- `loadPersistedState()` 会重新注入导入、任务、异常等种子数据
- 因此这部分状态不是完全的“全量持久化快照”

### C. 存储节点数据

- 前端入口：`client/src/lib/storageNodesApi.ts`
- Tauri 命令：`client/src-tauri/src/lib.rs`
- Rust 侧数据库文件名：`storage_nodes_mock.sqlite3`
- 浏览器 fallback key：`mare-storage-nodes-browser-v3`

这套数据负责：

- 挂载文件夹列表
- NAS 列表
- 网盘列表
- 扫描历史
- 连接测试结果相关状态

## 4. 当前架构拓扑

```text
React 页面/组件
  -> App.tsx 顶层编排
    -> systemRuntimeApi / runtimeConfig
    -> workspaceTabs / importCenter / noticeCenter / clientState
    -> fileCenterApi (sql.js + localStorage)
    -> storageNodesApi
         -> Tauri invoke -> Rust -> SQLite mock
         -> 或 浏览器 fallback -> localStorage
    -> data.ts 种子数据
  -> 中心服务 (Go)
       -> PostgreSQL
       -> agent/register, agent/heartbeat, runtime status
  -> 本地执行器 (Go)
       -> 注册到中心服务
       -> 周期性心跳
```

## 5. 关键架构边界

### 5.0 切片 0 已实现但业务迁移未完成

当前需要特别承认一个过渡态事实：

- 底座层已经三段式化
- 业务页数据源大多仍是本地原型态

因此不能把当前仓库表述为“已经完成服务化”，也不能再表述为“只有桌面原型客户端”。

## 5.1 文件中心与任务中心并未共用一个真实数据库

当前文件中心使用 `fileCenterApi` 的 `sql.js` 数据库，而任务中心、异常中心依赖 `clientState` 中的 `fileNodes`、`taskRecords`、`issueRecords` 等种子数据。

因此当前存在以下边界事实：

- 文件中心对文件、标签、端点状态的修改会落到 `fileCenterApi`
- 任务中心与异常中心的文件关联仍然主要依赖 `PersistedState.fileNodes`
- 两者尚未统一为单一权威资产模型

文档、设计和后续改造都必须承认这一事实。

## 5.2 存储节点已经具备“前端 -> Tauri -> Rust -> SQLite”链路

存储节点页面不是纯前端 mock。当前已经具备完整桌面命令链路：

- 前端发起 `invoke`
- Rust 侧读写 SQLite
- 返回结构化结果
- 前端刷新仪表盘

当前实现的 Tauri 命令包括：

- `storage_nodes_load_dashboard`
- `storage_nodes_save_mount_folder`
- `storage_nodes_save_nas_node`
- `storage_nodes_save_cloud_node`
- `storage_nodes_run_mount_scan`
- `storage_nodes_run_mount_connection_test`
- `storage_nodes_run_nas_connection_test`
- `storage_nodes_run_cloud_connection_test`
- `storage_nodes_update_mount_heartbeat`
- `storage_nodes_load_mount_scan_history`
- `storage_nodes_delete_mount_folder`
- `storage_nodes_delete_nas_node`
- `storage_nodes_delete_cloud_node`
- `storage_nodes_browse_local_folder`

## 5.3 通知、异常、任务、文件中心通过 `App.tsx` 直接耦合

当前跨页面联动主要不是事件总线，也不是服务层，而是 `App.tsx` 中的直接函数编排，例如：

- 文件中心同步 -> 创建传输任务
- 文件中心删除 -> 创建删除清理任务
- 导入中心提交 -> 创建导入任务 + 导入报告
- 异常处理 -> 关联通知失效
- 通知跳转 -> 写入聚焦请求，再切换工作区

这个设计适合当前原型阶段，但不适合作为生产最终形态。

## 6. 当前页面级架构职责

### 6.1 文件中心

- 页面：`FileCenterPage.tsx`、`FileDetailSheet.tsx`
- 数据源：`fileCenterApi.ts`
- 职责：目录浏览、筛选、分页、上传、新建目录、同步、删除、标签与标记编辑

### 6.2 导入中心

- 页面：`ImportCenterPage.tsx`
- 数据源：`PersistedState.import*`
- 职责：设备会话、目标编排、预检、文件清单、提交导入

### 6.3 任务中心

- 页面：`TaskCenterPage.tsx`
- 数据源：`PersistedState.taskRecords`、`taskItemRecords`
- 职责：运行期跟踪、任务级与子项级操作、详情抽屉、异常浮窗

### 6.4 异常中心

- 页面：`IssuesPage.tsx`
- 数据源：`PersistedState.issueRecords`
- 职责：统一治理、筛选、批量动作、详情查看、历史清理

### 6.5 通知中心

- 页面：`NotificationCenterSheet.tsx`
- 数据源：`PersistedState.noticeRecords`
- 职责：轻量提醒、已读 / 已跳转状态维护、跳转分发

### 6.6 存储节点

- 页面：`StorageNodesPage.tsx`
- 数据源：`storageNodesApi`
- 职责：挂载目录、NAS、网盘的配置与操作

### 6.7 设置

- 页面：`SettingsPage.tsx`、`SettingsPanels.tsx`
- 数据源：`PersistedState.settings` + `settingsDraft`
- 职责：客户端默认策略、预览型设置子页、标签管理工作台承载

## 7. 当前运行时状态模型

## 7.1 工作区状态

顶层工作区状态包含：

- `openWorkspaceViews`
- `mountedWorkspaceViews`
- `activeWorkspaceView`
- `recentlyClosedWorkspaceViews`
- `workspaceRefreshTokens`

设计结果是：

- 打开过的页面会保活
- 关闭标签不等于卸载页面
- 刷新是通过增加 `workspaceRefreshTokens` 强制重建对应页面

## 7.2 页面局部状态

各页面大量使用局部状态管理：

- 文件中心：目录、筛选、分页、选中项、待确认动作
- 导入中心：当前设备、筛选、抽屉、提交确认
- 任务中心：筛选、展开状态、选中任务、异常浮窗
- 异常中心：筛选、上下文聚焦、选中项、详情抽屉
- 存储节点：分页、表单、菜单、测试结果、历史、心跳编辑器

## 7.3 设置状态

设置页采用“双层模型”：

- `persisted.settings`：已保存配置
- `settingsDraft`：编辑中的草稿配置

只有点击“保存设置”后，草稿才会写回持久状态。

## 8. 当前实际数据模型

### 8.1 文件中心数据模型

当前文件中心围绕以下对象运转：

- `entries`
- `entry_endpoints`
- `entry_metadata`
- `tag_groups`
- `tags`
- `tag_library_scopes`
- `entry_tag_links`

这是目前仓库里最完整、最接近领域数据库的模型。

### 8.2 任务模型

任务主模型是 `TaskRecord`，子项模型是 `TaskItemRecord`。

关键字段包括：

- `kind`
- `businessType`
- `syncLinkType`
- `otherTaskType`
- `status`
- `priority`
- `fileNodeIds`
- `sourcePath`
- `targetPath`
- `issueIds`

### 8.3 异常模型

异常主模型是 `IssueRecord`，包含：

- 分类
- 严重级别
- 性质
- 来源对象
- 影响范围
- 能力开关
- 处理历史

### 8.4 通知模型

通知主模型是 `NoticeRecord`，包含：

- 通知类型
- 来源类型
- 关联异常
- 状态
- 跳转参数
- 能力开关

### 8.5 存储节点模型

存储节点侧模型以三类记录为核心：

- `MountFolderRecord`
- `NasRecord`
- `CloudRecord`

并辅以：

- `StorageConnectionTestResult`
- `StorageScanHistoryItem`

## 10. 当前架构限制

### 10.1 数据源尚未统一

当前最大限制是多个本地数据源并行存在：

- 文件中心 / 标签管理一套
- 任务 / 导入 / 异常 / 通知 / 设置一套
- 存储节点一套

### 10.2 `App.tsx` 过重

`App.tsx` 同时承担：

- 路由
- 编排
- 跨页面跳转
- 部分状态变更
- 任务生成
- 通知联动

这会增加后续生产化改造成本。

### 10.3 大部分业务仍是高保真 mock

除了切片 0 已落地的中心服务、本地执行器和客户端状态接线之外，绝大多数业务数据与行为仍是本地模拟，不应在文档中表述成已接入真实业务服务。

## 11. 当前版本结论

截至 2026-04-08，仓库中真正已经成立的技术架构可以概括为：

- 一个 React/Tauri 桌面工作台
- 一个 Go 中心服务底座
- 一个 Go 本地执行器底座
- 一个 PostgreSQL 开发态主库
- 一组最小共享 contracts
- 一个以 `App.tsx` 为中心的本地编排层
- 一个 `sql.js` 驱动的文件中心本地数据库
- 一个 `localStorage` 驱动的全局 mock 状态层
- 一个能跑通 Tauri invoke 的存储节点命令层

后续如果继续演进，优先级最高的不是继续扩展页面，而是基于已经落地的切片 0 底座，逐步收口数据权威来源与编排边界。
