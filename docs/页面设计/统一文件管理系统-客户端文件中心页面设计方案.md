## 1. 页面定位

文件中心仍然是客户端的资产主工作台，负责统一承接以下能力：

- 统一资产浏览
- 目录导航
- 搜索与筛选
- 端点同步与删除副本
- 标签与标记编辑
- 上传与新建目录
- 进入文件详情

页面层面对外仍然统一使用“同步”这一入口名称，但实际执行路径已经不是单一复制，而是根据源端点和目标端点的类型切换到不同执行器。

## 2. 当前真实同步架构

### 2.1 三条正式链路

当前正式支持的重操作链路只有三类：

1. `LOCAL/NAS -> CLOUD`
   - 通过 `CloudDrive2 remote upload` 执行上传
   - 中心服务读取本地或 NAS 文件，按块喂给 CD2
2. `CLOUD -> LOCAL/NAS`
   - 先由 CD2 解析远端文件下载地址
   - 再交给 `aria2` 下载到本地或 NAS
3. `LOCAL/NAS -> LOCAL/NAS`
   - 由中心服务直接执行文件复制

### 2.2 当前未实现的链路

以下能力当前不要写成“已支持”：

- CD2 官方“离线下载 + 自动转存上传”完整流水线
- Aria2 下载完成后再把本地物理路径移交给 CD2 继续上传
- 通过 WebDAV 做正式云端复制
- `CLOUD -> CLOUD` 正式同步
- 面向网盘链路的专用重试治理面板

当前项目里并不存在“先离线下载，再自动转存到云盘”的正式业务链路。现有实现只是上传链路和下载链路分别接入了 CD2 与 Aria2。

## 3. 当前状态同步口径

### 3.1 作业层

文件中心发起同步、删除副本、删除资产后，都会创建中心服务真实作业：

- `REPLICATE`
- `DELETE_REPLICA`
- `DELETE_ASSET`

任务中心通过 `jobsApi` 与 `/api/events/stream` 消费真实作业数据，不再消费页面内模拟任务。

### 3.2 外部执行器层

当前页面需要明确区分两层状态：

- 作业状态：`PENDING / QUEUED / RUNNING / PAUSED / FAILED / COMPLETED ...`
- 外部执行器状态：
  - `aria2`: `active / waiting / paused / error / complete / removed`
  - `CloudDrive2`: `WaitforPreprocessing / Preprocessing / Inqueue / Transfer / Pause / Finish / Error / FatalError ...`

页面展示应优先把外部状态翻译为用户可读阶段，而不是把“作业仍在运行”误解为“上传已经成功”。

### 3.3 集成健康门禁

文件中心在计算“同步 / 删除副本”可用性时，会同时考虑：

- CloudDrive2 运行状态（`cd2Online`）
- aria2 运行状态（`aria2Online`）
- 115 端点鉴权就绪（`cloudAuthReady`）

当 CloudDrive2 或 aria2 不可用，或 115 端点未就绪时，页面会展示“云端动作受限”的提示卡，并提供跳转到设置页或存储节点页的入口。

## 4. 当前已规避的问题

### 4.1 Aria2 进度不再读取本地文件大小

当前 Aria2 进度通过 JSON-RPC `tellStatus` 获取，读取：

- `totalLength`
- `completedLength`
- `downloadSpeed`

因此不会再把预分配文件大小、乱序写盘结果误当成下载进度。

### 4.2 不做边下边传

当前实现没有把 Aria2 输出流直接接给 CD2。上传和下载分别是独立分支：

- 上传分支：本地或 NAS 文件直接送入 CD2
- 下载分支：云端文件落地到本地或 NAS 后结束

这避免了“乱序下载文件流无法断点续传”的高风险设计。

### 4.3 Aria2 具备基础恢复条件

当前 Aria2 侧已具备以下基础条件：

- `continue=true`
- `aria2.session` 持久化
- 通过任务 ID 或目标路径恢复下载任务

因此服务重启或作业重新排队时，下载链路具备基础恢复能力。

### 4.4 CD2 上传成功标准已收敛到真实终态

上传成功不能再以“本系统已经把字节喂给 CD2”作为标准，当前正式口径是：

- 只认 `GetUploadFileList` 中对应任务进入 `Finish`
- 不认 CD2 缓存目录出现文件
- 不认本地读取完成
- 不认任务字节数暂时跑满

## 5. 当前仍然存在的边界与风险

### 5.1 外部状态仍未完全结构化持久化

当前系统已经记录外部任务引擎、任务 ID 和最近状态，但仍主要通过最近一次事件 payload 反查外部状态，而不是把外部状态作为独立字段长期维护。

这意味着：

- 状态源头仍不够单一
- 任务详情和恢复逻辑仍依赖“最近一条事件”语义
- 后续若要扩展恢复令牌、偏移量、会话信息，演进成本较高

### 5.2 CD2 上传通道断线容错不够

当前 `RemoteUploadChannel` 依赖单条长连接 gRPC 流承接远程上传请求。现状下，一旦 `stream.Recv()` 因网络抖动、连接重置或代理波动返回错误，本地实现会：

- 重置上传通道
- 直接终止当前挂在本地内存中的上传会话

这意味着：

- gRPC 流断开不一定等于服务端上传任务已经失败
- 但当前本地协调层会过早终止所有正在等待的上传会话

这是当前代码里已经真实存在的缺陷，不是理论风险。后续应优先重连通道并尝试继续等待，而不是“一条流断开就本地全量失败”。

### 5.3 高并发进度回写存在数据库压力风险

当前上传和下载的进度回调会高频触发作业层更新。每次真正落库时，不只是更新字节数，还会伴随：

- `job_items` 更新
- `jobs` 更新
- 聚合作业进度
- 写入 `job_events`

这意味着在并发传输任务上升后，系统会进入高频事务写入模式。当前代码路径已经真实存在这个结构性风险，但是否已经在现网触发“数据库锁死”，不能仅凭静态阅读直接下结论。

更准确的表述应是：

- 当前实现会把系统推向高频写库模式
- 并发放大后可能导致连接池压力、接口响应抖动、任务中心进度跳变
- 后续需要补节流、内存缓存或集中聚合回写

### 5.4 云到云链路没有正式支持

当前路由规划层仍可能把云副本视为候选来源，但真正的上传源读取器只支持：

- `LOCAL`
- `NAS`

不支持：

- `CLOUD`

因此 `CLOUD -> CLOUD` 目前不能宣传为已完成能力。后续需要在规划层显式阻断，或者另起独立链路实现。

### 5.5 Aria2 恢复逻辑对错误态任务处理不完整

当前 Aria2 恢复逻辑虽然已经具备以下基础：

- 可按旧任务 ID 查找任务
- 可按目标路径重新发现任务
- 已开启 `continue=true`

但仍存在一个真实边界问题：

- 如果恢复到的是旧的 `error` 任务
- 当前流程不会主动刷新下载地址
- 也不会显式把错误态任务重新激活
- 后续等待逻辑会很快再次把该任务判成失败

这在云端直链带过期时间时尤其危险，因为旧 URL 失效后，仅“重新找到旧 GID”并不等于真正恢复了下载。

这是当前代码里已经真实存在的缺陷，不应再被描述成“完整可靠的错误态断点恢复”。

### 5.6 CD2 恢复语义仍需继续收敛

当前 CD2 恢复主要依赖：

- `uploadId`
- `deviceId`
- 重新 `AttachUpload()`

还没有形成“恢复令牌、偏移、分片完成度”完全由本系统掌握的持久化模型。当前做法可以工作，但不应被写成“完整断点续传编排中心”。

### 5.7 哈希计算与上传读流存在读盘竞争风险

当前 CD2 上传在收到不同类型的哈希请求后，会为不同哈希任务分别启动异步处理；与此同时，远程上传读流本身也会持续读取同一文件。

这意味着在大文件上传场景下，单个文件可能同时出现：

- 上传读流读取
- MD5 全文件读取
- SHA1 全文件读取
- 其他特征哈希读取

如果底层介质是机械硬盘或普通 SSD，这种多路并发大文件读取会显著放大 I/O 竞争，进而增加：

- 读速下降
- 上传通道无响应概率
- 被服务端误判为超时或异常的概率

这不是理论想象，而是当前代码结构已经具备的真实风险，只是是否已经在所有环境中触发，要看文件大小、介质与并发量。

## 6. 文件中心中的页面口径

### 6.1 同步入口文案

文件中心中“同步”入口名称保持不变，但执行提示必须明确真实引擎：

- 同步到 `115`：实际通过 `CloudDrive2` 上传
- 从 `115` 同步到 `LOCAL/NAS`：实际通过 `aria2` 下载
- 非云端之间：实际为中心服务直接复制

### 6.2 任务中心阶段文案

任务中心应优先展示真实阶段，例如：

- `CD2 WaitforPreprocessing / Preprocessing`：文件预处理中
- `CD2 Inqueue`：缓存写入完成，等待云端上传
- `CD2 Transfer`：CloudDrive2 上传中
- `CD2 Finish`：已完成
- `aria2 active`：aria2 下载中
- `aria2 paused`：已暂停
- `aria2 error`：aria2 下载失败

不要把以下状态显示成“已完成”：

- CD2 缓存阶段
- 上传字节数暂时跑满但未进入 `Finish`
- CD2 仅完成本地缓存写入

## 7. 当前支持矩阵

### 7.1 已正式支持

- 文件中心发起真实同步作业
- 文件中心发起真实删除副本作业
- 文件中心发起真实删除资产作业
- 本地 / NAS 之间复制
- 本地 / NAS 上传到 115
- 从 115 下载到本地 / NAS
- 基础暂停、继续、取消、重试
- 基于作业系统的服务重启后重新排队

### 7.2 当前明确不应宣称支持

- CD2 官方离线下载作业的正式业务编排
- 离线下载完成后的自动转存上传流水线
- 云到云正式同步
- 完整的外部任务专用恢复元数据治理
- 网盘链路的专用并发控制配置界面

## 8. 后续改造建议

后续若继续推进“离线下载 + 云盘转存”方向，建议按以下顺序演进：

1. 先把外部任务状态改成结构化持久化
   - `external_task_engine`
   - `external_task_id`
   - `external_task_status`
   - `external_task_payload`
   - `last_external_sync_at`
2. 把 CD2 上传通道改成断线优先重连，而不是本地全量失败
3. 再把 CD2 进度同步改成集中缓存模型
4. 明确区分 `Pause`、`Cancelled`、`NotFound` 的恢复语义，并补齐 Aria2 错误态恢复
5. 收敛哈希计算与上传读流的 I/O 竞争
6. 最后独立新增“离线下载”作业意图，而不是继续复用现有 `REPLICATE`

建议拆成独立阶段：

1. 提交离线下载任务
2. 轮询离线任务真实终态
3. 校验离线产物
4. 启动 CD2 上传
5. 等待上传真实终态
6. 写入副本记录

## 9. 当前版本结论

截至 2026-04-13，文件中心的真实重操作能力应被描述为：

- “文件中心通过中心服务统一创建真实作业，并根据源端点/目标端点选择 `CloudDrive2`、`aria2` 或直接复制三类执行链路。”
- “当前已具备正式的云上传、云下载与非云端复制能力，但还不是完整的离线下载 + 自动转存编排中心。”
- “页面与任务中心的展示必须围绕真实外部状态展开，而不是围绕中间缓存态或本地字节流态展开。”
- “当前系统已经有一批确定存在的恢复与容错边界，后续不能再按顺风局假设继续扩展实现。”
