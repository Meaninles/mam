## 当前客户端实现状态（2026-04-08）

本节用于标注“当前已落地事实”。若与后文设计基线存在差异，以本节与“14.7 当前已实现版本与设计基线差异”为准。

### 已落地事实

- 页头右侧当前已形成两层入口结构：`动态导入入口`、`通知铃铛数字角标`。
- 原“设备插入绿色标签 + 导入中心按钮”已收敛为单一动态导入入口，当前文案口径为：`导入 / 已插入 1 个设备 / 已插入 X 个设备`。
- 导入中心当前已进入顶层多标签工作区，可作为工作区标签参与切换、排序、关闭与恢复，并在切换到其它页面后保留页面状态。
- 当前导入中心已经落地 `ImportDeviceSessionRecord / ImportDraftRecord / ImportSourceNodeRecord / ImportReportSnapshot / ImportTargetEndpointRecord` 等 mock 领域模型，并覆盖多设备、多状态、多报告、多预检、多文件状态组合。
- 当前导入中心主舞台已经具备：待导入设备池、当前设备工作台、来源摘要、目标编排、预检摘要、文件清单、结果摘要与提交区。
- 当前待导入设备列表已收敛为紧凑两行卡片：左侧展示名称、地址与可用空间；右侧提供 4 个图标按钮：打开会话、重新扫描、查看预检、查看详情。
- 当前目标编排已经支持：对某个目标端“应用到全部文件”与“全部取消”，并通过全量 / 部分 / 未导入三态视觉表达当前设备与目标端的覆盖关系。
- 当前提交导入后，任务会继续写入现有统一任务模型，并在“任务中心 > 传输任务”中作为 `IMPORT` 任务跟踪；导入中心同时保留结果摘要回看入口。

### 当前明显不足

- 当前实现仍未完全对齐本文原设计中的“页内标签层”：实际代码已经取消页内子标签栏，改为“左侧设备池 + 右侧当前设备工作台”的单层结构。
- 当前实现虽已进入顶层多标签工作区，但尚未通过左侧导航暴露入口，仍主要通过页头动态导入入口进入。
- 当前设备池列表与目标编排已按生产化工作台方向收敛，但部分基线中提到的更丰富页内会话切换、最近关闭导入页签等交互，在当前实现中仍未单独提供。
- 当前实现仍以客户端 mock 数据驱动为主，尚未接入真实设备事件、真实预检执行器与真实导入报告数据源。

因此，本方案目标不是微调现有导入页，而是将其升级为面向生产环境、对标专业拷卡工作流、并与当前客户端设计语言完全一致的正式导入中心页面。

## 1. 页面定位

“导入中心”是统一文件管理系统客户端中的现场介质导入工作台。

它不是文件中心里的普通上传弹层，也不是任务中心里的任务监控页，而是用于统一承接以下高频导入工作：

- 识别当前接入的读卡器、存储卡、移动硬盘、U 盘
- 统一管理所有“待导入端”
- 为每个待导入端建立独立导入会话
- 将来源文件按导入策略分发到一个或多个目标端
- 在提交前完成预检、冲突提示和风险暴露
- 在提交后衔接任务中心、异常中心和通知中心，形成可追踪、可恢复、可复盘的导入闭环

页面目标是让用户优先回答以下问题：

- 当前接入了哪些现场介质，哪些已经准备好导入
- 某个设备上的文件是否已完成扫描与确认
- 我将把哪些内容导入到哪些目标端
- 提交前是否存在冲突、容量不足、权限异常或路径风险
- 某个导入作业当前进行到了哪一步，是否有失败、漏拷或待补处理项
- 本次导入最终是否可信，结果是否可回看、可汇总、可继续处理

### 1.1 与页头动态入口的关系

导入中心继续保持页头右上角独立入口定位，不进入左侧导航一级页面集合。

但页头入口口径升级为“导入入口与设备接入状态合并”的动态入口，统一承载以下三种状态：

1. 无设备接入时：
   - 入口显示为普通按钮：`导入`
   - 语义是“进入导入中心”
2. 接入 1 个设备时：
   - 入口显示为状态按钮：`已插入 1 个设备`
   - 语义是“当前有一个待处理来源端”
3. 接入多个设备时：
   - 入口显示为状态按钮：`已插入 X 个设备`
   - 语义是“当前有多个待处理来源端”

边界约束：

- 设备接入状态不再以独立绿色标签悬挂在导入中心左侧，而是与导入入口合并为同一交互对象。
- 该入口仍属于页头信号层，不进入通知中心角标统计。
- 点击入口统一打开导入中心，不因设备数量不同跳转到不同页面。

### 1.2 与顶层多标签工作区关系

- 导入中心仍不是左侧导航一级页面，但纳入顶层多标签工作区的一级页面集合，需要在切换时保留导入中心页面状态在顶部多标签工作区。
- 顶层工作区当前承载文件中心、导入中心、任务中心、异常中心、存储节点、设置这 6 个一级页面上下文；其中导入中心仍通过页头动态入口进入，而不是通过左侧导航打开。

### 1.3 与其它页面的关系

- 文件中心：继续保留轻量 `上传文件 / 上传文件夹` 入口，用于普通文件上传；不替代导入中心的专业导入能力。
- 任务中心 > 传输任务：统一承接已提交的 `IMPORT` 任务，负责监控、管理和跳转，不替代导入中心的导入编排能力。
- 异常中心：统一承接导入过程中产生的失败、冲突、校验失败、设备异常、容量异常等治理闭环。
- 通知中心：只承接导入相关的关注入口和跳转，不承接导入治理闭环。
- 存储节点：继续负责目标端、挂载目录、NAS/网盘配置的管理，不承接现场介质导入会话。

导入中心只负责：

- 看设备
- 编排导入
- 做预检
- 发起导入
- 看结果摘要
- 跳到任务中心、异常中心、文件中心继续处理

导入中心不负责：

- 完整任务监控替代
- 完整异常治理替代
- 常驻存储节点管理
- 普通云端上传工作流

### 1.4 对标口径

本方案对标的是专业拷卡软件工作流，而不是通用文件上传页。

因此导入中心在定位上必须具备以下特征：

- 以“设备会话”和“导入作业”为中心，而不是以单个文件选择为中心
- 强调“一次导入，多目标分发”
- 强调“导入前预检、导入后校验、导入结果汇总”
- 强调“失败可恢复、结果可回看、过程可追踪”

首版不追求完整覆盖所有专业 DIT 增强能力，但必须把以上主链路一次定清。

## 2. 设计原则

### 2.1 生产可用优先

本方案以“后续接真实设备接入、真实导入执行与真实异常治理时，不再大改页面结构”为目标。

因此页面的角色划分、核心卡片、状态表达、会话模型、导入结果和跳转契约要尽量一次定清。

### 2.2 现场介质优先

导入中心首版只围绕以下现场介质来源设计：

- 读卡器
- 存储卡
- 移动硬盘
- U 盘

不在首版把范围扩展为“所有待导入来源统一平台”。

### 2.3 设备会话优先

导入中心必须以“设备会话”为核心，而不是以“批次下拉 + 文件勾选”这种轻量表单为核心。

这意味着：

- 页面首先展示当前有哪些待导入端
- 每个待导入端都应有独立上下文
- 同一时间允许用户并行查看多个设备或多个导入草稿

### 2.4 导入作业优先于任务监控

导入中心的主要职责是“建立、确认和提交导入作业”，而不是接管任务中心。

因此：

- 导入中心负责来源、目标、预检、提交、结果摘要
- 任务中心负责运行期管理和长时监控
- 两者通过稳定跳转契约衔接，而不是互相替代

### 2.5 基础闭环优先，专业增强后置

首版先打通基础工作流闭环：

- 设备识别
- 会话建立
- 目标编排
- 预检
- 提交
- 进度衔接
- 校验
- 结果汇总
- 异常进入异常中心

导入预设、自动整理、增强报告、批量转码、远端通知等专业增强能力作为二阶段规划，不压入首版。

### 2.6 与现有客户端风格完全一致

本页面必须延续当前客户端已有设计语言，重点参考：

- 异常中心：正式工作台化、紧凑列表、工具条卡片、详情抽屉
- 任务中心：列表优先、异常提示前置、重治理后移
- 文件中心：表格密度、工具条风格、轻量操作按钮
- 存储节点：页头反馈、紧凑卡片、上下文菜单

明确要求：

- 继续使用当前客户端统一的 `toolbar-card`、`workspace-card`、`compact-list`、`Sheet`、`ActionButton`、`SelectPill`、`TonePill`
- 不引入大屏监控面板、重图表看板或与现有客户端割裂的视觉语言
- 不把导入中心做成完全不同的“媒体管理台”视觉风格

## 3. 业务范围与边界

### 3.1 一阶段纳入范围

一阶段导入中心必须纳入以下能力：

- 现场介质接入识别
- 待导入端列表管理
- 设备会话打开、切换、关闭与恢复
- 来源摘要查看
- 文件与目录清单浏览
- 多目标端编排
- 导入前预检
- 提交导入作业
- 导入结果摘要
- 导入报告快照回看
- 与任务中心、异常中心、通知中心、文件中心的稳定跳转

### 3.2 一阶段不纳入范围

以下能力首版不在导入中心直接完成：

- NAS 热文件夹、网盘待拉取源、云端待导入源的统一接入
- 普通文件上传和普通文件夹上传
- 转码、代理文件生成、媒体预览分发
- 手机、邮箱或第三方 IM 远端通知
- 自动打标签、自动填写项目名、机位、卷号、备注
- 复杂规则预设、目录模板、重命名模板
- 自动重拷失败素材的闭环引擎
- 设备接入后转正式挂载管理的完整流程

### 3.3 与现有产品边界统一结论

- 文件中心保留轻量上传入口，不被导入中心吞并。
- 导入中心继续作为现场介质导入的专业入口。
- 任务中心继续负责已提交导入任务的统一监控与控制。
- 异常中心继续负责导入过程中的异常治理闭环。
- 通知中心继续只做“看与跳”，不承接导入强处理动作。

### 3.4 二阶段预留能力

二阶段建议在不推翻首版模型的前提下逐步补充：

- 导入预设
- 目录模板与命名规则
- 导入后自动打标签、项目名、机位、卷号、备注
- 更正式的导入报告导出
- 自动重拷失败素材
- 更细粒度校验策略
- 批量转码
- 导入完成远端通知
- 设备接入后转正式挂载/纳管的衔接能力

这些能力必须在架构上预留，但不应反向绑架首版页面结构。

### 3.5 后续 NAS 部署演进边界

- 当前阶段仍以 Windows 本机部署口径为准，不改变现有首版导入流程。
- 后续若采用 `Central Core + Local Edge Agent` 架构，Windows 本地设备接入、来源扫描、本地读取 / 传输 / 校验由 `Local Edge Agent` 执行。
- `Central Core` 统一保存待导入端、导入草稿、预检结果、导入任务、异常、通知与报告快照，是唯一权威状态源。
- 导入中心页面继续主要负责工作台 UI；Tauri / Rust 只保留本地桌面 / OS 桥接，不承接导入中心业务核心。
- 因此，后续 NAS 部署应演进为“中央编排 + 本地边缘执行”的两级导入架构，而不是让 NAS 直接扫描 Windows 本地 USB 或移动硬盘。

## 4. 核心业务流程

### 4.1 设备接入与待导入端建立

当系统检测到现场介质接入时，应统一创建“待导入端”记录。

每个待导入端至少应包含：

- 设备标识
- 设备类型
- 挂载路径
- 容量与可用空间
- 接入时间
- 扫描状态
- 当前是否存在未完成导入会话

页头动态入口据此更新为：

- `导入`
- `已插入 1 个设备`
- `已插入 X 个设备`

### 4.2 会话建立与文件清单生成

用户点击某个待导入端后，应创建或聚焦该设备的导入会话标签。

会话建立后执行：

1. 读取来源端基础信息
2. 扫描目录结构与文件清单
3. 生成来源摘要
4. 标记可导入、已导入、疑似重复、异常文件等状态

### 4.3 目标编排与预检

用户在当前会话内为文件集合选择目标端与目标策略。

提交前必须执行统一预检：

- 来源端可读
- 目标端可写
- 目标容量是否充足
- 目标路径是否冲突
- 目标路径是否被其他受管资产占用
- 当前设备是否仍在线
- 当前执行器是否可用

预检结果需要在导入中心中前置展示，而不是把风险全部推迟到提交后再暴露。

### 4.4 导入作业提交与任务衔接

用户确认后，导入中心负责创建导入作业，并将其提交到现有统一任务系统。

提交后：

- 当前会话进入“已提交 / 运行中 / 观察中”语义
- 任务中心中创建对应 `IMPORT` 任务
- 导入中心保留摘要与跳转入口
- 长时间运行管理交给任务中心继续承接

### 4.5 完成汇总与结果回看

导入完成后，导入中心需要保留结果摘要而不是直接“任务已提交就结束”。

至少应包括：

- 导入文件数
- 成功文件数
- 失败文件数
- 部分成功情况
- 校验结果
- 目标端结果摘要
- 报告快照入口

### 4.6 异常与恢复

导入中心可以前置展示异常提示，但异常治理闭环仍统一回到异常中心。

导入中心应承担的轻动作包括：

- 重新预检
- 打开任务中心
- 打开异常中心
- 从最近草稿继续编辑
- 重新打开刚关闭的导入会话

导入中心不应承担的重动作包括：

- 大量批量异常治理
- 路径冲突闭环处理
- 节点鉴权重建
- 强校验治理

## 5. 页面结构

导入中心建议采用“页头动态入口 + 轻工具条 + 双区工作台 + 抽屉/弹层”的四层结构。

### 5.1 应用页头层

应用页头继续由全局页头统一承接。

导入中心相关变化仅体现在右上角动态入口：

- 无设备：`导入`
- 1 个设备：`已插入 1 个设备`
- 多个设备：`已插入 X 个设备`

### 5.2 页内上下文层

当前实现已将导入中心纳入顶层多标签工作区，因此页内不再额外保留一层子多标签栏。

当前页内上下文采用：

- 左侧待导入设备池切换当前设备
- 右侧单工作台承载当前设备的来源摘要、目标编排、预检、文件清单与结果摘要
- 设备详情、预检详情、导入报告快照通过抽屉补充

原方案中的“待导入端 / 设备会话 / 导入摘要”页内标签结构，当前保留为后续可扩展设计基线，不视为 2026-04-08 当前实现已落地能力。

### 5.3 轻工具条层

工具条应保持紧凑，风格参考文件中心和异常中心。

建议从左到右依次为：

1. 会话级筛选
   - `全部`
   - `待扫描`
   - `可导入`
   - `导入中`
   - `异常`
2. 搜索框
3. 排序方式
4. 刷新设备
5. 重新扫描
6. 查看最近导入摘要

说明：

- 工具条不叠加统计大卡
- 批量动作只对“待导入端列表”提供轻量能力，不做复杂治理批量条

### 5.4 主工作区层

主工作区建议采用“左侧待导入端列表 + 右侧当前会话工作台”的双区结构。

左侧区域负责：

- 展示所有待导入端
- 显示设备状态
- 支持打开或聚焦设备会话

右侧区域负责：

- 来源摘要
- 文件清单
- 目标编排
- 预检结果
- 提交导入
- 导入结果摘要

### 5.5 抽屉与弹层层

导入中心内建议保留以下弹层或抽屉：

- 设备详情抽屉
- 预检详情抽屉
- 导入结果摘要抽屉
- 提交确认弹窗
- 放弃草稿确认弹窗
- 目标冲突确认弹窗

这些承载形式应与当前客户端既有 `Sheet`、确认弹窗、轻量详情抽屉保持一致。

## 6. 页面布局与视觉风格

### 6.1 整体布局

整体建议采用如下布局：

1. 工具条卡片
2. 左侧待导入端列表卡片
3. 右侧当前会话工作台卡片组

右侧工作台卡片组建议顺序为：

1. 来源摘要卡
2. 目标编排卡
3. 预检摘要卡
4. 文件清单卡
5. 结果摘要卡（仅已提交或已完成时展示）

### 6.2 信息密度

- 列表应保持紧凑，但必须保证设备名称、路径、状态、容量与异常提示可读。
- 文件清单表格应复用文件中心和任务中心的紧凑表格风格。
- 目标编排区应尽量使用胶囊、按钮组和轻量卡片，不做过度复杂的图形化编排器。

### 6.3 视觉重点

导入中心的视觉重点应始终围绕以下三件事：

- 现在有哪些设备待处理
- 当前会话是否已准备好提交
- 当前导入结果是否可信、是否存在异常

不建议把视觉重心放在：

- 大面积状态统计
- 重图表
- 花哨时间线
- 复杂媒体预览墙

## 7. 页内标签设计

### 7.1 标签类型与语义

页内标签建议至少支持以下三种实例：

1. `待导入端`
   - 常驻默认页签
   - 展示所有设备池
2. `设备会话`
   - 一个设备一条会话上下文
   - 承载导入草稿与预检结果
3. `导入摘要`
   - 承载刚提交或刚完成导入作业的摘要回看

### 7.2 打开与聚焦规则

- 点击右上角动态入口时，若导入中心未打开，则打开并默认聚焦 `待导入端` 页签。
- 点击某个设备时，若对应设备会话已打开，则直接聚焦；否则新开设备会话页签。
- 从任务中心、异常中心、通知中心跳回导入中心时，应优先聚焦对应设备会话或导入摘要页签。

### 7.3 关闭与恢复规则

- `待导入端` 页签默认不可关闭。
- 设备会话与导入摘要页签支持关闭。
- 建议保留最近关闭的 5 条导入页签，用于快速恢复。
- 关闭有未提交草稿的设备会话时，应先弹出确认。

### 7.4 上下文保留规则

设备会话页签切换后，以下状态应在会话内保留：

- 搜索条件
- 文件勾选
- 目标端选择
- 预检结果
- 展开状态
- 提交前草稿

### 7.5 标签状态表达

页签标题旁应允许轻量表达状态：

- `待扫描`
- `可导入`
- `导入中`
- `异常`
- `已完成`

状态表达采用当前客户端既有色彩语义，不新增完全不同的状态系统。

## 8. 待导入端列表设计

### 8.1 列表定位

待导入端列表是导入中心的默认主舞台。

它用于回答：

- 当前有几个现场介质接入
- 哪些设备已可导入
- 哪些设备正在导入或需要处理
- 用户下一步应该打开哪个会话

### 8.2 列表展示规则

列表按设备卡片或紧凑行卡片展示，建议默认按最近接入时间倒序。

每个待导入端至少展示：

- 设备名称
- 设备类型
- 接入路径
- 容量与可用空间
- 接入时间
- 当前状态
- 目录扫描摘要
- 当前是否存在异常

### 8.3 状态定义

待导入端建议支持以下状态：

- `待识别`
- `扫描中`
- `可导入`
- `导入中`
- `部分完成`
- `异常待处理`
- `已拔出`

说明：

- `已拔出` 不等于立即删除记录，应允许用户回看上下文与结果。
- `部分完成` 用于表达设备会话已提交过部分导入作业但仍有待处理内容。

### 8.4 行内操作

待导入端列表建议支持以下轻操作：

- 打开会话
- 重新扫描
- 查看预检结果
- 查看详情

不建议在该列表上直接提供：

- 大量批量治理动作
- 修改目标端强业务动作
- 存储节点配置动作

## 9. 当前会话工作区设计

### 9.1 来源摘要卡

来源摘要卡用于快速展示当前设备会话的基本信息。

建议字段：

- 设备名称
- 设备类型
- 来源路径
- 容量 / 可用空间
- 文件数 / 文件夹数
- 最近扫描时间
- 会话状态

### 9.2 文件清单卡

文件清单卡是会话工作区的主要内容之一。

建议展示：

- 文件名
- 相对路径
- 类型
- 大小
- 当前导入状态
- 是否存在冲突或异常提示

说明：

- 表格密度与文件中心保持一致
- 支持基础搜索
- 支持轻量筛选：`全部 / 可导入 / 已导入 / 冲突 / 异常`
- 首版不要求在导入中心内做复杂目录树导航

### 9.3 目标编排卡

目标编排卡用于配置当前会话的导入目标。

首版建议支持：

- 为当前设备会话选择一个或多个目标端
- 支持对全体文件应用同一目标端
- 支持对单个文件或文件集合做差异化目标选择
- 展示目标端摘要：名称、类型、可写状态、可用空间

### 9.4 预检摘要卡

预检摘要卡用于前置暴露提交前风险。

建议按以下维度汇总：

- 可通过项数量
- 风险项数量
- 阻塞项数量

并支持展开查看：

- 路径冲突
- 空间不足
- 目标不可写
- 来源不可读
- 设备已拔出
- 目标端鉴权异常

### 9.5 提交区

提交区建议位于右侧工作区上部或底部固定区域，避免用户滚动太久后找不到提交入口。

至少包含：

- `提交导入`
- `保存草稿`
- `重新预检`
- 阻塞项摘要

### 9.6 结果摘要卡

当会话已经提交或刚完成时，应展示结果摘要卡。

建议字段：

- 导入作业状态
- 成功文件数
- 失败文件数
- 部分成功数量
- 校验结果
- 最新更新时间
- 打开任务中心
- 打开异常中心
- 查看导入报告

## 10. 状态表达

### 10.1 页头动态入口状态

页头导入入口只承载“是否存在待处理设备”的摘要状态，不直接承载全部异常语义。

建议状态：

- `导入`
- `已插入 1 个设备`
- `已插入 X 个设备`

可选补充：

- 若所有设备均处于异常待处理状态，可在入口上附加轻量风险点提示
- 不建议把完整异常数量直接堆到导入入口上

### 10.2 设备会话状态

设备会话建议支持：

- `待扫描`
- `可导入`
- `预检失败`
- `待提交`
- `导入中`
- `部分成功`
- `已完成`
- `异常待处理`
- `已拔出`

### 10.3 导入作业状态

导入作业正式提交后，应继续复用现有统一任务系统状态语义。

导入中心内展示时可映射为：

- `已排队`
- `运行中`
- `已暂停`
- `部分成功`
- `失败`
- `已完成`
- `已取消`

### 10.4 文件级状态

文件清单内建议支持：

- `待导入`
- `已排队`
- `传输中`
- `校验中`
- `已完成`
- `失败`
- `冲突`
- `已跳过`

## 11. 操作设计

### 11.1 单会话操作

建议支持：

- 打开会话
- 保存草稿
- 重新扫描
- 重新预检
- 提交导入
- 查看任务
- 查看异常

### 11.2 轻量批量操作

首版仅建议在“待导入端列表”提供轻量批量能力：

- 批量刷新设备状态
- 批量重新扫描

不建议在首版支持：

- 批量提交多个设备的复杂导入作业
- 批量修改目标端策略
- 批量治理异常

### 11.3 跳转操作

导入中心必须支持稳定跳转：

- 去任务中心查看当前导入任务
- 去异常中心处理当前导入异常
- 去文件中心查看已导入资产
- 去存储节点查看目标端状态

### 11.4 能力型按钮规则

按钮能力必须遵循互斥和前置条件规则：

- 存在阻塞预检项时，`提交导入` 禁用
- 会话处于 `导入中` 时，提交区主按钮切换为 `查看任务`
- 已形成结果时优先显示结果摘要卡和报告快照入口，而不是额外增加 `查看摘要` 主按钮
- 已拔出设备会话不允许重新提交，但允许查看草稿和结果

## 12. 空态、加载态与错误态

### 12.1 无设备空态

当当前没有任何现场介质接入时，空态文案应明确提示：

- 当前没有可导入设备
- 可插入读卡器、存储卡、移动硬盘或 U 盘后继续
- 仍可查看最近导入摘要

### 12.2 筛选结果为空

当存在设备但当前筛选结果为空时，应提示：

- 当前筛选下没有匹配的待导入端或会话
- 可调整状态筛选、搜索词或排序方式

### 12.3 加载态

建议区分：

- 设备识别中
- 来源扫描中
- 预检执行中
- 结果汇总中

不建议所有状态都统一显示为模糊的“加载中”。

### 12.4 错误态

页面级错误建议仅用于以下情况：

- 导入中心初始化失败
- 设备池读取失败
- 当前会话数据无法恢复

其余设备级、文件级、目标级问题尽量下沉为局部错误或异常提示，而不是整页报错。

## 13. 一致性要求

### 13.1 与文件中心的一致性

- 表格密度、搜索框、筛选胶囊、轻量按钮风格与文件中心保持一致
- 导入中心不替代文件中心的轻量上传入口

### 13.2 与任务中心的一致性

- 导入提交后必须映射到 `IMPORT` 任务
- 长时间运行管理、暂停、继续、重试等能力口径与任务中心一致
- 导入中心不重复发明第二套任务状态系统

### 13.3 与异常中心的一致性

- 导入中心可以前置展示异常摘要，但重治理统一回异常中心
- 导入异常跳转时应复用现有聚焦参数契约

### 13.4 与通知中心的一致性

- 导入相关状态提醒继续遵循“看与跳”原则
- 导入入口动态状态不计入通知铃铛角标
- 通知中心不承接导入治理动作

### 13.5 与存储节点的一致性

- 导入中心只消费目标端可用状态，不承担存储端配置职责
- 目标端异常可跳到存储节点继续查看

### 13.7 明确不建议的设计

- 把导入中心做成普通上传弹层
- 把导入中心做成任务中心副本
- 在导入中心内塞入完整异常治理闭环
- 为导入中心发明完全不同的视觉系统
- 在首版中一次性塞入转码、远端通知、自动标签、复杂规则引擎

## 14. 技术架构设计

### 14.1 设计目标

导入中心的技术架构目标是：

- 在现有统一任务系统之上补齐“待导入端”和“设备会话”层
- 不推翻既有 `IMPORT` 任务模型
- 与任务中心、异常中心、通知中心保持稳定契约
- 支撑一阶段生产可用，同时为二阶段专业能力预留演进空间

### 14.2 核心领域模型建议

#### 14.2.1 待导入端模型 `ImportDeviceSessionRecord`

用于表达一个现场介质来源端在导入中心中的稳定身份。

建议字段方向：

- `id`
- `deviceKey`
- `deviceLabel`
- `deviceType`
- `mountPath`
- `connectedAt`
- `lastSeenAt`
- `capacitySummary`
- `scanStatus`
- `sessionStatus`
- `activeDraftId`
- `latestReportId`
- `issueIds`

#### 14.2.2 导入草案模型 `ImportDraftRecord`

用于表达某个设备会话在提交前的导入编排草稿。

建议字段方向：

- `id`
- `deviceSessionId`
- `libraryId`
- `selectedFileIds`
- `targetEndpointIds`
- `targetStrategy`
- `precheckSummary`
- `lastEditedAt`
- `hasBlockingIssues`
- `status`

#### 14.2.3 预检模型 `ImportPrecheckSummary`

用于表达导入提交前的统一预检结果。

建议字段方向：

- `blockingCount`
- `riskCount`
- `passedCount`
- `checks`
  - `sourceReadable`
  - `targetWritable`
  - `capacityReady`
  - `pathConflict`
  - `deviceOnline`
  - `executorReady`

#### 14.2.4 导入结果快照模型 `ImportReportSnapshot`

用于在导入中心回看一次导入作业的结果摘要。

建议字段方向：

- `id`
- `deviceSessionId`
- `taskId`
- `submittedAt`
- `finishedAt`
- `successCount`
- `failedCount`
- `partialCount`
- `verifySummary`
- `targetSummaries`
- `issueIds`

#### 14.2.5 页头入口摘要模型 `ImportEntrySignal`

用于统一表达右上角动态入口的状态。

建议字段方向：

- `connectedDeviceCount`
- `importingDeviceCount`
- `attentionDeviceCount`
- `label`
- `tone`

### 14.3 前端视图模型

#### 14.3.1 页头入口模型 `ImportEntryBadgeVM`

负责把设备池状态映射成页头动态入口显示。

#### 14.3.3 待导入端列表模型 `ImportDeviceListItemVM`

负责待导入端列表中的设备摘要、状态与跳转能力。

#### 14.3.4 当前会话工作台模型 `ImportSessionWorkspaceVM`

负责右侧工作台中的来源摘要、目标编排、预检、结果摘要等整体装配。

#### 14.3.5 文件清单模型 `ImportSourceNodeVM`

负责来源文件与目录在导入中心中的展示与状态映射。

### 14.4 查询与命令分离

#### 查询接口

建议至少包含：

- 查询页头导入入口摘要
- 查询待导入端列表
- 查询指定设备会话详情
- 查询指定导入草稿
- 查询指定导入结果摘要
- 查询最近导入摘要列表

#### 命令接口

建议至少包含：

- 打开或恢复设备会话
- 重新扫描设备
- 更新导入草稿
- 执行预检
- 提交导入作业
- 保存草稿
- 关闭会话
- 恢复最近关闭会话

### 14.5 与现有任务、异常、通知体系的衔接

- 导入中心提交后，正式执行仍进入现有 `IMPORT` 任务模型。
- 导入异常继续映射为现有 `IssueRecord`，由异常中心承接治理。
- 导入相关提醒继续映射为现有页头信号和通知快照，不单独造第三套消息体系。
- 文件中心与任务中心跳回导入中心时，应继续复用既有跨页聚焦参数设计。
- 后续若核心迁移到 NAS，`ImportDeviceSessionRecord / ImportDraftRecord / ImportPrecheckSummary / ImportReportSnapshot` 的正式状态应统一存放在 `Central Core`，`Local Edge Agent` 只负责回传设备事实、进度与本地执行结果。

### 14.6 刷新与恢复策略

建议区分以下几类刷新：

- 页头入口摘要：轻量轮询或事件触发刷新
- 设备池列表：打开导入中心时刷新，驻留期间低频刷新
- 当前设备会话：切换页签时刷新，提交前后强制刷新
- 导入结果摘要：提交后短周期刷新，完成后转低频

恢复策略建议：

- 导入中心关闭再打开时，恢复上次活跃页内标签
- 已拔出设备的会话仍保留最近结果与草稿快照
- 最近关闭会话可恢复
- 后续 NAS 部署时，页头入口摘要、设备池状态、预检结果与结果摘要也应优先从 `Central Core` 读取；若涉及 Windows 本地设备实时状态，则由 `Central Core` 汇总 `Local Edge Agent` 上报结果后统一下发给客户端。

### 14.7 当前已实现版本与设计基线差异

当前客户端已实现：

- 动态页头导入入口（`导入 / 已插入 1 个设备 / 已插入 X 个设备`）
- 导入中心进入顶层多标签工作区，并保留页面状态
- 待导入设备池与当前设备工作台双区结构
- 待导入设备紧凑两行卡片列表
- 设备详情、预检详情、导入报告快照抽屉

## 16. 必须定稿的约束

以下内容在后续实现前应视为定稿，不建议反复修改：

- 导入中心继续保持页头独立入口，不进入左侧导航一级页面集合
- 页头原“设备标签 + 导入按钮”合并为单一动态导入入口
- 一阶段范围仅覆盖读卡器、存储卡、移动硬盘、U 盘等现场介质
- 文件中心轻量上传入口继续保留
- 导入中心负责导入编排，不负责完整任务监控与异常治理闭环
- 导入异常继续统一进入异常中心
- 导入任务继续统一进入任务中心
- 导入中心首版不承接转码、远端通知、复杂预设和自动整理

## 17. 结论

本方案将客户端“导入中心”明确为一个位于页头右上角、面向现场介质、可生产化演进的正式导入工作台。

它在业务上统一收敛待导入端管理、设备会话管理、导入编排、预检、结果摘要与跨页衔接；在交互上保持“页头独立入口”的结构；在界面上严格延续当前客户端既有风格、信息密度与工作台语言；在技术上通过补齐待导入端、导入草案、预检摘要、结果快照和页头入口模型，为后续真实设备接入、真实任务执行和二阶段专业能力扩展提供稳定边界。
