## 1. 页面定位

“任务中心 > 其他任务”是统一文件管理系统客户端中的非传输后台任务监控与管理工作台。

它不是新任务发起页，也不是异常处置页，而是用于统一承接和管理以下四类已经发起的后台运行任务：

- 扫描任务 `SCAN`
- 元数据解析任务 `METADATA_EXTRACT`
- 校验任务 `VERIFY`
- 删除清理任务

其中“删除清理任务”在运行时可由两类底层任务承接：

- 删除执行任务 `DELETE`
- 后台清理任务 `ASSET_CLEANUP`

页面目标是优先回答以下问题：

- 当前有哪些非传输后台任务正在运行
- 哪些任务失败了、暂停了、需要我介入
- 某个任务现在进行到哪一阶段
- 这个任务影响了哪些文件、目录、资产或端点
- 这个任务是否产生异常，以及我应该去哪里处理

## 2. 设计原则

### 2.1 完全继承传输任务子页面的页面骨架

本页必须完全参考“任务中心 > 传输任务”子页面的设计风格、UI、UX、布局与交互语言。

明确要求：

- 继续使用当前客户端既有的 `toolbar-card`、`workspace-card`、`compact-list`、`Sheet`、`ActionButton`、`SelectPill`、`TabSwitch`、`TonePill`
- 列表卡片、详情抽屉、异常提示、工具条密度与传输任务页保持同一套视觉规则
- 不单独发明新的“大屏监控风格”“运维控制台风格”或明显不同的信息架构

### 2.2 后台运行任务优先

本页只纳入“后台运行任务”，但不是所有后台任务都纳入本页。

纳入原则必须同时满足：

- 用户可感知
- 持续时间通常超过瞬时反馈
- 需要查看状态、进度、结果或错误原因
- 需要支持暂停、继续、取消、重试、恢复中的至少一部分
- 会影响正式资产状态、媒体信息、校验结果或删除结果

### 2.3 列表优先，详情抽屉承接深查

本页与传输任务页一致，列表是主舞台，抽屉是深查容器。

因此以下内容都应服从列表主视图：

- 顶部过滤工具条
- 批量操作条
- 异常提示图标
- 详情抽屉

不应让统计卡、图表或复杂筛选器压过任务列表本身。

### 2.4 任务类型只作为筛选，不作为子页面

本页不按“扫描 / 元数据解析 / 校验 / 删除清理”拆成多个子页面。

这四类仅作为同页内的任务类型筛选维度。

### 2.5 异常前移展示，异常处置后移到异常中心

本页可以：

- 展示异常数量
- 展示异常摘要
- 提供跳转异常中心入口

本页不负责：

- 直接处置冲突
- 直接修改异常决策
- 将异常处理流程塞回列表或抽屉

## 3. 业务范围与边界

### 3.1 纳入范围

本页纳入以下四类任务：

- `SCAN` 扫描任务
- `METADATA_EXTRACT` 元数据解析任务
- `VERIFY` 校验任务
- `DELETE / ASSET_CLEANUP` 删除清理任务

其中：

- 扫描任务包含对挂载目录、受管目录或指定范围的后台扫描
- 元数据解析任务包含照片、视频、音频等媒体资产的异步解析
- 校验任务包含独立发起的校验任务跟踪，不重复承接已经属于传输任务内部阶段的校验过程
- 删除清理任务包含删除提交后的删除执行与 `PENDING_DELETE` 后续清理

### 3.2 不纳入范围

以下后台行为不进入本页：

- 对账扫描任务 `RECONCILE_SCAN`
- 索引任务
- 修复 / 恢复任务
- 连接测试
- 存储端心跳与在线探测
- 标签统计更新、标签检索更新
- staging 清理、缓存整理、短时内部维护任务
- CloudDrive2、aria2 等外部执行器的内部原生任务
- 异常记录本身
- 数秒内完成、用户无需感知的微任务

### 3.3 与其它页面的关系

- 存储节点页：扫描任务的主要发起页之一，仍保留“立即扫描”“查看扫描历史”等入口
- 文件中心：可触发刷新索引、删除、查看解析结果、查看校验状态等相关动作，但不承接后台执行监控
- 任务中心 > 传输任务：负责导入与同步等跨端链路任务，本页负责非传输后台任务
- 异常中心：负责统一处置异常，本页只负责展示异常提示与跳转

### 3.4 当前客户端需明确的边界结论

- 任务中心现有顶层仍保留“传输任务 / 其他任务”切换
- “其他任务”不再是简陋占位列表，而应升级为生产可用子页面
- 当前仅定义四类页面任务类型，不额外引入对账扫描、索引、修复恢复等新类型

## 4. 页面结构

页面建议与传输任务子页面保持同一套四层结构：

### 4.1 页头层

沿用任务中心既有页头，不在本子页内重新定义应用级头部。

### 4.2 子页切换层

保留任务中心当前的一级切换：

- `传输任务`
- `其他任务`

### 4.3 工具条层

“其他任务”子页面内部工具条建议从左到右依次为：

1. 任务类型筛选
   - `全部`
   - `扫描`
   - `元数据解析`
   - `校验`
   - `删除清理`
2. 状态筛选
3. 资产库筛选
4. 排序方式
5. 搜索框
6. 批量操作入口

说明：

- “任务类型筛选”是过滤器，不是二级子页
- 继续使用与传输任务页一致的 `TabSwitch + SelectPill + 搜索框` 结构
- 工具条保持轻量，不新增统计卡带

### 4.4 列表主体层

列表主体继续采用与传输任务页一致的“任务卡片 + 紧凑行布局 + 可展开二级区”的组织方式。

默认只展示一级任务。

用户手动展开后，在当前任务卡片内展示二级任务区域。

### 4.5 详情抽屉层

详情抽屉继续使用右侧 `Sheet` 样式，承担深查而非主工作区职责。

## 5. 页面布局与视觉风格

### 5.1 整体布局

建议继续采用与传输任务页一致的结构：

1. 工具条卡片
2. 列表卡片
3. 详情抽屉

不新增独立左侧树，不做常驻右侧详情栏，不做仪表盘式头图区。

### 5.2 信息密度

信息密度应与当前文件中心、存储节点页、传输任务页一致，偏紧凑但必须保持可读。

### 5.3 视觉焦点

页面视觉焦点顺序建议为：

1. 任务名称与任务类型
2. 当前状态与当前阶段
3. 进度与处理速度
4. 异常提示
5. 可操作按钮

不应把以下内容做成视觉主角：

- 汇总统计卡
- 大型图表
- 低频筛选项

## 6. 列表设计

### 6.1 列表展示规则

默认仅展示一级任务。

用户点击展开按钮后，在当前任务卡片下方展开二级任务区。

二级任务用于承接：

- 扫描任务的分段扫描项、目录项或文件项
- 元数据解析任务的文件项
- 校验任务的文件项或端点校验项
- 删除清理任务的删除项、清理项或资产项

三级任务统一只在详情抽屉中查看，不在列表里继续无限展开。

### 6.2 一级任务行字段

一级任务行建议固定包含以下字段：

- 选择框
- 展开按钮
- 任务名称
- 范围路径小字或范围摘要小字
- 任务类型标签：`扫描` / `元数据解析` / `校验` / `删除清理`
- 进度条
- 当前阶段摘要
- 范围胶囊
  - 例如：`1 个目录`、`128 个文件`、`34 项资产`
- 处理速度
  - 例如：`120 项/秒`、`85 MB/s`、`42 文件/分`
- 剩余时间
- 状态胶囊
- 优先级
- 异常提示图标
- 操作区

说明：

- 不显示外部执行器名称
- 不显示内部数据库主键
- 不显示不稳定的技术细节字段
- 不新增与传输任务页风格冲突的复杂列

### 6.3 一级任务摘要规则

一级任务摘要建议按任务类型分别生成：

- 扫描：显示扫描范围、已处理项数、是否进入提交阶段
- 元数据解析：显示媒体类型范围、已完成解析数、当前解析阶段
- 校验：显示校验模式、校验范围、已通过 / 已失败摘要
- 删除清理：显示删除来源、受影响资产数、是否进入后台清理

一级摘要必须与详情抽屉中的数据口径保持一致。

### 6.4 二级任务展开区字段

二级任务展开区建议采用轻量内嵌行列表，字段为：

- 名称
- 名称下方路径
- 节点类型
  - `目录`
  - `文件`
  - `资产`
  - `端点`
  - `清理步骤`
- 状态
- 当前阶段
- 进度
- 大小或数量摘要
- 异常提示图标

说明：

- 二级区不显示完整一级任务字段
- 二级区不放完整批量工具条
- 仅保留少量状态按钮和查看能力
- 已删除关联文件或已失效对象继续沿用当前任务页的红灰失效态与删除线语义

### 6.5 删除清理任务的特殊展示要求

删除清理任务在列表层应明确区分两个阶段：

- 删除执行中
- 后台清理中

当任务已经进入 `PENDING_DELETE` 后续清理时，列表不得仍展示为普通“删除中”。

应显式显示“等待清理”或“清理中”的阶段语义。

## 7. 异常提示与异常浮窗

### 7.1 异常提示图标

继续沿用传输任务页的“小红色数字图标”方案。

规则：

- 无异常不显示
- 有异常显示红色异常徽标与数字
- 数字上限显示 `99+`

### 7.2 异常浮窗内容

异常浮窗展示当前任务关联的异常摘要列表，建议字段为：

- 严重级别
- 异常类型
- 简短说明
- 所属子项名称
- 当前处置状态

### 7.3 异常浮窗操作

浮窗底部固定保留：

1. `查看全部异常`
   - 跳转异常中心
   - 自动带上 `taskId`
2. 单条异常点击
   - 跳转异常中心
   - 自动定位到对应异常记录

说明：

- 本页不直接处理异常
- 异常中心筛选实现可后续补齐，但跳转契约应先定型

## 8. 详情抽屉设计

### 8.1 抽屉定位

详情抽屉是“深度查看与局部操作”容器，不是主要工作区。

列表仍然是主舞台。

### 8.2 抽屉结构

建议分为四个区块：

#### 1. 任务概览

- 任务名称
- 任务类型
- 当前状态
- 当前阶段
- 优先级
- 发起来源
- 所属资产库
- 总体进度
- 当前速度
- 剩余时间
- 创建时间
- 最近更新时间

#### 2. 执行范围

- 范围类型
  - 全库 / 指定端点 / 指定目录 / 指定资产 / 指定文件集合
- 作用端点
- 根路径
- 文件数 / 目录数 / 资产数
- 校验模式
- 解析目标类型
- 删除模式

#### 3. 子项明细

以树状或分层列表展示：

- 一级：主任务
- 二级：分组 / 目录 / 文件 / 资产 / 端点
- 三级：具体文件项或步骤项

补充字段包括：

- 状态
- 当前阶段
- 进度
- 大小
- 速度
- 结果摘要

#### 4. 异常与结果摘要

- 当前任务异常总数
- 最近异常
- 失败项数量
- 已完成项数量
- 跳转文件中心
- 跳转异常中心

### 8.3 抽屉操作

抽屉顶部或底部操作区建议保留：

- `暂停`
- `继续`
- `取消`
- `重试`
- `修改优先级`
- `查看文件中心`
- `查看异常中心`

说明：

- 所有动作按能力动态显示
- 不展示当前状态不支持的按钮

## 9. 筛选、排序与搜索

### 9.1 任务类型筛选

固定为：

- `全部`
- `扫描`
- `元数据解析`
- `校验`
- `删除清理`

说明：

- 仅为筛选，不代表二级子页面
- 删除清理筛选需要同时命中 `DELETE` 与 `ASSET_CLEANUP`

### 9.2 状态筛选

建议支持以下筛选顺序：

- `全部`
- `活跃中`
- `待执行`
- `运行中`
- `等待确认`
- `已暂停`
- `异常待处理`
- `等待清理`
- `部分成功`
- `失败`
- `已完成`
- `已取消`

其中：

- `活跃中` 是聚合筛选，不是底层任务状态
- `活跃中` 建议聚合 `运行中 / 已暂停 / 等待确认 / 等待清理`
- `等待清理` 主要用于删除清理任务的页面表达

### 9.3 资产库筛选

支持按资产库过滤任务。

### 9.4 搜索

搜索建议覆盖：

- 任务名称
- 根路径
- 文件名
- 目录名
- 资产名
- 端点名
- 异常摘要

### 9.5 排序

推荐默认排序规则：

1. `运行中`
2. `失败`
3. `异常待处理`
4. `等待清理`
5. `已暂停`
6. `等待确认`
7. `待执行`
8. `部分成功`
9. `已完成`
10. `已取消`

同状态下再按：

1. 优先级
2. 创建时间倒序

界面可提供的排序方式建议为：

- `默认排序`
- `优先级`
- `进度`
- `名称`

## 10. 状态表达

### 10.1 主任务状态

主任务建议统一使用以下页面状态语义：

- `待执行`
- `运行中`
- `等待确认`
- `已暂停`
- `异常待处理`
- `等待清理`
- `部分成功`
- `已完成`
- `失败`
- `已取消`

其中：

- `活跃中` 只作为筛选状态
- `等待清理` 是删除清理任务的重要页面态

### 10.2 子项状态

子项建议允许使用更细粒度状态：

- `待预检`
- `扫描中`
- `解析中`
- `校验中`
- `删除中`
- `清理中`
- `提交中`
- `重试等待`
- `已暂停`
- `已完成`
- `失败`
- `已取消`

### 10.3 类型阶段表达

四类任务建议统一有“当前阶段”字段，但阶段文本按类型区分：

- 扫描任务：
  - `准备扫描`
  - `枚举目录`
  - `写入临时结果`
  - `归并提交`
  - `更新目录节点`
  - `清理临时数据`
- 元数据解析任务：
  - `识别媒体类型`
  - `读取元数据`
  - `写入媒体子表`
  - `更新解析状态`
- 校验任务：
  - `准备校验`
  - `读取文件`
  - `比对摘要`
  - `写入校验结果`
  - `更新校验摘要`
  - `等待时间窗`
- 删除清理任务：
  - `提交删除请求`
  - `删除副本`
  - `写入删除状态`
  - `等待清理`
  - `清理扩展数据`
  - `完成移除`

### 10.4 显示规则

- 页面展示的是系统统一任务状态，不直接暴露底层原始状态
- 即使底层某个步骤完成，只有本地统一状态进入完成语义后才展示为 `已完成`
- `校验中` 不得显示为完成
- `等待清理` 不得显示为完成
- `部分成功` 必须单独可见，不可被归并为完成

## 11. 操作设计

### 11.1 单任务操作

首版支持：

- 暂停
- 继续
- 取消
- 重试
- 修改优先级
- 打开详情

说明：

- 按能力互斥渲染 `暂停 / 继续 / 重试`
- 已完成显示 `完成` 按钮占位
- 不可取消的任务应禁用或隐藏取消按钮

### 11.2 批量操作

首版支持的批量操作：

- 批量暂停
- 批量继续
- 批量取消
- 批量重试
- 批量修改优先级
- 清空选择

批量能力按选中任务能力交集计算。

### 11.3 跳转操作

首版支持以下跳转：

- 跳转异常中心
- 跳转文件中心
- 从扫描任务跳转存储节点相关对象

说明：

- 跳转文件中心优先带上相关路径、资产或文件定位条件
- 跳转异常中心优先带 `taskId`

## 12. 空态、加载态与错误态

### 12.1 无任务空态

当没有任何其他任务时，空态文案建议提示：

- 可在存储节点发起扫描任务
- 文件入库后将自动触发元数据解析任务
- 可在文件中心或后续校验入口查看校验 / 删除结果

### 12.2 筛选结果为空

当只是当前筛选下无结果时，应提示：

- 当前筛选条件下没有匹配任务
- 支持一键清空筛选

### 12.3 加载态

首屏进入和切换筛选时，建议使用轻量骨架屏，不使用大面积转圈。

### 12.4 错误态

任务列表加载失败时：

- 在列表卡片内展示错误提示
- 提供 `重试` 按钮

## 13. 一致性要求

### 13.1 与传输任务页的一致性

必须保持以下一致：

- 顶层工具条结构
- 卡片式列表容器
- 行内任务卡片布局
- 异常图标与异常浮窗样式
- 详情抽屉结构与按钮风格
- 批量操作条位置与交互节奏

### 13.2 与文件中心的一致性

必须保持以下一致：

- 搜索框样式
- 筛选器样式
- 行内按钮风格
- 抽屉密度

### 13.3 与存储节点页的一致性

必须保持以下一致：

- 子页切换方式
- 页内紧凑布局
- 轻量反馈模式

### 13.4 明确不建议的设计

以下设计不应采用：

- 单独做运维大屏
- 大型统计图占据首屏
- 将四类任务拆成四个二级页面
- 将异常处理流程直接塞进本页
- 在列表里无限层级展开
- 为“其他任务”单独发明与传输任务页完全不同的视觉语言

## 14. 技术架构设计

### 14.1 设计目标

- 先完成前端 UI 定型，后续接真实后端时不改主要 UI / UX
- 与当前 `TaskCenterPage` 顶层结构兼容
- 与现有 `TaskRecord / TaskItemRecord / IssueRecord` 模型渐进兼容
- 支持后续真实任务持久化、轮询、推送、恢复与异常联动
- 与传输任务页保持相同的组件分层方法

### 14.2 页面任务类型模型

### 14.5 状态与阶段映射

建议统一使用一层页面映射：

- `PENDING` -> `待执行`
- `RUNNING` -> `运行中`
- `PAUSED` -> `已暂停`
- `COMPLETED` -> `已完成`
- `FAILED` -> `失败`
- `CANCELED` -> `已取消`
- `PARTIAL_SUCCESS` -> `部分成功`

页面派生状态：

- `活跃中`
- `等待清理`

说明：

- `等待清理` 可由 `DELETE / ASSET_CLEANUP` 的阶段和上下文派生
- 不强制所有任务底层新增一个独立数据库状态值
- `等待确认`、`异常待处理` 等页面状态应继续与现有任务中心状态体系兼容

### 14.6 列表与详情的数据装载策略

为保证后续生产环境性能，建议分层加载：

#### 首屏列表

只加载一级任务摘要。

#### 二级任务展开

用户展开一级任务时，再按需加载该任务二级节点。

#### 详情抽屉

用户打开详情抽屉时，再加载：

- 完整任务详情
- 完整节点树
- 三级节点
- 异常摘要列表

### 14.7 刷新与实时同步策略

建议采用分层刷新：

- `运行中`：高频轮询或推送
- `已暂停 / 待执行 / 等待清理`：中频刷新
- `已完成 / 已取消 / 失败`：低频刷新或不自动刷新

说明：

- 页面永远以本地统一任务状态为准
- 后端、扫描服务、删除清理队列只是状态事实来源

### 14.8 异常中心联动模型

建议继续沿用稳定关联键：

- `taskId`
- `taskItemId`
- `issueId`
- `libraryId`
- `assetId`
- `endpointId`

用途：

- 当前页显示异常数量与异常浮窗
- 异常中心按任务筛异常
- 文件中心按资产回查任务和异常

### 14.9 与当前客户端实现的演进建议

建议在现有 `client/src/pages/TaskCenterPage.tsx` 和 `client/src/data.ts` 基础上做渐进演进：

#### 1. 保留顶层页签模型

- 继续保留 `TaskTab = 'transfer' | 'other'`
- 不新增顶层导航

#### 2. 收缩“other”分支的特殊分叉逻辑

当前“other”分支只是简单行列表，后续应重构为与传输任务页同级别的完整渲染器，而不是继续在同一分支里堆条件渲染。

建议拆分：

- `TransferTaskBoard`
- `OtherTaskBoard`
- `TaskCenterToolbar`
- `TaskCenterDetailSheet`

#### 3. 扩展 `TaskRecord`

建议为非传输任务补充：

- `otherTaskType?: 'SCAN' | 'METADATA_EXTRACT' | 'VERIFY' | 'DELETE_CLEANUP'`
- `stageLabel?: string`
- `scopeLabel?: string`
- `scopePath?: string`
- `resultSummary?: string`
- `waitingReason?: string`

#### 4. 扩展 `TaskItemRecord`

建议补充：

- `resultLabel?: string`
- `missing?: boolean`
- `nodeKind?: 'directory' | 'file' | 'asset' | 'endpoint' | 'step'`

#### 5. 保持现有能力判断模式

继续沿用现有 `canPause / canResume / canRetry / canCancel` 风格，但从状态硬编码逐步演进为能力字段驱动。

### 14.10 删除清理任务的技术口径

页面层统一称为“删除清理任务”。

实现层建议遵循：

- 用户删除后先生成 `DELETE` 任务
- 若进入 `PENDING_DELETE`，再由 `ASSET_CLEANUP` 接力
- 页面可将两者聚合为同一主任务的两个阶段，也可展示为同组主从任务

但无论采取哪种实现方式，页面都必须保证用户看到的是完整且连续的删除清理体验。

## 16. 必须定稿的约束

以下内容在后续实现前应视为定稿：

- “其他任务”只纳入 `扫描 / 元数据解析 / 校验 / 删除清理`
- 对账扫描、索引、修复恢复不纳入本页
- 任务类型只作为筛选，不拆成多个二级页面
- 页面骨架、视觉风格、交互节奏必须完全参考传输任务子页面
- 顶部继续使用工具条卡片，不增加统计卡大区
- 默认只展示一级任务
- 二级任务允许在列表中手动展开
- 三级节点只在详情抽屉查看
- 删除清理必须明确展示“等待清理 / 清理中”阶段
- 异常提示继续使用红色数字图标与浮窗
- 页面只做异常提示与跳转，不做异常处置

## 17. 结论

本方案将“任务中心 > 其他任务”明确为一个以列表为核心的生产可用后台任务监控台。

它在业务上只收敛扫描、元数据解析、校验、删除清理四类非传输后台任务，在界面上严格继承传输任务子页面的结构、风格与交互语言，在技术上通过稳定的页面任务类型、视图模型、能力模型和查询命令边界，为后续真实后端接入与任务恢复机制保留足够空间。

按本方案实现后，后续接入真实扫描服务、元数据解析链路、校验任务和删除清理队列时，应尽量只替换数据来源与命令实现，而不再改动该子页面的主要 UI、UX 和信息架构。
