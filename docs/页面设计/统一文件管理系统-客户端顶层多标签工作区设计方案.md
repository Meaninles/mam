## 1. 页面定位

客户端顶层多标签工作区不是新增业务页面，而是客户端一级模块之上的“工作区承载层”。
它位于当前应用页头区域，用浏览器顶部标签页的交互方式，承接以下一级模块的并行切换：

- 文件中心
- 任务中心
- 异常中心
- 存储节点
- 设置

设计目标不是做成通用路由系统，而是在不改变现有一级模块定位的前提下，解决当前“进入新页面后旧页面被卸载、再次打开要重新加载”的问题，让用户能够在多个主工作台之间快速往返，同时保留各自上下文。

该能力必须满足以下总目标：

- 方便用户：多个一级页面可并行打开并快速切换
- 保留上下文：切回时尽量回到离开前状态
- 控制性能：不采用所有页面永久常驻的粗暴方案
- 保持边界清晰：只在客户端内实现，不与后端服务耦合

## 2. 设计原则

### 2.1 只服务一级页面

多标签能力只面向左侧导航栏的一级页面，不纳入子页面、二级页签、详情抽屉、弹窗、向导和其它局部工作流。
这与产品文档中的一级信息架构保持一致，也能显著降低状态模型复杂度。

### 2.2 单页面单实例

同一个一级页面在同一时刻只允许存在一个标签实例。
若该页面已打开，用户再次点击左侧导航时，行为应为“聚焦已有标签”，而不是重复新开。

### 2.3 浏览器式体验，但不照搬浏览器复杂度

标签栏视觉和交互应参考浏览器顶部标签页，包括：

- 当前标签高亮
- 支持关闭
- 支持拖拽排序
- 支持右键菜单
- 支持移动前后位置

但不引入当前阶段不必要的复杂能力，例如重复标签、固定标签、跨窗口同步等。

### 2.4 客户端内闭环

该能力是纯前端 / 客户端工作区能力：

- 不新增后端接口依赖
- 不要求后端保存标签状态
- 不把标签顺序、激活页、工作区快照设计为服务端权威状态

当前客户端如继续使用 `sql.js`、`localStorage`、内存状态或其它本地 mock 机制，均不影响本方案成立。

### 2.5 快切优先，资源受控

切换体验要明显优于当前“卸载后重进”的方式，但也不能把所有页面无上限长期保活。
推荐采用“有限保活 + 状态快照”的混合方案，在用户体验和性能之间做稳定平衡。

## 3. 业务范围与边界

### 3.1 纳入范围

- 左侧导航一级页面的打开、切换、关闭、排序
- 顶部标签栏展示与交互
- 每个一级页面各自的工作区状态保存
- 标签右键菜单
- 标签拖拽排序与左右移动
- 最近关闭标签的恢复
- 有限保活与页面快照恢复
- 现有跨页跳转逻辑向“打开或聚焦标签”模型收口

### 3.2 不纳入范围

- 导入中心
- 文件详情抽屉
- 批量标记弹层
- 标签编辑弹层
- 存储节点内部子页
- 任务中心内部“传输任务 / 其它任务”切换
- 设置页内部子页
- 任意页面中的对话框、悬浮层、右侧详情面板
- 重启应用后的标签恢复
- 与后端服务同步标签状态

### 3.3 与其它页面的关系

- 文件中心、任务中心、异常中心、存储节点、设置仍保持现有一级模块职责不变
- 导入中心仍是独立入口，但不进入本次顶层多标签体系
- 任务中心与异常中心、存储节点与任务中心、文件中心与任务中心之间的跳转，后续都应统一通过“打开或聚焦标签”完成

### 3.4 本方案明确结论

- 只允许一级页面进入标签栏
- 同一一级页面不允许重复打开
- 点击左侧导航时，已打开则聚焦，未打开则新开
- 不要求重启后恢复标签
- 需保留页面内的筛选、搜索、分页、滚动位置、当前二级页签、已选项、表单草稿等上下文
- 需支持浏览器式右键菜单和标签顺序移动
- 所有实现均在客户端内完成，不与后端服务耦合

## 4. 页面结构

### 4.1 页头层

延续现有客户端页头结构，仍保留：

- 左侧反馈信息区
- 右侧导入中心、通知等全局入口

在此基础上，于页头中部新增“顶层标签栏层”。

### 4.2 顶层标签栏层

标签栏位于当前页头中部、你已确认的红圈区域，承担一级页面切换。
标签栏的视觉语义应是“浏览器式工作区”，而不是页面内部的普通分段切换器。

标签项包含：

- 页面图标
- 页面名称
- 关闭按钮
- 当前激活态
- 可选未保存标记

### 4.3 页面视口层

标签栏下方仍是当前主内容区。
内容区一次只展示一个激活标签对应的页面，但非激活标签不一定立即销毁，而由保活策略决定是否继续挂载。

### 4.4 右键菜单层

在标签项上右键时弹出统一上下文菜单，用于承接批量关闭、位置移动、刷新等操作。
右键菜单是标签体系的重要组成，不应退化为浏览器原生 `contextmenu`。

## 5. 标签交互设计

### 5.1 打开规则

左侧导航点击规则如下：

1. 若目标页面未打开，则新建标签并激活
2. 若目标页面已打开，则直接切换到该标签
3. 不允许生成重复标签

新标签默认插入在当前激活标签右侧。
这样既保持浏览器式顺序感，也避免所有新标签都堆到最末尾。

### 5.2 关闭规则

- 点击标签关闭按钮可关闭当前标签
- 右键菜单可关闭当前、关闭其它、关闭左侧、关闭右侧
- 关闭当前标签后，优先激活右侧相邻标签；若无右侧，则激活左侧标签
- 若关闭后已无任何标签，则自动打开并激活“文件中心”

### 5.3 排序规则

标签应支持两种排序方式：

- 鼠标拖拽直接调整顺序
- 右键菜单中的“向左移动”“向右移动”

排序调整只影响当前会话内的标签顺序，不做跨重启持久化要求。

### 5.4 最近关闭规则

客户端内维护最近关闭标签栈，用于“重新打开刚关闭的标签”。
由于顶层页面总数有限，最近关闭栈保留最近 8 次关闭记录即可。

恢复规则：

- 恢复时按关闭前的页面类型重建标签
- 恢复时优先回填该页面的最后工作区快照
- 若没有可用快照，则使用该页面默认初始状态

### 5.5 标签右键菜单

建议标签右键菜单至少包含以下项：

- 刷新当前页面
- 关闭当前标签
- 关闭其他标签
- 关闭左侧标签
- 关闭右侧标签
- 重新打开刚关闭的标签
- 向左移动
- 向右移动

禁用规则：

- 若左侧无标签，则“关闭左侧标签”“向左移动”置灰
- 若右侧无标签，则“关闭右侧标签”“向右移动”置灰
- 若无最近关闭记录，则“重新打开刚关闭的标签”置灰
- 当仅剩一个标签且该标签为当前唯一工作区时，“关闭当前标签”可执行，但结果应回到默认打开的文件中心，不出现空白主区域

## 6. 页面状态保留设计

### 6.1 必须保留的状态

每个一级页面都应保留自己的工作区状态。
其中至少包括：

- 筛选条件
- 搜索词
- 分页与每页数量
- 排序字段与方向
- 当前二级页签或分段选择
- 已选项
- 滚动位置
- 打开的详情抽屉 / 面板及对应对象
- 未提交表单草稿

### 6.2 可以不保留的瞬时状态

以下状态不要求跨标签切换保留：

- hover 态
- tooltip 展开态
- 右键菜单是否打开
- toast / feedback 的短暂展示
- 纯装饰性动画过程

### 6.3 各页面状态边界

#### 文件中心

应至少保留：

- 当前资产库
- 当前目录
- 面包屑 / 前进后退历史
- 搜索、筛选、排序、分页
- 已选文件
- 当前滚动位置
- 文件详情抽屉或批量操作上下文
- 新建目录草稿、标签编辑草稿、批量标记草稿

#### 任务中心

应至少保留：

- 当前顶层子页签
- 状态筛选、类型筛选、排序、搜索
- 已选任务
- 当前滚动位置
- 当前打开的详情抽屉

#### 异常中心

应至少保留：

- 当前筛选条件
- 搜索词
- 分页与滚动位置
- 当前打开的异常详情或联动上下文

#### 存储节点

应至少保留：

- 当前子页
- 搜索、筛选、分页
- 当前滚动位置
- 当前编辑抽屉或表单草稿

#### 设置

应至少保留：

- 当前设置子页
- 当前滚动位置
- 未提交的设置草稿
- 标签管理页中的当前分组、搜索条件、详情面板上下文

## 7. 页面布局与视觉风格

### 7.1 整体布局

标签栏应与现有客户端浅色、圆角、轻阴影风格保持一致，但要明确区别于普通 segmented control。
建议：

- 标签容器整体为浅底色圆角轨道
- 当前标签与非当前标签有明显层级差
- 当前标签可略微抬起或拥有更清晰边界
- 标签与右侧全局按钮之间留出呼吸空间

### 7.2 信息密度

标签区是高频操作区，应比内容区更紧凑：

- 标签高度略低于页面卡片高度
- 单标签宽度允许自适应，但需设置最小值与最大值
- 名称过长时省略并支持悬浮完整标题

### 7.3 视觉重点

视觉重点应遵循：

- 先看到当前激活标签
- 再看到其它已打开工作区
- 最后看到右侧全局按钮

不建议把标签做成普通按钮列表，也不建议使用过重的色块或多层描边，使页头变得嘈杂。

## 8. 空态、加载态与错误态

### 8.1 标签栏空态

理论上不会出现真正空态。
当用户关闭到最后一个标签时，应自动回到文件中心，并立即显示一个文件中心标签。

### 8.2 页面切换加载态

切换到保活中的标签时，应即时显示，无额外骨架屏。
切换到仅保存快照的标签时，可先显示上次快照，再触发局部刷新。
仅在确实没有可用快照时，才显示页面自身加载态。

### 8.3 页面刷新错误态

标签刷新失败不应导致标签关闭，也不应清空已保留的工作区状态。
应维持页面现有错误提示方式，并允许用户再次刷新。

## 9. 一致性要求

### 9.1 与产品信息架构一致

本方案严格对齐产品文档中的一级模块结构：

- 文件中心
- 任务中心
- 异常中心
- 存储节点
- 设置

不把子页面、子页签、详情抽屉提升为顶层标签。

### 9.2 与现有页面设计一致

- 文件中心仍是“统一资产 + 目录导航”的主工作台
- 任务中心仍保留自身内部页签，不被顶层标签替代
- 存储节点仍保留“挂载文件夹管理 / NAS 管理 / 网盘管理”内部切换
- 设置仍保留自身子页体系与标签管理页

顶层多标签只解决“一级页面并行切换”，不改变各页内部设计。

### 9.3 与技术架构现状一致

技术架构文档中当前客户端仍处于前端原型阶段，文件中心与标签管理等能力已在客户端内通过 `sql.js`、`localStorage` 和本地状态实现。
因此本方案默认：

- 顶层多标签首先在客户端内落地
- 不等待 Go 核心服务或服务端接口完成
- 不将标签管理纳入后端权威状态

### 9.4 明确不建议的设计

- 不建议把每个子页面、详情页都做成顶层标签
- 不建议继续围绕单一 `activeView` 叠补逻辑
- 不建议把所有标签永久常驻挂载
- 不建议将标签状态与业务数据缓存混为一层
- 不建议把顶层多标签与导入中心强行合并

## 10. 技术架构设计

### 10.1 设计目标

前端技术方案必须同时满足：

- 顶层标签可打开、切换、关闭、排序
- 每个一级页面拥有独立工作区状态
- 非激活页面切回时尽量秒恢复
- 性能可控，不因多标签导致不必要的持续渲染和副作用堆积
- 与后端服务零耦合

### 10.2 前端状态分层

建议将现有顶层状态拆成四层：

#### 1. 标签元数据层

负责：

- 已打开哪些标签
- 当前激活标签是谁
- 标签顺序
- 最近关闭记录

建议模型：

```ts
type WorkspaceView = 'file-center' | 'task-center' | 'issues' | 'storage-nodes' | 'settings';

interface WorkspaceTabMeta {
  id: WorkspaceView;
  title: string;
  closable: boolean;
  createdAt: number;
  lastActivatedAt: number;
}
```

#### 2. 页面工作区状态层

负责：

- 各页面自己的筛选、分页、滚动位置、抽屉状态、草稿等

建议模型：

```ts
interface WorkspaceStateMap {
  'file-center': FileCenterWorkspaceState;
  'task-center': TaskCenterWorkspaceState;
  issues: IssuesWorkspaceState;
  'storage-nodes': StorageNodesWorkspaceState;
  settings: SettingsWorkspaceState;
}
```

#### 3. 页面数据缓存层

负责：

- 列表查询结果缓存
- 详情数据缓存
- 根据本地状态签名复用已有结果

该层不等同于工作区状态。
工作区状态描述“用户停留在哪里”，数据缓存描述“当前可复用哪些已取回的数据”。

#### 4. 视口保活层

负责：

- 哪些页面继续挂载
- 哪些页面仅保留快照并卸载组件

### 10.3 推荐标签状态模型

建议使用 reducer 或等价的集中式状态机管理标签行为：

```ts
interface WorkspaceTabsState {
  activeTabId: WorkspaceView;
  order: WorkspaceView[];
  openTabs: Record<WorkspaceView, WorkspaceTabMeta>;
  recentlyClosed: WorkspaceView[];
}
```

建议命令集合：

- `openOrFocusTab(view)`
- `closeTab(view)`
- `closeOtherTabs(view)`
- `closeTabsToLeft(view)`
- `closeTabsToRight(view)`
- `moveTab(view, direction)`
- `reorderTabs(from, to)`
- `reopenLastClosedTab()`
- `refreshTab(view)`

### 10.4 页面工作区模型建议

现有 [App.tsx](B:/new_project/mare/client/src/App.tsx) 中的单例状态需要收敛为按页面归属的工作区状态。

例如 `FileCenterWorkspaceState` 建议至少包含：

```ts
interface FileCenterWorkspaceState {
  libraryId: string;
  currentFolderId: string | null;
  folderHistory: Array<string | null>;
  historyIndex: number;
  searchText: string;
  fileTypeFilter: string;
  fileStatusFilter: string;
  partialSyncEndpointNames: string[];
  fileSort: string;
  fileSortDirection: 'asc' | 'desc';
  currentPage: number;
  pageSize: number;
  selectedFileIds: string[];
  scrollTop: number;
  openDrawer: 'detail' | 'tag-editor' | 'batch-annotation' | 'new-folder' | null;
  draft: Record<string, unknown>;
}
```

其它页面也应采用同样策略，不再在 `App.tsx` 顶层只放一份 `taskTab`、`settingsTab`、`issueTypeFilter` 之类的全局单例状态。

### 10.5 查询与命令分离

该方案仍建议延续当前页面设计文档常用的“查询与命令分离”思路，但范围限定在客户端内：

#### 查询职责

- 获取标签顺序与激活页
- 获取某个页面的工作区状态
- 获取某个页面的缓存数据
- 获取保活集合

#### 命令职责

- 打开 / 聚焦 / 关闭标签
- 调整标签顺序
- 写入页面工作区状态
- 刷新指定页面
- 更新最近关闭栈

### 10.6 保活与性能策略

由于本方案禁止重复页面实例，顶层标签理论最大数量就是一级导航页数量。
因此“总打开数量”天然受控，真正需要控制的是“保活数量”。

建议规则如下：

- 顶层标签总数上限：固定为一级导航页数量
- 默认保活数量：`3`
- 保活策略：`当前激活 1 个 + 最近访问 2 个`
- 其它非激活页面：保留工作区快照，但可卸载组件

这样做的原因：

- 文件中心和任务中心页面较重，不宜全部永久常驻
- 用户最常往返的通常也是最近使用的少数页面
- 一级页面总量有限，保活 3 个即可获得明显优于当前实现的切换体验

### 10.7 刷新与缓存策略

建议对每个页面采用“快照优先、局部刷新”的客户端策略：

- 切换回保活页：直接显示
- 切换回非保活页但有快照：先显示快照，再局部刷新
- 无快照：走页面默认加载流程

缓存触发应仅来自客户端自身动作，例如：

- 切换资产库
- 修改筛选条件
- 执行删除、同步、编辑、扫描等本地动作
- 用户主动刷新当前页面

不要求后端推送参与多标签管理。

### 10.8 跨页联动模型

现有跨页跳转逻辑后续应统一改成：

```ts
openOrFocusTab(view, {
  patchWorkspaceState,
});
```

示例：

- 从任务中心打开异常中心时，聚焦或新开异常中心，并注入对应任务过滤条件
- 从存储节点查看相关任务时，聚焦或新开任务中心，并切到“其它任务”或对应筛选态
- 从任务详情回到文件中心时，聚焦或新开文件中心，并回填目标目录与选中项

这样所有联动都在客户端壳层闭环，不需要后端参与标签调度。

## 11. 实现建议

### 11.1 推荐组件拆分

建议在当前前端结构下增加以下模块：

- `client/src/components/WorkspaceTabBar.tsx`
  - 顶层标签栏
- `client/src/components/WorkspaceViewport.tsx`
  - 页面视口与保活切换
- `client/src/components/WorkspaceTabContextMenu.tsx`
  - 标签右键菜单
- `client/src/lib/workspaceTabs.ts`
  - 标签 reducer、命令与最近关闭逻辑
- `client/src/lib/workspaceState.ts`
  - 各页面工作区状态模型与更新器
- `client/src/lib/workspaceRegistry.ts`
  - 一级页面标题、图标、默认状态、组件映射

### 11.2 当前代码结构下的职责调整

#### `App.tsx`

应新增职责：

- 承载顶层标签壳层
- 组织页头、标签栏、页面视口
- 统一收口左侧导航点击行为
- 协调跨页联动

应收缩职责：

- 不再持有大量只存在一份的页面单例状态
- 不再通过 `activeView === ... ? <Page /> : null` 直接主导所有页面生命周期

#### 各页面组件

应改为“受控工作区页面”：

- 从工作区状态读取页面上下文
- 通过更新器写回页面状态
- 不依赖自己是唯一实例，但也不要求支持重复实例

#### 本地数据访问层

现有 `client/src/lib/clientState.ts`、`client/src/lib/fileCenterApi.ts` 等数据层可继续复用。
顶层多标签不应重写业务数据层，只需增加客户端工作区壳层与状态分层。

### 11.3 样式复用建议

- 继续复用现有圆角、轻阴影、浅色边框风格
- 标签 hover、active、drag-over 态通过已有按钮与卡片色板延展，不引入新的视觉体系
- 不使用浏览器默认原生拖拽截图作为最终交互效果，建议做轻量定制高亮
- 右键菜单沿用当前客户端更多操作菜单的视觉风格，保持统一

## 12. 必须定稿的约束

- 顶层多标签仅面向左侧导航一级页面
- 同一一级页面只允许一个标签实例
- 导入中心不进入顶层标签体系
- 不要求跨重启恢复标签和工作区
- 页面上下文需保留筛选、搜索、分页、滚动位置、当前二级页签、已选项、表单草稿
- 标签支持关闭、拖拽排序、左右移动、右键菜单、最近关闭恢复
- 总打开数量天然受一级页面数约束，不再额外设计更高上限
- 性能控制重点是保活数量，而不是总打开数量
- 所有实现均在客户端内完成，不与后端服务耦合

## 13. 结论

客户端顶层多标签工作区应被定义为一级页面之上的浏览器式工作区壳层，而不是零散补丁。
其正确落点是“标签元数据 + 页面工作区状态 + 数据缓存 + 有限保活”四层结构。

在当前项目中，这一方案既符合产品信息架构，也符合现有前端原型阶段的技术现实：

- 不改变文件中心、任务中心、异常中心、存储节点、设置的业务职责
- 不把导入中心、子页面、抽屉和弹层混入顶层标签
- 不依赖后端改造
- 能明显改善用户频繁跨页切换时的体验
- 在性能上比“全部页面永久常驻”更稳妥

因此，后续 AI 编码实现应以本文作为顶层多标签能力的页面设计与前端架构基线。
