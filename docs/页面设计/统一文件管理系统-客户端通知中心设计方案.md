- 跨页跳转复用 `issueId / taskId / taskItemId / libraryId / endpointId / fileNodeId / path` 契约。
- 从通知中心跳转异常中心时，若携带 `issueId`，当前会自动聚焦对应异常并打开详情抽屉；若仅携带任务、来源或路径上下文，则保持列表聚焦。
- mock 数据已覆盖设备插入、提醒类、处置类、已读、已跳转、已失效等组合场景。

## 1. 页面定位

客户端“通知中心”不是新的顶层页面，也不是“异常中心”的轻量副本，而是位于应用页头右侧、由通知按钮打开的右侧浮窗式关注入口。

它的职责只有两类：

- 让用户快速知道“当前有哪些真正值得注意的问题”
- 把用户准确送到异常中心、文件中心、任务中心、存储节点或导入中心的对应上下文

通知中心不负责承接完整异常治理，不负责批量处理，不负责历史清理，也不负责展示所有异常存量。

完整异常治理仍然统一落在“异常中心”；通知中心只承接“新到的重要异常通知”和“极少量入口级系统提醒”。

### 1.1 与页头其它信号的关系

当前客户端页头右侧已有 `导入中心` 与 `通知` 两个入口。

本方案将页头信号明确拆成两层：

1. 设备接入提醒：
   - 放在导入中心左侧
   - 使用绿色小标签
   - 表达“某个移动硬盘 / U 盘已插入”
   - 不进入右侧通知浮窗
2. 通知中心：
   - 由铃铛按钮打开右侧浮窗
   - 只承接真正重要的异常通知
   - 通过数字角标表达未消费通知数量

### 1.2 与顶层多标签工作区关系

- 通知中心浮窗不进入顶层多标签体系
- 通知中心本身不是一级页面，不生成新的顶层标签
- 通知中心中的跳转行为必须复用现有顶层多标签规则：
  - 目标页面已打开时直接聚焦
  - 目标页面未打开时新开对应一级页面
  - 跳转携带的一次性聚焦请求在目标页消费后失效

### 1.3 与异常中心的分工

两者必须明确分工：

- 通知中心：只做“看”和“跳”
- 异常中心：做“筛选、聚合、批量治理、轻处理、历史回查”

通知中心中不应直接承接以下动作：

- 重试
- 延后处理
- 忽略
- 刷新检测
- 归档
- 批量处理
- 冲突闭环处理

这些能力继续保留在异常中心或原业务页面。

## 2. 设计原则

### 2.1 最少打扰优先

通知中心必须优先减少噪音，而不是追求“覆盖所有异常”。

只有真正会影响用户当前工作流、数据可信度、关键决策或近期可执行动作的问题，才允许进入通知中心。

换言之：

- 不是所有异常都要通知
- 不是所有风险都要打断用户
- 不是所有页面反馈都要进入页头关注入口

### 2.2 三层信号分层优先

本方案将客户端关注信号统一拆为三层：

1. 页头即时提醒：
   - 仅承接设备接入类环境变化提示
2. 通知中心右侧浮窗：
   - 仅承接重要异常通知
3. 异常中心：
   - 承接完整异常集合与治理工作流

三层之间不互相替代，也不重复堆叠。

### 2.3 完全继承传输任务子页面的页面骨架

通知中心虽然不是“任务中心 > 传输任务”子页面，但其设计风格、UI、UX、布局与交互语言必须完全参考该子页面，并与当前客户端既有页面保持一致。

明确要求：

- 继续使用当前客户端既有的 `toolbar-card`、`workspace-card`、`compact-list`、`Sheet`、`ActionButton`、`SelectPill`、`TonePill`
- 浮窗内部的结构、信息密度、按钮位置、列表节奏和抽屉风格必须沿用传输任务子页面口径
- 不单独发明新的消息中心风格、社交消息列表风格或移动端推送中心风格
- 不引入统计大卡、复杂分栏、重图表面板或常驻右栏详情

### 2.4 列表优先

通知中心浮窗中的主要内容必须是“通知列表”。

因此以下内容都应从属于列表，不应喧宾夺主：

- 顶部统计卡
- 高级筛选区
- 多维仪表盘
- 常驻详情区
- 大面积说明文案

### 2.5 看与跳优先

通知中心中的主要交互只有三种：

- 查看当前通知摘要
- 打开目标页面
- 标记已读

任何超出这个边界的治理动作，都应跳回异常中心或原业务页面完成。

### 2.6 与现有客户端风格完全一致

通知中心必须延续当前客户端已有设计语言，重点参考：

- 任务中心 > 传输任务子页面：工具条卡片、紧凑列表、异常提示、右侧 `Sheet`
- 任务中心 > 其他任务子页面：完全继承传输任务子页面骨架的约束
- 文件中心：页头入口组织、搜索框、列表密度
- 存储节点：页头轻量反馈、紧凑布局、低频动作放入菜单

## 3. 业务范围与边界

### 3.1 信号分层

#### 3.1.1 页头设备接入提醒

仅纳入：

- 移动硬盘已插入
- U 盘已插入

表现方式：

- 绿色小标签
- 位置在导入中心左侧
- 示例文案：`现场移动硬盘 T7 已插入`

边界约束：

- 不进入通知中心浮窗
- 不计入铃铛数字角标
- 不视为异常
- 点击后跳转导入中心或后续定义的存储接入上下文
- 当前阶段设备接入可由 Windows 本机直接感知；后续若核心迁移到 NAS，则由 `Local Edge Agent` 感知设备并回传给 `Central Core`，再由 `Central Core` 生成统一 `HeaderSignal`。

#### 3.1.2 通知中心右侧浮窗

只纳入“真正值得用户关注的重要通知”，且必须符合通知准入白名单。

#### 3.1.3 异常中心

继续纳入完整异常集合，包括：

- 冲突
- 传输
- 校验
- 节点与权限
- 容量与资源
- 扫描与解析
- 清理与治理

通知中心只从异常中心对应的异常模型中筛出一小部分高价值记录映射为通知，不原样平铺所有异常。

### 3.2 通知准入白名单

只有满足以下任一条件的异常，才允许进入通知中心：

1. 会直接阻塞当前或近期核心操作
2. 会影响数据正确性或可信度
3. 会让用户继续操作时做出错误判断
4. 需要用户尽快做人工决策

不满足以上条件的异常，即使进入异常中心，也不进入通知中心。

### 3.3 纳入范围

#### 3.3.1 处置类通知

默认纳入以下类型：

- 文件名冲突
- 路径冲突
- 双端变更冲突
- 目标路径占用
- 外部未纳管文件覆盖风险
- 上传失败
- 下载失败
- 网络中断后未恢复
- 外部任务丢失或状态漂移
- 校验失败
- 强校验失败
- 原文件损坏
- 鉴权失效
- 节点读写权限异常
- 节点不可达
- 空间不足
- 扫描失败
- 元数据解析失败
- 删除清理阻塞

纳入原则：

- 仅推送当前仍需要用户关注的活跃问题
- 仅推送对当前或近期操作真正有影响的问题
- 长期沉积但近期没有新的状态变化、没有新的关注价值的问题，不在通知中心持续占位

#### 3.3.2 提醒类通知

默认纳入以下类型：

- 鉴权提醒
- 空间预警
- 元数据字段缺失
- 后台治理待确认项

动态纳入以下类型：

- 临时文件待处理
- 历史版本残留

动态规则：

- 默认不进入通知中心
- 当它们已经明显影响当前工作流、目录判断、容量判断或近期动作决策时，升级为提醒类通知
- 若后续继续升级到阻塞主流程，则进一步升级为处置类通知

### 3.4 不纳入范围

以下内容不进入通知中心：

- 普通任务进度变化
- 暂停 / 继续 / 等待清理 / 已完成等普通任务状态变化
- 页面内即时成功反馈
- 存储节点页头中的保存成功、连接测试成功、删除成功等轻反馈
- 已解决、已忽略、已归档的历史异常
- 仅用于调试、监控或统计的内部信号
- 自动重试尚未超出重试窗口、暂时不需要人工介入的瞬时失败

## 4. 页面结构

通知中心不是完整页面，但其浮窗内部结构必须沿用传输任务子页面的轻量结构，并在当前版本收敛为“页头信号层 + 浮窗头部层 + 轻工具条层 + 列表主体层”。

### 4.1 页头信号层

页头右侧信号从左到右建议为：

1. 设备接入绿色标签
2. 导入中心按钮
3. 通知按钮与数字角标

说明：

- 设备接入标签为临时环境信号
- 导入中心仍保持独立入口
- 通知按钮打开右侧 `Sheet`

### 4.2 浮窗头部层

通知中心浮窗头部继续使用当前 `Sheet` 风格，不新增特殊页头语言。

头部当前实现包含：

- 标题：`通知中心`
- 关闭按钮沿用现有 `Sheet` 行为

### 4.3 轻工具条层

浮窗内部当前使用轻量工具条卡片，风格参考传输任务子页面工具条。

当前从左到右为：

1. 搜索框
2. 单一筛选控件（下拉选项：`全部通知 / 未消费 / 处置类 / 提醒类`）

说明：

- 搜索与筛选并列一行，搜索在前、筛选在后。
- 当前版本不提供独立来源筛选与排序控件。
- 不增加统计卡带，不增加复杂高级筛选。

### 4.4 列表主体层

浮窗主体必须优先展示通知列表。

每条通知作为一个可追踪、可打开、可消费的通知单元，不是简单的文本提示。

默认仅展示当前仍有效的通知记录：

- 活跃处置类通知
- 未消费提醒类通知
- 最近已读但仍保留在列表中的通知

### 4.5 详情承载方式

通知中心不新增常驻详情区，也不新增独立通知详情页面。

当需要深查时，统一通过跳转进入：

- 异常中心
- 任务中心
- 文件中心
- 存储节点
- 导入中心

## 5. 页面布局与视觉风格

### 5.1 整体布局

通知中心浮窗内部建议采用与传输任务子页面一致的结构：

1. 工具条卡片
2. 列表卡片

不采用以下设计：

- 社交消息流样式
- 独立左导航树
- 常驻详情栏
- 顶部统计大卡
- 重图表仪表盘

### 5.2 信息密度

列表行密度应参考当前文件中心、存储节点和传输任务子页面，偏紧凑但可读。

原则：

- 主信息一眼可读
- 次级信息用副标题和小胶囊表达
- 非核心信息不新增列

### 5.3 视觉重点

通知中心的视觉焦点顺序必须是：

1. 通知标题与问题类型
2. 当前状态与优先级
3. 来源对象与影响范围摘要
4. 主动作按钮

不应让以下内容成为视觉主角：

- 统计数量
- 历史说明
- 长段文字

## 6. 列表设计

### 6.1 列表展示规则

默认列表仅显示一级通知记录。

一条通知记录代表一个可消费的关注单元，而不是异常中心记录的原样复制。

### 6.2 一级通知行字段

一级通知行当前固定包含以下字段：

- 通知标题（单行省略，悬浮可查看完整标题）
- 通知摘要（最多两行省略，悬浮可查看完整摘要）
- 影响对象与来源文本（对象标签 + 来源标签）
- 底部信息与操作区（左侧时间，右侧主动作与更多操作）

补充说明：

- 提醒类/处置类/已读/未消费语义在当前版本由无障碍隐藏文本承载，不在视觉层显式展示标签。
- 当前版本不显示严重级别胶囊、所属资产库文本和显式状态标签。

### 6.3 标题与摘要规则

标题区当前采用“标题 + 摘要”两段结构：

- 第一行：通知标题（单行）
- 第二段：通知摘要（最多两行）

示例表达：

- `片头配乐_v4_master.wav 强校验失败`
- `影像 NAS 01 写入权限异常`

### 6.4 影响对象摘要

影响对象摘要建议按来源动态生成：

- 传输任务通知：显示任务标题、源端点 -> 目标端点
- 其他任务通知：显示任务标题、当前阶段、对象范围摘要
- 文件中心通知：显示资产或目录路径
- 存储节点通知：显示挂载名称、节点名称、挂载路径
- 系统提醒：显示设备名称或系统环境对象

### 6.5 操作区

通知行内操作区当前支持：

- 主动作按钮
- `更多`

主动作按钮规则（已落地）：

- 处置类：`去处理`
- 提醒类：`去查看`

`更多` 菜单当前保留低频动作，按能力动态显示：

- 标记已读
- 打开异常中心
- 打开任务中心
- 打开文件中心
- 打开存储节点
- 打开导入中心

### 6.6 行内状态语义承载（当前实现）

- 行内视觉状态主要通过“未读高亮样式 + 主动作按钮语义”表达。
- 提醒类/处置类/已读/未消费等状态语义通过无障碍隐藏文本承载，供可访问性与语义识别使用。
- 设备已插入语义不进入通知行，而由页头绿色设备标签承载。

## 7. 排序、搜索与空态

### 7.1 排序

默认排序建议为：

1. 处置类优先于提醒类
2. 阻塞型优先于风险型
3. 严重级别高优先于低
4. 最近发生时间倒序

### 7.2 搜索

搜索范围建议包括：

- 标题
- 摘要
- 来源标签
- 资产库名称
- 对象路径或节点名称

### 7.3 空态

无通知空态建议区分两类：

- 若当前无任何有效通知：提示“当前没有需要优先关注的通知”
- 若是筛选结果为空：提示“当前筛选条件下没有匹配通知”

### 7.4 加载态

建议继续使用轻量骨架屏，不使用大面积加载转圈。

### 7.5 错误态

通知列表加载失败时：

- 在列表卡片内显示错误提示
- 提供 `重试` 按钮

## 8. 通知分类、优先级与生命周期

### 8.1 通知分类

通知中心只保留两类通知：

- `处置类通知`
- `提醒类通知`

### 8.2 已读规则

提醒类通知：

- 用户打开通知中心浮窗后，当前所有未读提醒类通知即视为已读
- 已读后从角标中扣除
- 记录仍可保留在通知列表中

处置类通知：

- 仅在以下两种情况下消角标：
  - 用户手动标记已读
  - 用户点击跳转到目标处理页面
- 仅打开通知中心浮窗，不自动清除处置类通知角标

### 8.3 状态语义

通知中心建议独立使用以下消费状态语义：

- `未消费`
- `已读`
- `已跳转`
- `已失效`

说明：

- 这些状态只作用于通知记录
- 不等于异常中心中的 `待处理 / 待确认 / 处理中 / 已解决 / 已归档`

### 8.4 角标规则

铃铛按钮上的数字角标统计“未消费通知数”，而不是“异常总数”。

统计范围：

- 未读提醒类通知
- 未读且未跳转的处置类通知

不统计：

- 设备接入绿色标签
- 已读提醒类通知
- 已手动已读或已跳转的处置类通知
- 已失效通知

显示规则：

- 0：不显示数字角标
- 1-99：显示实际数字
- 超过 99：显示 `99+`

### 8.5 去重规则

同一异常不应因轮询或状态刷新反复生成多条通知。

建议规则：

- 同一 `issueId` 只保留一条活跃通知
- 同一设备接入事件只保留一条当前会话信号
- 状态变化时更新原通知，不追加重复新记录

### 8.6 升级与降级规则

通知类型允许动态升级或降级：

- `临时文件待处理`
- `历史版本残留`

默认：

- 不进入通知中心，或以提醒类进入

升级条件：

- 已明显影响当前工作流、目录判断、容量判断或近期动作决策

升级结果：

- 由“无通知”升级为提醒类
- 或由提醒类升级为处置类

降级条件：

- 风险解除
- 影响消退
- 后续不再需要用户关注

降级结果：

- 退出通知中心
- 保留在异常中心继续历史治理

## 9. 与其他页面的联动设计

### 9.1 与异常中心联动

通知中心跳转异常中心时，必须复用现有聚焦参数：

- `issueId`
- `taskId`
- `taskItemId`
- `libraryId`
- `endpointId`
- `fileNodeId`
- `path`

跳转效果：

- 自动切换到异常中心
- 自动定位对应异常
- 若跳转参数中携带 `issueId`，当前实现会自动打开对应异常详情抽屉
- 不重复新开已有标签

### 9.2 与任务中心联动

传输任务或其他任务来源的通知，应可直接跳转任务中心，并自动带入：

- `taskId`
- `taskItemId`
- `issueId`
- `openIssuePopover`

说明：

- 若问题更适合在异常中心看全量上下文，主动作优先跳异常中心
- 若问题更适合回任务上下文查看，更多操作可提供“打开任务中心”

### 9.3 与文件中心联动

文件、目录、同步与删除副本相关通知，应支持跳转文件中心，并携带：

- `libraryId`
- `fileNodeId`
- `path`

### 9.4 与存储节点联动

节点鉴权、写权限、连通性与空间类通知，应支持跳转存储节点，并携带：

- `endpointId`
- `path`

### 9.5 与导入中心联动

设备接入绿色标签点击后，应优先跳转导入中心。

后续若实现更明确的设备接入流程，也可在导入中心中继续引导进入存储节点接入或挂载配置。

补充架构边界：

- 后续若采用 `Central Core + Local Edge Agent` 架构，通知中心与页头信号层读取的仍应是 `Central Core` 汇总后的状态。
- `Local Edge Agent` 负责设备与本地执行事实的上报，但不直接向通知中心写入通知或信号记录。

## 10. 一致性要求

### 10.1 与传输任务子页面的一致性

通知中心必须保持以下一致：

- 工具条卡片结构
- 紧凑列表卡片密度
- 搜索框形式
- 胶囊与按钮语义
- 轻量浮层节奏
- 右侧 `Sheet` 风格

### 10.2 与其他任务子页面的一致性

通知中心必须遵循“完全继承传输任务子页面页面骨架”的同类原则：

- 不单独发明新的信息架构
- 不破坏既有客户端风格
- 不让通知中心显得像独立产品

### 10.3 与文件中心的一致性

应保持以下一致：

- 页头入口位置
- 搜索框风格
- 行内按钮与菜单风格
- 紧凑列表密度

### 10.4 与存储节点的一致性

应保持以下一致：

- 页头右侧信号布局
- 轻量反馈方式
- 低频动作进入菜单

### 10.5 明确不建议的设计

以下设计不应采用：

- 把通知中心做成完整异常中心副本
- 把所有异常原样平铺进通知流
- 把普通任务状态变化做成页头通知
- 把通知中心做成社交消息流或移动推送列表
- 在通知中心中承接异常治理动作闭环
- 为通知中心发明与当前客户端割裂的视觉语言

## 11. 技术架构设计

### 11.1 设计目标

技术方案要保证：

- 先把 UI/UX 定型，后接后端时不需要改主要界面结构
- 通知记录与异常记录分离，但可稳定映射
- 设备接入提醒与异常通知共用统一页头信号层，但生命周期独立
- 跳转继续复用既有跨页定位键
- 去重、升级、降级与角标聚合规则可稳定落地

### 11.2 核心领域模型建议

#### 11.2.1 页头信号模型 `HeaderSignal`

建议区分页头信号，而不是把所有信号都塞进通知列表。

建议字段：

- `id`
- `signalType`
  - `DEVICE_INSERTED`
- `tone`
- `title`
- `summary`
- `createdAt`
- `expiresAt`
- `deviceId`
- `deviceName`
- `jumpParams`

#### 11.2.2 通知主记录模型 `NoticeRecord`

通知记录不是 `IssueRecord` 本体，而是用于驱动通知中心浮窗的映射快照。

建议字段：

- `id`
- `kind`
  - `ACTION_REQUIRED`
  - `REMINDER`
- `sourceType`
  - `ISSUE`
  - `SYSTEM_EVENT`
- `sourceId`
- `issueId`
- `title`
- `summary`
- `severity`
- `libraryId`
- `status`
  - `UNREAD`
  - `READ`
  - `JUMPED`
  - `STALE`
- `createdAt`
- `updatedAt`
- `readAt`
- `jumpedAt`
- `sortKey`
- `source`
- `capabilities`
- `jumpParams`

#### 11.2.3 通知来源模型 `NoticeSource`

建议显式建模来源对象。

建议字段：

- `sourceDomain`
- `issueCategory`
- `issueNature`
- `taskId`
- `taskItemId`
- `fileNodeId`
- `endpointId`
- `path`
- `sourceLabel`
- `routeLabel`

#### 11.2.4 通知能力模型 `NoticeCapabilities`

建议字段：

- `canMarkRead`
- `canOpenIssueCenter`
- `canOpenTaskCenter`
- `canOpenFileCenter`
- `canOpenStorageNodes`
- `canOpenImportCenter`

### 11.3 通知与 `IssueRecord` / 系统事件映射

#### 11.3.1 从 `IssueRecord` 映射到通知

建议采用白名单映射，而不是“所有活跃异常自动生成通知”。

映射规则：

- 输入：`IssueRecord`
- 中间层：通知准入判断
- 输出：`NoticeRecord`

映射要点：

- 同一 `issueId` 只生成一条活跃通知
- 通知标题与摘要基于 `IssueRecord.title / summary / source`
- 通知主动作基于来源域与能力字段推导
- 提醒类或处置类由异常类型、性质和动态升级规则共同决定

#### 11.3.2 从系统事件映射到页头信号

设备插入信号不生成 `NoticeRecord`，而是生成独立 `HeaderSignal`。

原因：

- 它不是异常
- 它不应占用通知流
- 它应使用单独的绿色视觉语义

后续若核心迁移到 NAS：

- Windows 本地设备事件由 `Local Edge Agent` 感知。
- `Local Edge Agent` 只上报设备事实，不直接生成通知或页头展示对象。
- `Central Core` 负责把设备事实归并为 `HeaderSignal`，并继续保持其与 `NoticeRecord` 的边界分离。

### 11.4 查询与命令分离

建议前端与后端边界分为两类。

#### 查询接口

用于读取：

- 页头绿色设备信号
- 通知角标摘要
- 通知列表摘要
- 单通知跳转目标信息

#### 命令接口

用于执行：

- 打开通知中心
  - 自动批量消费提醒类通知
- 标记单条通知已读
- 打开通知目标页
  - 若为处置类，同时写入“已跳转”

说明：

- 通知中心不直接承接异常治理命令
- 后续服务化时只替换查询与命令实现，不重构主要 UI 结构

### 11.5 右侧浮窗视图模型

#### 11.5.1 列表模型 `NoticeListVM`

建议字段：

- `id`
- `kindLabel`
- `title`
- `summary`
- `severityTone`
- `sourceDomainLabel`
- `libraryLabel`
- `objectLabel`
- `statusLabel`
- `updatedAtLabel`
- `primaryActionLabel`
- `capabilities`

#### 11.5.2 页头摘要模型 `NoticeBadgeVM`

建议字段：

- `totalUnreadCount`
- `actionRequiredCount`
- `reminderCount`
- `displayCountLabel`

#### 11.5.3 设备提醒模型 `DeviceInsertedSignalVM`

建议字段：

- `id`
- `label`
- `deviceName`
- `createdAtLabel`
- `jumpTarget`

### 11.6 角标聚合、去重与升级降级策略

#### 11.6.1 聚合策略

数字角标只统计未消费通知，不统计异常总量。

#### 11.6.2 去重策略

建议按以下主键去重：

- 异常通知：`issueId`
- 设备接入提醒：`deviceId + sessionId`

#### 11.6.3 升级策略

当以下类型影响升级时，允许重新计算通知类型：

- `临时文件待处理`
- `历史版本残留`

升级时：

- 更新同一条通知记录
- 不新增重复通知
- 如需提升注意力，可刷新排序和角标参与状态

#### 11.6.4 失效策略

以下情况应将通知标记为 `STALE` 或从主列表移出：

- 对应异常已解决且不再需要用户关注
- 对应异常已归档
- 设备已移除
- 通知源对象已不存在且无回查价值

### 11.7 数据装载与刷新策略

为保证后续生产环境性能，建议采用分层加载。

#### 首屏页头

只加载：

- 设备接入信号
- 通知角标摘要

#### 打开通知中心浮窗

再加载：

- 当前有效通知列表
- 用于跳转的最小上下文

#### 跳转深查

用户跳转目标页时，再按目标页既有策略加载：

- 异常详情
- 任务上下文
- 文件上下文
- 节点上下文

#### 刷新频率

- 处置类通知：中高频刷新
- 提醒类通知：中低频刷新
- 已读或已跳转通知：低频刷新
- 已失效通知：不自动刷新

### 11.8 当前已实现版本与设计基线差异

以下为当前客户端收敛版本，优先级高于早期“演进建议”口径：

- 通知中心已完成从基础消息流到 `NoticeRecord + HeaderSignal` 模型的升级，不再处于“后续演进”阶段。
- 页头“红点”已升级为数字角标，统计口径为“未消费通知数”。
- 设备插入提醒已从通知流拆出，采用导入中心左侧绿色标签承载。
- 通知中心 UI 已收敛为紧凑列表风格，不再展示顶部统计大卡片和说明文案。
- 通知语义标签在视觉层已隐藏，当前通过无障碍文本保留语义；如后续需要显式标签，应作为新迭代需求评审，不视为本版缺陷。

仍保留为规划基线（尚未在当前版本落地）的项：

- 更细粒度的通知准入自动化策略（例如跨周期动态阈值调度、策略中心可视化配置）。
- 与后端实时事件流的生产接入（当前为客户端 mock 与本地状态驱动）。

## 13. 风险与约束

### 13.1 关键风险

- 若通知准入范围失控，通知中心会迅速退化为异常中心副本
- 若提醒类与处置类语义混淆，角标会丧失可信度
- 若设备接入提醒仍进入通知流，会与真正异常通知抢注意力
- 若不做去重，同一异常会在轮询中反复刷通知

### 13.2 必须定稿的约束

以下内容在后续实现前应视为定稿：

- 通知中心不是异常中心副本，只展示真正用户应该关心的通知
- 设备接入提醒独立于通知中心浮窗，以绿色小标签显示在导入中心左侧
- 通知中心风格、UI、UX、布局必须完全参考传输任务子页面
- 通知中心只做“看”和“跳”，不承接异常治理闭环
- 仅白名单中的重要异常允许映射为通知
- 提醒类通知打开浮窗即已读
- 处置类通知仅在手动已读或点击跳转后消角标
- 普通任务状态变化不进入通知中心
- `临时文件待处理`、`历史版本残留` 采用动态升级策略
- 跳转继续复用现有跨页定位键，不另造一套参数

## 14. 结论

本方案将客户端“通知中心”明确为一个位于页头右侧、以右侧浮窗承载的轻量关注入口。

它在业务上通过“设备接入提醒 / 通知中心 / 异常中心”三层信号分层，避免把所有异常都堆进页头；在交互上严格收缩为“看与跳”，不侵占异常中心的治理职责；在页面上完全继承传输任务子页面的结构、风格与交互语言；在技术上通过独立的通知快照模型、白名单映射规则、去重与角标聚合策略，为后续 AI 实现和真实后端接入提供稳定边界。
