# 统一文件管理系统 CD2 远程上传只进缓存不进 115 的缺陷修复记录

## 问题概述

在切片 11 的 `CD2 + 115 + aria2` 真实联调过程中，`CD2 remote upload` 一度出现如下现象：

- 任务中心里上传任务看起来已经结束，或者长时间停留在“文件预处理”。
- CD2 本地待上传缓存目录里已经出现对应文件。
- 115 网盘里却没有真正上传完成的文件。
- 早期还伴随出现过：
  - `GeneralFailure Upload session not found`
  - `GeneralFailure remote upload client unresponsive during read`
  - `UPLOAD_CHANNEL_IDLE_TIMEOUT`

这个缺陷已经确认不是“接口方向错误”，而是多个实现问题叠加导致，其中**最终导致“只进缓存不进云端”的直接根因**是：**CD2 全局系统设置里 `sync_with_cloud=false`**。

## 最终确认的根因

### 根因 1：CD2 全局系统设置关闭了云同步

真实容器中的 `/Config/systemsettings.json` 一度为：

```json
{
  "sync_with_cloud": false,
  "upload_delay_secs": 10
}
```

这会导致 `remote upload` 只把文件写入 CD2 的本地待上传缓存/任务体系，而不会继续同步到 115 云端。

这正是“CD2 里看起来有任务、缓存里也有文件，但 115 里没有文件”的直接原因。

### 根因 2：业务代码把“喂给 CD2 完成”误判成“上传到 115 完成”

早期实现里，上传等待逻辑存在以下错误：

- 只要 `GetUploadFileList.transferedBytes >= size`，就直接返回成功。
- 这只能说明“本系统 -> CD2 remote upload 通道”这段数据喂送完成。
- 不能说明“CD2 -> 115”这段后台上传已经完成。

因此会出现：

- 本地副本尚未真正同步到 115。
- 但系统已经提前写入云副本记录，任务中心也显示成功。

### 根因 3：早期 remote upload 协议实现不完整

这不是本次最终“只进缓存不进 115”的直接根因，但它在联调早期真实存在，必须一并记住，避免后续再次退化：

- `StartRemoteUpload` 与 `RemoteUploadChannel` 没有严格复用同一会话上下文时，会报：
  - `Upload session not found`
- 上传通道未真正建立就发起 `StartRemoteUpload` 时，会出现：
  - `remote upload client unresponsive during read`
- `device_id` 若不是稳定持久值，会破坏重连恢复。
- `hash_data` 若不实现，会在部分文件或服务端校验路径下失败。

## 本次修复后的正式结论

### 结论 1：CD2 上传前必须确保 `sync_with_cloud=true`

现在正式代码会在每次调用 `StartRemoteUpload` 之前：

1. 读取 CD2 `SystemSettings`
2. 检查 `syncWithCloud`
3. 若为 `false`，自动调用 `SetSystemSettings` 改为 `true`

也就是说：

- 不能假设用户的 CD2 实例已经被正确配置。
- 中心服务必须主动保证 CD2 处于“允许同步到云端”的状态。

### 结论 2：上传成功只能认 CD2 真实终态 `Finish`

现在上传完成判定只能依赖：

- `RemoteUploadStatusChanged(status=Finish)`，或
- `GetUploadFileList` 中对应任务进入 `Finish`

禁止再使用以下条件作为成功判定：

- `transferedBytes == size`
- CD2 缓存目录里出现临时文件
- `FindFileByPath` 过早查到缓存态路径

### 结论 3：任务中心要展示 CD2 真实状态，而不是自造成功语义

当前正式状态映射应以 CD2 外部任务状态为准：

- `WaitforPreprocessing` / `Preprocessing` -> `文件预处理中`
- `Inqueue` -> `上传排队中`
- `Transfer` -> `上传中`
- `Pause` -> `已暂停`
- `Finish` -> `已完成`
- `Skipped` -> `已跳过`
- `Cancelled` -> `已取消`
- `Error` / `FatalError` -> `上传失败`

## 已落地的代码修复点

### 1. 自动拉起云同步

文件：

- `services/center/internal/integration/cd2_client.go`
- `services/center/internal/integration/provider_cd2_115.go`

关键实现：

- 新增 `getSystemSettings()`
- 新增 `setSystemSettings()`
- 在 `CD2115Driver.StartUpload()` 中调用 `ensureCloudSyncEnabled()`

职责：

- 如果检测到 CD2 当前 `syncWithCloud=false`，自动修正为 `true`

### 2. 上传完成判定改为只认真实终态

文件：

- `services/center/internal/integration/provider_cd2_115.go`

关键实现：

- `WaitUpload()` 不再使用 `bytesDone >= bytesTotal` 判定成功
- 只在外部任务状态进入 `Finish` 时返回成功
- `Cancelled` 返回取消
- `Skipped / Error / FatalError` 返回明确失败

### 3. 上传任务状态解析改为按 CD2 语义

文件：

- `services/center/internal/integration/provider_cd2_115.go`
- `client/src/lib/jobsApi.ts`
- `client/src/pages/TaskCenterWorkspace.tsx`

关键实现：

- 后端 `TransferProgress` 增加：
  - `ExternalStatus`
  - `ErrorMessage`
  - `Terminal`
  - `Successful`
- 前端消费 `externalTaskEngine / externalTaskStatus`
- 任务中心显示真实阶段，而不是继续把缓存态误解成成功

### 4. 为 remote upload 协议回归补单测

文件：

- `services/center/internal/integration/provider_cd2_115_test.go`

测试覆盖：

- 上传状态字符串到枚举的解析
- 哪些状态是终态
- 状态文案映射
- 默认错误文案映射

## 后续实现时的强制约束

后续任何 vibe coding、重构、抽象、性能优化，都必须遵守以下规则：

### 规则 1：不得假设 CD2 实例已经正确配置

必须由中心服务在上传前主动校验并确保：

- `sync_with_cloud=true`

如果后续接入更多 CD2 配置能力，也不能把这项依赖下放给人工记忆或运维手动操作。

### 规则 2：不得再用字节数跑满当作上传成功

以下判定全部无效：

- `bytesDone == bytesTotal`
- `CD2 缓存目录已有文件`
- “任务列表暂时能查到文件”

唯一有效的成功条件是：

- CD2 上传任务真实进入 `Finish`

### 规则 3：不得把缓存态副本写入正式资产副本

只有在上传任务真实完成后，才允许写入：

- `asset_replicas` 云副本记录
- 文件中心端点状态
- 任务中心完成态

### 规则 4：不得拆散 remote upload 会话约束

以下约束必须始终保持：

- `RemoteUploadChannel` 与 `StartRemoteUpload` 复用同一 CD2 用户会话
- `device_id` 必须稳定持久化
- `hash_data` 必须保持可用
- 上传通道必须是共享长连接，不得回退成“每次临时建一个 client 随便发”

### 规则 5：生产回归必须覆盖真实上传验证

后续涉及以下改动时，必须至少做一次真实上传烟测：

- CD2 适配层
- 上传作业状态机
- 外部任务状态映射
- 任务中心阶段展示
- 云副本写回逻辑

最少要验证：

1. 当 `sync_with_cloud=false` 时，上传前会被自动修正
2. 上传后文件能在 115 真实路径查到
3. 任务中心最终显示为 `已完成`
4. 不会只在缓存存在、却提前写入云副本

## 后续若再次出现类似现象，优先排查顺序

按以下顺序排查，不要再从头盲猜：

1. 先查 CD2 `SystemSettings.syncWithCloud` 是否为 `true`
2. 再查 `GetUploadFileList` 里对应任务是否真实进入 `Finish`
3. 再查是否只是 `transferedBytes` 跑满、但状态仍停在 `Preprocessing / Transfer`
4. 再查 `RemoteUploadChannel` 是否和 `StartRemoteUpload` 处于同一会话
5. 再查 `device_id` 是否稳定复用
6. 最后才去看 115 厂商侧网络、令牌、权限问题

## 一句话防回归结论

**CD2 remote upload 的“写入待上传缓存”不等于“文件已经上传到 115”；系统必须先确保 `sync_with_cloud=true`，并且只能以 CD2 上传任务真实进入 `Finish` 作为成功标准。**
