# 统一文件管理系统-CD2 远程上传只进缓存不进 115 的缺陷修复记录

## 文档说明

- 首次修复结论沉淀时间：2026-04-12
- 本次刷新时间：2026-04-13
- 目的：保留历史缺陷根因，同时补充当前架构下已经确认存在的缺陷与结构性风险，避免后续再次把缓存态、局部成功态或临时可恢复态误当成最终成功态。

## 1. 历史现象

在真实联调 `CD2 + 115 + aria2` 过程中，曾经出现过以下问题：

- 任务中心里上传任务看起来已经结束，或者长时间停留在“文件预处理”
- CD2 本地待上传缓存目录里已经出现对应文件
- 115 网盘里却没有真正上传完成的文件
- 早期还伴随出现：
  - `GeneralFailure Upload session not found`
  - `GeneralFailure remote upload client unresponsive during read`
  - `UPLOAD_CHANNEL_IDLE_TIMEOUT`

## 2. 已确认的历史根因

### 2.1 `sync_with_cloud=false`

历史上导致“只进缓存不进 115”的直接根因，是 CD2 全局系统设置里：

```json
{
  "sync_with_cloud": false
}
```

在这种情况下：

- 本系统到 CD2 的远程上传通道是通的
- CD2 本地缓存也会接收到文件
- 但 CD2 不会继续把缓存任务同步到 115

因此不能把“CD2 缓存里已有文件”视为“115 已完成上传”。

### 2.2 错把缓存完成当作上传完成

早期实现还存在另一类误判：

- 只要本系统把字节喂给 CD2 完成
- 或者缓存目录里已经有对应文件
- 就提前把任务写成成功
- 甚至提前写入云副本记录

这会造成：

- 任务中心显示成功
- 文件中心显示已有云副本
- 但 115 实际还没有真正完成上传

## 3. 已完成的正式修复

### 3.1 上传前自动确保 `sync_with_cloud=true`

当前正式实现会在上传前读取 CD2 `SystemSettings`：

- 若 `syncWithCloud=true`，直接继续
- 若 `syncWithCloud=false`，自动修正为 `true`

这条规则不能再依赖人工配置或运维记忆。

### 3.2 成功标准改成只认真实终态

当前上传成功标准已经收敛为：

- 只认 `GetUploadFileList` 中对应任务进入 `Finish`

以下信号都不能再作为成功依据：

- CD2 缓存目录里出现临时文件
- 本地读流完成
- 本地哈希完成
- 传输字节数暂时跑满

### 3.3 任务中心展示真实外部阶段

当前任务中心已经按 CD2 外部任务状态展示阶段语义，而不是继续把缓存态解释成成功态。典型映射口径如下：

- `WaitforPreprocessing / Preprocessing`：文件预处理中
- `Inqueue`：缓存写入完成，等待云端上传
- `Transfer`：上传中
- `Finish`：已完成
- `Error / FatalError`：上传失败

## 4. 本次刷新后新增确认的问题

### 4.1 CD2 上传通道断线会过早杀死本地会话

当前上传通道基于单条长连接 gRPC 流承载。现状下，只要 `RemoteUploadChannel` 的 `Recv()` 返回错误，本地协调层就会：

- 重置上传通道
- 终止当前挂在本地内存中的上传会话

但这并不等价于：

- 服务端上传任务已经失败
- 云端状态已经不可恢复

这说明当前实现对短时网络抖动的容忍度仍然不够，后续必须补“断线优先重连”而不是“本地立即全量失败”。

### 4.2 高频进度回写会把系统推向高频写库模式

当前上传和下载的进度刷新都会持续触发作业层更新。每次落库时，不只是写字节数，还会伴随：

- `job_items` 更新
- `jobs` 更新
- 聚合作业进度
- 写入 `job_events`

这已经是当前代码路径中的真实结构性风险。是否已经在生产中稳定复现“数据库锁死”，不能仅靠静态代码直接下结论，但高并发下的连接池压力、写放大和界面进度抖动风险是客观存在的。

### 4.3 Aria2 恢复逻辑对错误态任务处理不完整

当前 Aria2 恢复流程虽然会重新找到旧任务 ID 或目标路径对应任务，但如果找到的是旧的 `error` 任务：

- 不会主动刷新下载 URL
- 不会显式把错误态任务重新激活
- 后续等待逻辑会很快再次失败

这在云端直链过期后尤其危险，因为旧 URL 失效时，“找到旧 GID”并不等于真正恢复了下载。

### 4.4 哈希计算与上传读流存在读盘竞争风险

当前上传过程中，同一个文件可能同时出现：

- 上传读流读取
- MD5 全文件读取
- SHA1 全文件读取
- 其他特征哈希读取

如果底层介质是机械硬盘或普通 SSD，这种多路并发大文件读取会显著放大 I/O 竞争，进而提高：

- 读速下降概率
- 上传通道无响应概率
- 被服务端误判为超时或异常的概率

这属于当前代码结构已经具备的真实风险，只是触发程度会受到文件大小、介质类型和并发量影响。

## 5. 当前架构的准确认知

需要特别说明一件事：当前项目虽然同时接入了 `CloudDrive2` 和 `aria2`，但还没有实现正式的“离线下载 + 自动转存上传”业务流水线。

当前真实链路是：

1. `LOCAL/NAS -> CLOUD`
   - 直接走 `CD2 remote upload`
2. `CLOUD -> LOCAL/NAS`
   - 先由 CD2 解析下载地址
   - 再交给 `aria2` 下载
3. `LOCAL/NAS -> LOCAL/NAS`
   - 由中心服务直接复制

当前项目里不存在：

- Aria2 下载完成后再把文件物理路径交给 CD2 继续上传
- 通过 WebDAV 做正式接力复制
- CD2 官方离线下载任务的业务编排中心

因此，后续文档和实现都不能把当前系统描述成“完整离线下载 + 云盘转存自动化流水线”。

## 6. 刷新后的强制规则

后续任何改动，只要涉及以下模块：

- CD2 适配层
- Aria2 适配层
- 作业状态机
- 文件中心同步入口
- 任务中心阶段展示

都必须继续满足这些规则。

### 规则 1：不得把缓存态当成功态

以下都不是成功：

- CD2 缓存目录已有文件
- 本地读流完成
- 哈希完成
- 字节数跑满但任务未进入 `Finish`

只有 `Finish` 才是上传成功。

### 规则 2：不得假设 CD2 实例已经被正确配置

上传前必须主动校验并确保：

- `sync_with_cloud=true`

### 规则 3：任务中心必须展示真实外部状态语义

前端展示必须区分：

- 作业状态
- 外部引擎状态

不能只显示一个模糊的“运行中”。

### 规则 4：不得把当前系统写成完整离线流水线

当前项目还没有正式实现：

- CD2 离线下载任务提交
- 离线任务状态轮询
- 离线下载完成后的自动转存上传

相关文档、PR 描述和实现说明都必须如实表述。

### 规则 5：对“顺风局可跑通”的实现不能默认等价于“逆风局可恢复”

只要涉及：

- 网络抖动
- 进程重启
- 大文件上传
- 多任务并发

都必须显式讨论恢复边界，而不是默认按顺风局行为描述系统。

## 7. 后续建议

若继续推进“离线下载 + 云盘转存”方向，建议按以下顺序演进：

1. 先把外部任务状态做成结构化持久化
2. 再做 CD2 上传通道的断线重连
3. 再做连接复用与集中状态缓存
4. 明确 `Pause`、`Cancelled`、`NotFound` 的恢复语义
5. 补齐 Aria2 错误态任务的 URL 刷新与恢复逻辑
6. 收敛哈希计算与上传读流的 I/O 竞争
7. 最后独立新增离线下载作业意图，不继续复用当前 `REPLICATE`

建议阶段拆分为：

1. 提交离线下载任务
2. 等待离线任务真实终态
3. 校验离线产物
4. 启动 CD2 上传
5. 等待上传真实终态
6. 写入副本记录

## 8. 结论

这次历史缺陷的核心教训仍然成立：

**CD2 remote upload 的“写入待上传缓存”不等于“文件已经上传到 115”；系统必须先确保 `sync_with_cloud=true`，并且只能以 CD2 上传任务真实进入 `Finish` 作为成功标准。**

在此基础上，本次刷新额外明确：

- 当前项目已具备正式的云上传与云下载能力，但还不是完整的“离线下载 + 云盘转存”自动化流水线
- 当前实现已经暴露出一批逆风局下的恢复与容错边界，后续不能再按“网络稳定、文件小、任务少”的顺风局假设继续扩展实现
