# 统一文件管理系统-垂直切片实现路径

## 文档说明

- 更新时间：2026-04-11
- 适用范围：当前仓库中的桌面客户端、后续新增的中心服务、本地执行器，以及基于现有设计文档推进的全链路实现工作
- 文档目标：给出一套按整体功能垂直切片推进的实现路径，要求每个切片都可独立运行、独立验证，并尽量只依赖前置切片
- 约束前提：
  - 以真实实现为目标推进开发和测试，最终不保留 mock、fallback 或降级路径
  - 客户端现有 UI / UX 作为既有交互基线，后续实现尽量承接当前页面结构与操作语义
  - 前置切片优先提供稳定依赖，后置切片可以依赖前置切片，前置切片原则上不反向依赖后置切片

## 1. 当前阶段判断

当前项目不是从零开始构建，而是从“高保真客户端原型”走向“真实可运行系统”。

现状可以归纳为三点：

1. 客户端页面、交互、信息架构和主要业务闭环已经被验证，具备较强的产品参考价值。
2. 当前权威状态仍然分散在多套本地数据源中，但存储节点切片已经先行收口到中心服务与 PostgreSQL。
3. 后续正式形态已经有明确方向，即三段式架构：客户端、中心服务、本地执行器。

因此，实现路径不应按“页面逐个补接口”的方式推进，而应按“一个完整业务闭环 = 一个垂直切片”的方式推进。每个切片都必须覆盖从数据库到服务、从接口到客户端消费、从状态变更到验收路径的完整链路。

## 2. 实现总原则

### 2.1 切片设计原则

每个垂直切片必须同时满足以下条件：

- 有明确的主业务目标，而不是松散的技术项集合
- 有独立的数据边界、接口边界和状态边界
- 有可独立执行的最小用户路径
- 有独立验收标准，可以在不依赖后续切片的前提下证明该切片成立
- 完成后可以作为后续切片的稳定依赖

### 2.2 依赖控制原则

- 优先建设底层稳定能力，再建设依赖这些能力的业务切片
- 优先做“查询和权威建模”再做“复杂执行和治理”
- 优先做独立性强的业务域，再做跨域联动强的业务域
- 任何切片一旦完成，就不再依赖未来切片补齐其基本闭环

### 2.3 实现方式原则

- 所有新切片均以真实数据库、真实服务、真实接口、真实执行路径为前提
- 不引入开发态专用 mock 逻辑，也不保留生产运行中的 fallback 路径
- 测试采用真实实现目标下的测试驱动方式推进，重点覆盖状态机、边界条件、错误路径和关键用户流
- 客户端逐步从本地状态编排迁移为“查询中心服务 + 发送命令 + 订阅事件”的工作台

## 3. 目标架构基线

后续实现以三段式架构为统一基线：

1. 客户端：继续沿用现有 Tauri + React 桌面端，负责界面承载、操作入口和状态展示。
2. 中心服务：使用 Go 实现，作为唯一权威业务中枢与唯一权威数据源。
3. 本地执行器：使用 Go 实现，负责本机资源探测、本地扫描、本地文件执行以及结果回写。

关键落地约束如下：

- PostgreSQL 为唯一主数据库
- 统一资产模型为唯一资产权威来源
- 作业、异常、通知、导入等状态全部由中心服务管理
- 客户端不再承载权威业务状态
- 通知消费状态仍可保留在客户端本地，但通知源数据由中心服务提供

## 4. 垂直切片划分

以下切片顺序按“尽量只依赖之前的切片，不依赖之后的切片”的原则排列。

### 切片 0：系统接线底座

**目标**

建立整个系统从客户端到中心服务、再到 PostgreSQL 的最小真实链路，为后续所有业务切片提供统一运行底座。

**范围**

- 中心服务工程骨架
- 本地执行器工程骨架
- 迁移框架与基础运行配置
- 数据库连接与基础健康检查
- 统一错误模型、响应约定、基础日志与配置读取
- 最小共享 contracts / DTO / 运行态契约
- 开发态 PostgreSQL 运行编排
- 客户端对中心服务与本地执行器的基础接线能力

**本切片完成后应达到的状态**

- 中心服务可启动、可连接数据库、可执行迁移
- 本地执行器可启动、可注册、可持续发送心跳
- 客户端可识别中心服务与本地执行器的运行状态
- 项目后续所有真实切片都可基于该底座继续推进

**依赖**

- 无

**当前已落地实现（截至 2026-04-08）**

- 已新增 `services/center`，并落地：
  - `GET /healthz`
  - `GET /readyz`
  - `GET /api/runtime/status`
  - `POST /agent/register`
  - `POST /agent/heartbeat`
- 已新增 `services/agent`，并落地：
  - 本地身份持久化
  - 注册与心跳
  - 基础重试
- 已新增 `shared/contracts`
- 已新增开发态 PostgreSQL Docker 编排
- 已新增根目录一键重启脚本 `one-link-start.ps1`
- 已新增统一启动命令：
  - `cnpm run one-link-start`
  - `cnpm run one-link-start:with-db`
- 已在客户端顶层壳层接入系统状态灯：
  - 中心服务异常：红灯

**范围**

- 挂载管理
- NAS 管理
- 网盘管理
- 连接测试
- 扫描历史读取
- 存储节点与资产库的绑定关系

**本切片完成后应达到的状态**

- 客户端存储节点页完全切到真实后端
- 新增、编辑、删除、测试、查看历史均走真实服务
- 数据重启后可持久保留

**现实口径**

- 其中“中心服务 + PostgreSQL”部分已经落地
- “客户端完全去 mock / 去 fallback”还未完成
- “本地执行器完整接入”还未完成

3. 切片 2：资产库与文件中心只读查询

- 最后再收口长期运行与后台治理问题，避免过早陷入复杂调度细节

## 7. 每个切片的验收要求

虽然本文不展开到任务级拆分，但每个切片都应满足统一验收要求：

- 有真实数据库迁移和真实数据模型
- 有真实接口或真实事件流
- 有客户端上的最小可见入口
- 有一条完整、可重复执行的人工验收路径
- 有与该切片相匹配的自动化测试
- 不依赖 mock、种子状态硬编码或后续切片兜底

只有满足这些条件，一个切片才应被视为“完成”，并作为后续切片的前置依赖。

## 8. 结论

本项目后续实现不应采用“大而全的一次性服务化重写”，而应采用“真实实现目标下的垂直切片推进”。

整套切片的核心顺序是：

- 先统一底座
- 再统一存储与资产
- 再统一作业与任务中心
- 再把文件中心写操作、异常、通知、导入与云端接入逐步接上
- 最后补齐后台治理和长期运行能力

在这一路径下，前置切片将持续沉淀为后置切片的稳定依赖，整个系统也能在开发过程中逐步从“可演示的客户端原型”过渡为“可真实运行的生产系统”。

## 9. 切片8（异常中心）当前交付

- **真实异常域一体化**：后端通过 `services/center/migrations/0006_jobs.sql` 与 `0007_issues.sql` 在 PostgreSQL 建立 `jobs`/`job_events`、`issues`/`issue_events`/`issue_object_links` 模型，`services/center/internal/jobs` 负责作业调度、事件、SSE，`services/center/internal/issues` 负责同步、能力与历史，`services/center/internal/http` 将 `/api/issues`、`/api/issues/actions`、`/api/issues/history/clear` 与 `/api/jobs` / `/api/events/stream` 等接口开放给客户端。
- **客户端直连中心 API**：`client/src/lib/issuesApi.ts` 与 `client/src/lib/jobsApi.ts` 基于 `centerBaseUrl` 拉取中心数据，`client/src/App.tsx` 通过 `issuesApi.listAll()`、`jobsApi.subscribe()` 更新 `issueRecords` 与任务数据，`IssuesPage` 与 `TaskCenterWorkspace` 的展示、操作、跳转均基于这些接口，不再依赖 mock 的 `PersistedState.issueRecords` 或 `taskRecords`。
- **异常去重与次数累计**：异常域现已按稳定组合键复用历史治理对象，不要求与最近一次异常相邻；同一问题再次出现时会回并到原异常，并通过 `occurrenceCount` 累计发生次数，客户端异常中心已显示次数徽标。
- **异常中心联动任务/文件/存储节点**：`client/src/App.tsx` 提供 `openTaskCenterForIssue`、`openFileCenterForIssue`、`openStorageNodesForIssue`，`StorageNodesPage` 的 `onOpenIssueCenter` 与 `TaskCenterWorkspace` 的 `preselectedTaskIds` 确保存储事件、异常、任务三端上下文在跳转时一致，后端通过 `issue.source` 填入 `taskId`/`fileNodeId`/`endpointId`。
- **任务中心重连异常统计**：`TaskCenterWorkspace` 使用 `jobsApi.list` 加载分页作业、`jobsApi.detail` 拉取详单、`jobsApi.subscribe` 监听作业事件，所有暂停/恢复/重试/取消操作走 `/api/jobs/{id}/{action}` 与 `/api/job-items/{id}/*`，并通过 `issuesApi.listByJobIds` 将异常摘要与事件关联反馈。
- **界面能力调整**：异常记录的菜单只保留“重试”“确认”“忽略”“归档”四种命令，已删去“延后处理”“刷新检测”按钮，所有导航/筛选/分页等描述应围绕当前 `issuesApi`+`jobsApi` 能力重新梳理。
