## 1. 文档总览

本任务计划用于实现“垂直切片 12：后台治理与巡检”，目标是让系统从“可手动使用的真实系统”演进为“可后台持续运行的真实系统”。本计划严格以 [统一文件管理系统-垂直切片实现路径.md](/B:/new_project/mare/docs/实现路径/统一文件管理系统-垂直切片实现路径.md) 中切片 12 定义为准，并结合当前仓库真实代码现状拆分可分阶段交付的实现任务。

切片12范围（来自实现路径文档）：

1. 定时扫描
2. 心跳与健康检查
3. 自动重试策略
4. 深度校验
5. 文件消失检测
6. 后台清理
7. 长时间运行下的状态治理与审计记录

切片12完成后应达到的状态：

1. 客户端退出后，中心服务仍可持续运行关键任务
2. 系统具备基础的长期运行能力和后台治理能力
3. 单机单用户场景下的生产可用性明显增强

## 2. 需求与范围

功能性要求（必须满足）：

1. 中心服务可常驻运行，客户端退出不影响后台调度与关键任务执行。
2. 支持“定时扫描”与“心跳/健康检查”的真实后台调度闭环，而不是仅保存策略字段用于展示。
3. 支持“自动重试策略”：对可恢复的失败任务按策略自动退避并重试，达到预算后沉淀为异常并通知。
4. 支持“深度校验”：以作业形式执行，结果回写副本校验字段，并在失败时沉淀异常。
5. 支持“文件消失检测”：基于已存在的 `MISSING` 事实层，生成可治理的异常与通知，而非静默写字段。
6. 支持“后台清理”：删除与清理从“同步立即执行”演进为“可追踪、可审计、可后台执行”的治理链路。
7. 支持“审计记录”：后台治理行为具备可追溯的记录（最少满足问题定位与长期运行观测）。

非目标（本切片不做或不强求）：

1. 不做多机多用户协同调度。
2. 不做跨客户端的通知已读同步。
3. 不追求覆盖所有云端能力的深度校验（优先落地 LOCAL/NAS 可达路径，CLOUD 先给出可解释的降级与异常）。

约束与优先级：

1. 不引入 mock/fallback 作为最终形态，新增链路必须可在真实数据库与真实服务上运行。
2. 复用现有作业内核与 SSE 事件流，不重写核心执行循环。
3. 任何后台调度必须具备幂等性与可恢复性，避免重启后重复创建大量作业。

## 4. 架构变更建议

切片12建议的架构变更点（尽量小改动、可增量落地）：

1. 新增中心服务后台治理调度器模块（建议包名 `services/center/internal/governance`）。
2. 为“定时扫描/心跳检查/缺失检测/清理/校验”引入可持久化的调度状态与幂等锁，避免重复触发。
3. 扩展作业域以支持“自动重试退避”：为 `job_items` 增加 `next_retry_at`（或等价字段），并在 claim/执行逻辑中尊重该字段。
4. 扩展存储域以支持“定时扫描退避”：为 `mount_runtime` 增加 `next_scan_at`（或等价字段），并在调度器中维护。
5. 扩展异常域以支持“非作业失败”来源的治理异常（例如文件缺失、执行器离线等），建议保持与现有 issues/notifications 投影一致：新增 issues 服务的“治理来源同步入口”，由调度器触发。
6. 补齐 `CONNECTION_TEST`、`VERIFY_REPLICA`、`VERIFY_ASSET` 的作业执行器注册，使其能在后台被调度与审计。
7. 新增最小审计表（建议 `governance_audit_logs`），记录调度器产生的治理行为与原因，用于长期运行定位。

## 5. 分阶段实施计划

### 阶段 1: 后台调度器骨架与定时扫描闭环 (最小可用后台运行)

目标：让“定时扫描”从字段展示变成真实后台行为，且不依赖客户端常驻。

1. **新增迁移：mount_runtime 增加 next_scan_at** (File: `services/center/migrations/0015_governance_mount_scan.sql`)
   动作： 为 `mount_runtime` 增加 `next_scan_at TIMESTAMPTZ`，并补一个可用索引（例如 `(next_scan_at)`），默认值允许为空。
   原因： 当前只有 `last_scan_at` 与 `scan_policy`，缺少可持久化的“下一次扫描时间”，无法形成后台调度闭环。
   依赖： 现有 `0002_storage_local_folders.sql` 已创建 `mount_runtime`。
   风险：低，字段新增不影响现有读写；注意回填策略与空值兼容。

2. **新增治理调度器 Service** (File: `services/center/internal/governance/service.go`)
   动作： 新增 `governance.Service`，提供 `Start(ctx)`，内部 ticker 循环执行 `tick()`，tick 做两类事：扫描调度、心跳调度（后续阶段 增加更多治理任务）。
   原因： 切片12的核心是让中心服务具备后台持续运行能力，这需要一个常驻调度器，而不是依赖客户端触发。
   依赖： 阶段 1 步骤 1（需要 `next_scan_at` 支持）。
   风险：中，需保证 tick 幂等、可重入、重启可恢复。

3. **实现“扫描到期判定 + 领取锁”** (File: `services/center/internal/governance/mount_scan_scheduler.go`)
   动作： 在 DB 中以事务方式“领取到期挂载”，典型做法：
   - `SELECT ... FOR UPDATE SKIP LOCKED` 找到 `next_scan_at <= now()` 的挂载或运行态记录。
   - 领取后立即把 `next_scan_at` 更新为下一次（或置空等待手动），并记录触发原因。
   - 对 `scan_policy = 'ON_START'` 的挂载，在服务启动后做一次性调度，成功后将其转换为 `SCHEDULED` 或保留 ON_START 但 next_scan_at 置空。
   原因： 避免重复创建扫描作业；保证并发安全；保证中心服务重启不会造成扫描风暴。
   依赖： 阶段 1 步骤 2。
   风险：中，SQL 领取与更新要正确，避免“领取后创建作业失败导致丢调度”。

4. **为“定时扫描”创建 SCAN_DIRECTORY 作业并关联 mount** (File: `services/center/internal/governance/jobs_factory.go`)
   动作： 调度器为每个到期挂载 创建一个 `SCAN_DIRECTORY` 作业，`source_domain` 使用 `SCHEDULED` 或 `SYSTEM_POLICY`，并用 `job_object_links` 将 `mount_id` 作为目标关联。
   原因： 复用既有作业内核与 `SCAN_DIRECTORY` executor，避免新增一套扫描执行链路。
   依赖： 作业系统已存在；`SCAN_DIRECTORY` executor 已注册并支持 `SourceDomainStorageNodes` 路径（当前为挂载扫描）。
   风险：中，创建作业必须幂等（同一 mount 同一窗口内不重复创建）。

5. **补齐挂载 scan_policy 的可配置入口（后端最小）** (File: `services/center/internal/storage/service.go`)
   动作： 在保存挂载时允许写入 `scan_policy`（目前写死为 `'MANUAL'`），同时在响应 DTO 中暴露扫描策略，至少支持 `MANUAL/ON_START/SCHEDULED`。
   原因： 切片12要实现“定时扫描”，必须让策略能从 UI 或默认规则进入数据库。
   依赖： 阶段 1 步骤 3 和 步骤 4（调度器消费 scan_policy）。
   风险：低，属于字段贯通，但需保持 UI 兼容。

6. **在中心服务启动时启动治理调度器** (File: `services/center/internal/app/app.go`)
   动作： 在 `NewServer` 里构造 `governance.Service` 并加入 `multiStarter`，与 `jobService`、`integrationService` 一起 `Start(ctx)`。
   原因： 确保“客户端退出后中心仍运行”这一验收条件在架构层成立。
   依赖： 阶段 1 步骤 2。
   风险：低，注意 ctx 生命周期与关闭行为。

验收路径（阶段 1）：

1. 配置一个挂载，把 `scan_policy` 设置为 `SCHEDULED`，并设置一个较短的扫描频率（先用代码常量/环境变量，阶段 6 再做 UI 策略配置）。
2. 关闭客户端。
3. 观察中心服务日志与任务中心：应能看到后台创建的 `SCAN_DIRECTORY` 作业并执行完成。

### 阶段 2: 心跳与健康检查（挂载/网盘/执行器）闭环

目标：把“心跳策略 next_heartbeat_at”从展示升级为后台巡检作业，产生可治理信号。

1. **为 mount heartbeat 建立“到期领取 + 更新 next_heartbeat_at”流程** (File: `services/center/internal/governance/mount_heartbeat_scheduler.go`)
   动作： 类似 阶段 1 的扫描领取逻辑，从 `mount_runtime.next_heartbeat_at` 领取到期挂载，更新为 `computeNextHeartbeat()` 的下一次时间。
   原因： 当前 heartbeat 策略只在 `UpdateLocalFolderHeartbeat` 时计算写入，没有后台消费。
   依赖： `services/center/internal/storage/service.go` 里已有 `computeNextHeartbeat` 与 next_heartbeat_at 字段。
   风险：中，需避免 heartbeat 频繁触发导致大量连接测试请求。

2. **注册 CONNECTION_TEST 作业执行器** (File: `services/center/internal/app/app.go`)
   动作： 在 `jobService.RegisterExecutor(jobs.JobIntentConnectionTest, ...)` 中实现：
   - 若关联对象是 `MOUNT`，调用 `LocalFolderService.RunLocalFolderConnectionTest` 或对应云端/ NAS 的连接测试。
   - 若关联对象是 `STORAGE_NODE`，调用对应节点连接测试。
   - 若关联对象是 “集成网关（CD2/aria2）”，调用 `integration.Service` 的 runtime/test 能力（必要时新增一个统一的测试方法）。
   原因： 让调度器可以用作业系统统一承载健康检查，失败会自然进入异常/通知投影。
   依赖： 调度器创建 CONNECTION_TEST 作业（阶段 2 步骤 3）。
   风险：中，需要清晰定义“失败错误码/错误信息”，才能与自动重试策略协作。

3. **调度器按到期创建 CONNECTION_TEST 作业** (File: `services/center/internal/governance/jobs_factory.go`)
   动作： 为到期挂载创建 `CONNECTION_TEST` 作业，并关联 mount 或 storage_node；source_domain 使用 `SCHEDULED`，created_by 使用 `SYSTEM`。
   原因： 健康检查需要具备事件流、审计与失败治理，作业体系最适配。
   依赖： 阶段 2 步骤 2。
   风险：低，中等规模的作业量可控；需做去重窗口。

4. **执行器离线治理信号（最小）** (File: `services/center/internal/governance/agent_health.go`)
   动作： 调度器周期性读取 `/agents` 或直接查询 agents 表，对 `heartbeat_timeout` 超过阈值的 agent 生成治理异常（阶段 4 引入“治理来源异常”的写入接口）。
   原因： 切片12范围包含“心跳与健康检查”，agent 是长期运行的关键依赖。
   依赖： 阶段 4（需要治理异常写入能力），可先记录审计不出异常。
   风险：中，避免误报与抖动，需要去重与节流。

验收路径（阶段 2）：

1. 把一个 NAS/Cloud 节点配置为不可用（错误凭据或断网）。
2. 关闭客户端。
3. 等待 heartbeat 到期，应出现后台创建的 CONNECTION_TEST 作业，失败时进入异常/通知。

### 阶段 3: 自动重试策略（作业域退避与预算）

目标：把“手动重试”升级为“自动重试策略”，并可被后台长期运行可靠地驱动。

1. **新增迁移：job_items 增加 next_retry_at** (File: `services/center/migrations/0016_jobs_auto_retry.sql`)
   动作： 为 `job_items` 增加 `next_retry_at TIMESTAMPTZ`，必要时增加 `(job_id, status, next_retry_at)` 索引。
   原因： 当前 `WAITING_RETRY` 没有退避时钟，执行循环会立即再次尝试或只能依赖人工。
   依赖： 现有 `0006_jobs.sql`。
   风险：低，字段新增兼容。

2. **扩展 jobs 领取逻辑：尊重 next_retry_at** (File: `services/center/internal/jobs/service_execution.go`)
   动作： 修改 `claimNextJob` 查询，使 `WAITING_RETRY` 作业只有在存在 `job_items.status='WAITING_RETRY' AND (next_retry_at IS NULL OR next_retry_at <= now())` 时才会被领取。
   原因： 避免“等待重试”作业被频繁领取导致空转。
   依赖： 阶段 3 步骤 1。
   风险：中，需要写出正确的 SQL 条件并有测试覆盖。

3. **扩展 item 选择逻辑：跳过未到期 WAITING_RETRY 子项** (File: `services/center/internal/jobs/service_execution.go`)
   动作： 在执行 job 时，加载 item 时把 `next_retry_at` 读出来，若 item 为 `WAITING_RETRY` 且 `next_retry_at > now` 则跳过并保留 job 状态为 `WAITING_RETRY`。
   原因： 支持单作业内部多个子项的退避，不阻塞其他可执行项。
   依赖： 阶段 3 步骤 1。
   风险：中，需避免“作业永远无法完成”的状态泄漏。

4. **实现自动重试决策与退避计算** (File: `services/center/internal/jobs/auto_retry.go`)
   动作： 新增 `shouldAutoRetry(job, item, err)` 与 `computeBackoff(attemptCount)`：
   - 默认仅对明确可恢复错误启用，优先覆盖 `TRANSFER` 家族与 `CONNECTION_TEST`。
   - 预算由配置给出（环境变量或系统策略，阶段 6 再接 UI）。
   - 退避策略例如：10s, 30s, 2m, 5m（上限封顶）。
   原因： 切片12范围明确包含“自动重试策略”，且需长期运行稳定。
   依赖： 阶段 3 步骤 2 与 步骤 3。
   风险：高，错误分类不当会造成重试风暴或吞掉真实错误；必须有节流与审计。

5. **改造 finishItemFailed：可恢复错误转 WAITING_RETRY** (File: `services/center/internal/jobs/service_execution.go`)
   动作： 当执行失败时，若满足自动重试条件：
   - 不把 item 置为 `FAILED`，而是置为 `WAITING_RETRY` 并写入 `next_retry_at`。
   - job status 置为 `WAITING_RETRY`，并发布 `JOB_ITEM_*` 事件与审计记录（阶段 7）。
   - 超过预算或不可恢复则保持 `FAILED`，走现有 issues/notifications 同步。
   原因： 让自动重试成为第一类行为，不依赖人工点击“重试”。
   依赖： 阶段 3 步骤 4。
   风险：高，会影响既有失败处理逻辑，必须补足集成测试。

6. **保留手动重试入口并与自动重试兼容** (File: `services/center/internal/jobs/service_controls.go`)
   动作： `RetryJob` 需要清理 `next_retry_at`，确保人工重试立即可执行。
   原因： 运维与用户仍需手动介入。
   依赖： 阶段 3 步骤 1。
   风险：低。

验收路径（阶段 3）：

1. 人为制造一次可恢复失败（例如暂时断开 aria2 或模拟 CD2 中断）。
2. 观察子项进入 `WAITING_RETRY`，并在退避后自动再次执行。
3. 超过重试次数后应进入异常中心并产生通知。

### 阶段 4: 文件消失检测 -> 异常/通知治理闭环（非作业失败来源）

目标：基于已存在的 `MISSING` 事实层，生成可治理异常，并通过通知中心提示。

1. **新增 issues 服务入口：同步“缺失事实”治理异常** (File: `services/center/internal/issues/governance_missing.go`)
   动作： 新增 `SyncMissingFacts(ctx)` 或 `SyncMissingReplicas(ctx, limit)`：
   - 查询 `asset_replicas.replica_state='MISSING' AND missing_detected_at IS NOT NULL` 且未被处理过的记录。
   - 对每条缺失事实创建或更新 issue（`source_domain = SYSTEM_GOVERNANCE`，category 可用 `SCAN_PARSE` 或新增更细分类，优先不新增 enum）。
   - dedupe_key 基于 `replica_id` 或 `(mount_id, physical_path)` 保证同一缺失不会反复新增。
   - issue 的 `source_snapshot` 填充 `libraryId/endpointId/path` 等跳转字段，保证通知跳转闭环可用。
   原因： 当前 issues 同步仅面向“作业失败”，缺失事实属于扫描治理，需要独立入口。
   依赖： 现有 issues 表与 helpers 逻辑。
   风险：中，需要谨慎设计 dedupe 与“缺失恢复后如何 resolved/stale”。

2. **缺失恢复处理：从 MISSING 回到 AVAILABLE 时关闭/降级异常** (File: `services/center/internal/issues/governance_missing.go`)
   动作： 设计规则：
   - 若同一 replica 再次被扫描到，indexer 会把 `missing_detected_at` 清空并恢复状态，此时治理同步应将对应 issue 标记为 `RESOLVED` 或进入 `STALE`。
   原因： 长期运行必须避免异常永远堆积。
   依赖： 阶段 4 步骤 1。
   风险：中，避免误关闭与抖动，需引入“稳定窗口”。

3. **调度器周期触发缺失治理同步** (File: `services/center/internal/governance/service.go`)
   动作： 在治理 tick 中加入 `issues.SyncMissingFacts` 调用，并做好节流与错误隔离（单次最多处理 N 条）。
   原因： 让“文件消失检测”在后台持续运行，即便客户端退出也能形成治理信号。
   依赖： 阶段 4 步骤 1。
   风险：低，注意对 DB 压力。

4. **通知中心联动：复用现有 SyncJobNotifications 之外的同步方式** (File: `services/center/internal/notifications/service.go`)
   动作： 为治理来源 issue 提供通知同步入口：
   - 方案A：新增 `SyncGovernanceNotifications(ctx)`，按活动治理 issue upsert 通知。
   - 方案B：在 issues 服务内部创建/更新 issue 后主动调用通知服务 upsert（需要解耦接口）。
   原因： 当前通知同步主要以 jobID 为维度，治理缺失不一定附着 job。
   依赖： 阶段 4 步骤 1。
   风险：中，需要避免破坏现有通知去重规则。

验收路径（阶段 4）：

1. 创建一个资产并完成一次扫描，确认副本存在。
2. 在物理路径上删除该文件，再触发扫描（手动或定时）。
3. `asset_replicas` 应变为 `MISSING`，后台应生成治理异常并推送通知。

### 阶段 5: 深度校验（VERIFY_REPLICA / VERIFY_ASSET）作业化

目标：补齐深度校验的作业执行链路，先覆盖 LOCAL/NAS，CLOUD 给出可解释的治理结果。

1. **注册 VERIFY_REPLICA / VERIFY_ASSET 执行器** (File: `services/center/internal/app/app.go`)
   动作： 为 `jobs.JobIntentVerifyReplica` 与 `jobs.JobIntentVerifyAsset` 注册执行器，最小实现：
   - VERIFY_REPLICA：读取关联副本（`asset_replica_id`），对 LOCAL/NAS 计算 quick hash 或轻量 hash，并更新 `verification_state/quick_hash/hash_verified_at`。
   - VERIFY_ASSET：为一个 asset 下的多个 replicas 创建子项或在执行器内逐个校验（优先选择作业子项模型）。
   原因： 切片12明确包含深度校验，但当前仅有 intent 预留，没有执行路径。
   依赖： `asset_replicas` 字段已存在；作业系统已存在。
   风险：高，涉及 I/O 与性能；必须做限流与取消支持。

2. **实现 hash 计算与限流** (File: `services/center/internal/assets/verify_hash.go`)
   动作： 新增安全的 hash 计算函数：
   - 支持取消（ctx）。
   - 支持大文件流式读取，避免一次性读入内存。
   - 默认使用 SHA-256 或更轻量算法（需明确与文档“轻量 hash”一致，必要时用 BLAKE3 但要考虑依赖引入）。
   原因： 深度校验的核心是可复现的内容校验。
   依赖： 阶段 5 步骤 1。
   风险：高，性能与资源占用是主要风险。

3. **校验失败治理** (File: `services/center/internal/jobs/service_execution.go` 与 `services/center/internal/issues/helpers.go`)
   动作： 校验失败应写入 item 的 `latest_error_code/message`，使 issues 同步能把其分类为 `VERIFY`（现有 helpers 已支持）。
   原因： 深度校验的结果必须可治理、可通知，而不只是 DB 字段变化。
   依赖： 现有 `issues/helpers.go` 已把 `VERIFY_*` 分类为 VERIFY。
   风险：中。

4. **提供“手动触发校验”的后端 API（最小）** (File: `services/center/internal/http/router.go`)
   动作： 新增端点：
   - `POST /api/assets/{id}/verify` 触发 VERIFY_ASSET
   - `POST /api/replicas/{id}/verify` 触发 VERIFY_REPLICA
   返回创建的 jobID，便于任务中心跟踪。
   原因： 深度校验需要可验收路径与可操作入口，哪怕 UI 先不接。
   依赖： 阶段 5 步骤 1。
   风险：中，注意输入校验与权限边界（当前单机单用户可简化，但仍要防注入与空参数）。

验收路径（阶段 5）：

1. 触发一个副本校验作业。
2. 修改文件内容后再次校验，应得到 `FAILED/MISMATCH`，并在异常/通知中可见。

### 阶段 6: 后台清理（从同步删除到后台治理链路）

目标：把删除与清理纳入后台治理与审计，避免中心服务长期运行下“不可追溯的直接删除”。

1. **引入“删除请求 -> 后台作业执行”的路径（最小闭环）** (File: `services/center/internal/assets/mutations.go`)
   动作： 改造 `DeleteEntry`：
   - 不直接物理删除与 `DELETE FROM assets`。
   - 创建 `DELETE_REPLICA` 或 `DELETE_ASSET` 作业，并把 asset/replica/directory 作为 object link 关联。
   - 将 `assets.lifecycle_state` 更新为 `DELETE_PENDING`（或保持现有字段语义，避免新增值时需同步 UI 映射）。
   原因： 切片12范围包含后台清理，且资产域设计建议用生命周期表达删除过程。
   依赖： 作业执行器 `DELETE_*` 已注册，但当前 `ExecuteAssetDeletion` 仍是直接执行路径，需要评估是否复用。
   风险：高，会改变用户删除的即时反馈与行为，需要对 UI 文案与任务中心联动做兼容。

2. **后台清理调度：扫描 DELETE_PENDING 并创建/补齐清理作业** (File: `services/center/internal/governance/cleanup_scheduler.go`)
   动作： 调度器周期检查 `assets.lifecycle_state='DELETE_PENDING'` 或 `asset_replicas.replica_state='DELETE_PENDING'` 的记录，若没有对应运行中作业则创建清理作业。
   原因： 支持客户端退出后的持续清理，避免“用户点了删除但没完成”。
   依赖： 阶段 6 步骤 1。
   风险：中，必须做幂等与去重，否则会重复创建清理作业。

3. **清理完成后的状态收口** (File: `services/center/internal/assets/heavy_ops.go` 与 `services/center/internal/assets/indexer.go`)
   动作： 清理执行器成功后：
   - 更新副本状态为 `DELETED`，资产生命周期进入 `DELETED`。
   - 或者在满足“无剩余副本”时才删资产（参考总体设计文档前提）。
   原因： 后台清理不仅要执行，还要可观测、可审计地收口状态。
   依赖： 阶段 6 步骤 1。
   风险：高，容易出现“数据删了但物理未删”或相反，需要一致性策略与错误恢复。

验收路径（阶段 6）：

1. 在文件中心触发删除。
2. 关闭客户端，观察中心服务仍能执行 `DELETE_*` 作业并最终收口状态。
3. 失败时应进入异常并可重试/确认。

## 6. 测试策略

单元测试（必须）：

1. `services/center/internal/governance`：到期判定、退避计算、去重窗口逻辑（纯函数可测）。
2. `services/center/internal/jobs`：自动重试决策、`next_retry_at` 过滤逻辑（SQL 拼接与行为）。

集成测试（必须，优先 Go integration test）：

1. 定时扫描：插入 mount + mount_runtime.next_scan_at=now，启动 governance tick，断言创建 `SCAN_DIRECTORY` 作业并执行。
2. 自动重试：构造一个会失败的 executor，断言第一次失败进入 WAITING_RETRY 且设置 next_retry_at，到了时间后再次执行，超过预算后进入 FAILED 并产生 issue。
3. 缺失治理：写入 `asset_replicas` MISSING 事实，触发 SyncMissingFacts，断言 issues/notifications 记录被 upsert。

端到端验收（人工）：

1. 关闭客户端后，中心服务仍能持续创建与执行计划任务（扫描/健康检查）。
2. 网络波动下，任务能自动重试并最终收口为成功或可治理异常。
3. 文件被外部删除后，系统产生缺失异常与通知，并在文件恢复后自动降级/关闭异常。

## 7. 风险与缓解

1. **风险：调度器重复创建作业导致作业风暴**
   Mitigation: DB 领取锁（`FOR UPDATE SKIP LOCKED`）、去重窗口、审计与限速。

2. **风险：自动重试吞掉真实错误或造成长期无意义重试**
   Mitigation: 仅对明确可恢复错误启用；预算与退避封顶；达到预算后强制进入异常并提示处理建议。

3. **风险：深度校验占用大量 I/O 导致系统卡顿**
   Mitigation: 限流并发；大文件流式 hash；可取消；低性能模式策略（阶段 6/后续策略化）。

4. **风险：后台清理改变现有删除语义，导致用户困惑或数据不一致**
   Mitigation: 保留清晰的任务中心可见路径；失败可重试；在 UI 文案中明确“进入后台清理队列”；上线前补足回归测试。

5. **风险：治理异常与通知去重不当造成重复提醒**
   Mitigation: dedupe_key 设计以事实主键为核心；通知侧保持同一问题只保留一条活动通知。

## 8. 完成标志

切片12完成的判定清单（全部满足）：

1. 中心服务启动后自动运行治理调度器，客户端退出不影响后台任务运行。
2. 至少一种挂载支持定时扫描闭环：到期自动创建扫描作业并执行，扫描历史可追溯。
3. 心跳/健康检查具备后台闭环：到期自动创建连接测试作业，失败产生异常与通知。
4. 自动重试策略生效：可恢复失败会进入 WAITING_RETRY 并退避后自动再试，超过预算进入异常。
5. 文件缺失事实会被治理：MISSING 事实能生成治理异常与通知，恢复后能自动收口。
6. 深度校验至少覆盖 LOCAL/NAS：可触发 VERIFY 作业并回写校验字段，失败可治理。
7. 删除/清理具备后台链路：删除请求可转为后台作业并可追踪，不再只有同步直接物理删除路径。
8. 后台治理行为具备审计记录，可通过 API 查询用于排障。
