## 文档说明

- 关联计划文档：[统一文件管理系统-网盘同步任务状态规划与任务拆分方案.md](/B:/new_project/mare/docs/实现路径/统一文件管理系统-网盘同步任务状态规划与任务拆分方案.md)
- 目标：为后续 AI 实现和人工联调提供一份可直接执行的真实客户端测试清单，不只覆盖进度显示，还覆盖恢复与韧性问题。

## 1. 测试目标

本轮测试的目标是确认以下结论同时成立：

- `CloudDrive2` 上传不会在预处理或排队阶段错误显示传输进度
- `aria2` 下载不会再因为错误兜底导致进度假满到 `100%`
- `CD2` 上传通道断线不会立刻把本地会话全部判死
- 上传任务在系统长时间崩溃或中心服务长时间停机后，即使遇到 `Upload session not found` 也能在同一任务内自动恢复并继续断点续传
- `aria2` 恢复时不会继续复用错误态旧任务
- 传输进度高频刷新下，系统不会因为每秒重事务而明显卡顿
- 任务中心中的状态、阶段、进度、速度、ETA 与真实链路一致
- 文件中心中的端点状态投影与任务中心结果一致
- 暂停、继续、取消、重试、重启恢复不会破坏上述规则
- 真实 `20MB` 文件能够完成上传和下载

## 2. 测试范围

### 2.1 本轮覆盖

- 文件中心：
  - 上传到 `115`
  - 从 `115` 下载
  - 删除 `115` 副本后的状态投影
- 任务中心：
  - 一级任务与二级任务显示
  - 阶段、进度、速度、ETA 展示
  - 暂停、继续、取消、重试
  - 重启恢复
  - 长时间停机后的同任务自动恢复
- 后端链路：
  - `CD2_REMOTE_UPLOAD`
  - `ARIA2`
  - `Job / JobItem / ExternalTask / EndpointProjection`

### 2.2 本轮不覆盖

- CD2 官方离线下载业务编排
- `CLOUD -> CLOUD` 正式同步
- 与本轮问题无直接关系的普通 `COPY` 链路重构
- 导入中心、异常中心、通知中心的专项改造

## 3. 环境前提

### 3.1 基础环境

- 客户端可正常打开文件中心与任务中心
- 中心服务已启动
- `CloudDrive2` 网关配置有效
- `aria2` 运行组件在线
- 已存在可用的 `115` 云节点
- 文件中心中存在可用于上传与下载的测试素材

### 3.2 真实联调前提

- 准备一个可写入的 `115` 目标目录
- 准备一个可读写的本地或 NAS 目录
- 至少准备两个真实测试文件：
  - 小文件：便于快速验证阶段切换
  - `20MB` 文件：用于真实传输完成验证
- 所有上传验证文件必须为新生成素材，不能复用已上传过的文件；如果出现秒传，必须重新生成新文件后重测
- 长时间崩溃恢复场景需额外准备一个不低于 `384MB` 的新生成大文件，用于确保进入真实上传阶段且不会秒传
- 能够主动触发中心服务重启，用于恢复场景验证
- 能够控制系统崩溃或中心服务长时间停机时长，并确认恢复前旧 `CD2` 上传会话已经实际失效

## 5. 自动化验证项

### 5.1 后端自动化

- `P0` `aria2` 恢复状态边界
  - 覆盖 `active / waiting / paused / complete / error / removed`
  - 期望 `error / removed` 不再被复用
- `P0` 传输进度节流
  - 高频回调下不再每次都落库
  - 首次上报、超过 5 秒、超过 5% 进度增量时才持久化
- `P0` `CD2` 通道断线容错
  - `RemoteUploadChannel` 断线后不立即关闭本地上传会话
  - 等待侧仍允许后续重建通道
- `P1` 云副本不能作为上传源
  - `CLOUD -> CLOUD` 在当前实现下不得被视作已支持上传链路

### 5.2 前端自动化

- `P0` 上传前置阶段展示
  - `WaitforPreprocessing / Preprocessing / Inqueue` 只显示阶段，不显示误导性的进度推进
- `P0` 下载失败阶段展示
  - `aria2 error` 显示失败位置、外部状态和原始错误
- `P1` 详情、列表、展开子项口径一致
  - 同一任务在 3 处视图中的阶段解释一致

- 期望：

  - 速度消失、进度冻结
  - 继续后恢复到正确阶段

### 7.3 上传进行中服务重启

- 优先级：`P0`

### 7.5 上传进行中长时间崩溃恢复，覆盖 `Upload session not found`

- 优先级：`P0`
- 当前结果（2026-04-13）：已通过
- 真实记录：
  - 文件：`codex_ui_random_768mb_20260413_214000.bin`
  - 大小：`768MB`
  - 资产：`asset-4e940ee84c1dc956`
  - 作业：`job-id-1776087433916843100-79d6735c`
  - 停机前进度：`301,989,888 / 805,306,368`
  - 结果：恢复后同一任务自动续传，跨过旧失败点并最终 `COMPLETED`
- 步骤：
  1. 在真实客户端中选择一个新生成且未上传过的大文件，发起 `LOCAL/NAS -> 115` 上传任务
  2. 等待任务进入真实上传阶段，记录 `jobId`、`jobItemId`、`externalTaskId`、`bytesDone`、`bytesTotal`
  3. 模拟系统崩溃或让中心服务长时间停机
  4. 在停机期间持续等待，直到确认旧 `CD2` 上传会话已经失效，而不是仅等待固定十分钟
  5. 恢复系统或中心服务，再恢复真实客户端
  6. 观察同一任务是否自动恢复，期间不允许依赖用户手动点击“重试”
  7. 若恢复过程中出现 `Upload session not found`，继续观察系统是否会在同一任务内自动完成旧会话接管失败后的清理、重建会话与续传
  8. 最终检查任务中心状态、文件中心端点状态以及 `115` 云端文件是否真实存在且大小正确
- 期望：
  - 停机恢复后，同一上传任务不会因为 `Upload session not found` 直接永久失败
  - 恢复后不需要用户手动重试
  - 系统会在同一任务内自动恢复上传，并继续断点续传
  - 恢复后的进度能够跨过停机前记录的失败点
  - 最终任务状态为 `COMPLETED`
  - 文件中心目标端点最终显示 `已同步`
  - `115` 云端文件真实存在且大小正确
