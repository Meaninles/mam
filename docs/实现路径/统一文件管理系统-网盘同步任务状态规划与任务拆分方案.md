# 统一文件管理系统-网盘同步任务状态规划与任务拆分方案

## 文档说明

- 更新时间：2026-04-13
- 适用范围：`client`、`services/center`、`shared/contracts`
- 目标：为后续实现提供一份可直接执行的网盘同步任务状态规划与任务拆分文档，并把本对话已确认的问题一并纳入约束。
- 约束：
  - 所有“网盘同步任务”统一按真实代码现状规划，不混入未落地能力
  - 任务拆分最多 4 个，每个任务都必须可独立实现、验证和回归
  - 本文档覆盖的不只是进度显示问题，还包括上传通道断线、错误态恢复、写库节流和大文件 I/O 风险

## 1. 当前真实链路

当前正式存在的同步链路只有三类：

1. `LOCAL/NAS -> CLOUD`
   - 通过 `CloudDrive2 remote upload` 上传
2. `CLOUD -> LOCAL/NAS`
   - 由 CD2 解析下载地址，再交给 `aria2` 下载
3. `LOCAL/NAS -> LOCAL/NAS`
   - 中心服务直接复制

当前明确不存在：

- CD2 官方离线下载任务的正式业务编排
- Aria2 下载后再把本地路径交给 CD2 接力上传
- `CLOUD -> CLOUD` 正式同步

## 2. 本轮需要统一解决的问题

本轮不再只把问题理解成“进度条不准”，而是按四类根因统一收口：

### 2.1 指标与阶段边界不严谨

- 下载链路会误把不可靠数据当作已下载字节
- 上传链路会在还没进入真实传输阶段时展示进度或速度
- 一级任务、二级任务和文件中心端点投影的状态语义容易混用

### 2.2 上传通道脆弱

- `CloudDrive2 RemoteUploadChannel` 一旦断开，本地协调层会过早终止会话
- 短时网络抖动不应直接等价于任务失败

### 2.3 恢复逻辑不完整

- Aria2 若恢复到错误态旧任务，会立刻再次失败
- CD2 当前主要依赖 `uploadId + deviceId + AttachUpload()`，恢复元数据还不够结构化

### 2.4 高负载下韧性不足

- 传输进度会高频写库，未来并发上升后会放大数据库压力
- 同一大文件在上传读流和多种哈希计算之间存在读盘竞争风险

## 3. 统一状态模型

### 3.1 分层对象

| 层级 | 对象 | 职责 |
| --- | --- | --- |
| 1 | `Job` | 调度、批量操作、聚合结果 |
| 2 | `JobItem` | 单文件 / 单目录 / 单副本执行 |
| 3 | `ExternalTask` | `CD2 / aria2` 外部任务状态与恢复 |
| 4 | `EndpointProjection` | 文件中心端点结果态投影 |

### 3.2 内部状态

`Job` 使用：

- `PENDING`
- `QUEUED`
- `RUNNING`
- `PAUSED`
- `WAITING_RETRY`
- `FAILED`
- `PARTIAL_SUCCESS`
- `COMPLETED`
- `CANCELED`

`JobItem` 使用：

- `PENDING`
- `QUEUED`
- `RUNNING`
- `PAUSED`
- `WAITING_RETRY`
- `FAILED`
- `COMPLETED`
- `SKIPPED`
- `CANCELED`

### 3.3 外部状态

`CD2_REMOTE_UPLOAD`：

- `WaitforPreprocessing`
- `Preprocessing`
- `Inqueue`
- `Transfer`
- `Pause`
- `Finish`
- `Cancelled`
- `Skipped`
- `Error`
- `FatalError`

`ARIA2`：

- `waiting`
- `active`
- `paused`
- `complete`
- `removed`
- `error`

### 3.4 端点投影状态

文件中心端点状态只允许使用：

- `未同步`
- `同步中`
- `部分同步`
- `已同步`

约束：

- 文件级不出现 `部分同步`
- 目录级可出现 `部分同步`
- 有运行中同步任务时优先显示 `同步中`

## 4. 指标门控规则

状态与指标必须绑定，不允许只要有字节字段就直接展示进度、速度和 ETA。

### 4.1 下载链路

| 引擎 | 外部阶段 | 进度 | 速度 | ETA | 说明 |
| --- | --- | --- | --- | --- | --- |
| `ARIA2` | `waiting` | 否 | 否 | 否 | 尚未进入真实下载 |
| `ARIA2` | `active` | 是 | 是 | 是 | 唯一可信下载阶段 |
| `ARIA2` | `paused` | 是，冻结 | 否 | 否 | 保留进度，不显示运行指标 |
| `ARIA2` | `complete` | 是，100% | 否 | 否 | 完成态 |
| `ARIA2` | `removed` | 否 | 否 | 否 | 取消态 |
| `ARIA2` | `error` | 否 | 否 | 否 | 失败态 |

下载进度数据源优先级：

1. `completedLength`
2. `totalLength`
3. 禁止默认用目标文件大小兜底

约束：

- 默认禁止使用目标文件大小作为通用 `bytesDone`
- 若无法安全判断，则宁可展示保守进度，也不能展示虚高进度

### 4.2 上传链路

| 引擎 | 外部阶段 | 进度 | 速度 | ETA | 说明 |
| --- | --- | --- | --- | --- | --- |
| `CD2_REMOTE_UPLOAD` | `WaitforPreprocessing` | 否 | 否 | 否 | 前置阶段 |
| `CD2_REMOTE_UPLOAD` | `Preprocessing` | 否 | 否 | 否 | 前置阶段 |
| `CD2_REMOTE_UPLOAD` | `Inqueue` | 否 | 否 | 否 | 已排队但未进入真实上传 |
| `CD2_REMOTE_UPLOAD` | `Transfer` | 是 | 弱显示 | 否 | 唯一可信上传阶段 |
| `CD2_REMOTE_UPLOAD` | `Pause` | 是，冻结 | 否 | 否 | 暂停态 |
| `CD2_REMOTE_UPLOAD` | `Finish` | 是，100% | 否 | 否 | 完成态 |
| `CD2_REMOTE_UPLOAD` | `Cancelled` | 否 | 否 | 否 | 取消态 |
| `CD2_REMOTE_UPLOAD` | `Skipped` | 否 | 否 | 否 | 跳过态 |
| `CD2_REMOTE_UPLOAD` | `Error / FatalError` | 否 | 否 | 否 | 失败态 |

约束：

- 只在 `Transfer` 阶段启用进度
- `globalBytesPerSecond` 只能作为总上传速度语义使用
- 当前版本不承诺单任务上传 ETA

## 5. 恢复规则

### 5.1 Job / JobItem

- 重启后先把内部运行态恢复为 `QUEUED`
- `PAUSED` 不允许自动恢复到 `RUNNING`
- `COMPLETED / FAILED / CANCELED / SKIPPED` 不允许被误恢复

### 5.2 Aria2

- 只允许复用 `active / waiting / paused / complete`
- 恢复到 `error / removed` 旧任务时，不再直接复用
- 对错误态任务，后续需要补“刷新 URL 或重新入队”能力

### 5.3 CD2

- 当前允许依赖 `uploadId + deviceId + AttachUpload()` 继续挂接
- 通道断开不应直接把本地所有上传会话判死
- 后续需要补更结构化的恢复元数据

## 6. 本轮任务拆分

总任务数限定为 4 个，每个任务都必须独立交付。

### 任务 1：收口指标门控与进度写库节流

目标：

- 把上传/下载指标门控规则固化到实现
- 降低高频进度回写带来的数据库压力

范围：

- `services/center/internal/assets/heavy_ops_cloud.go`
- `services/center/internal/jobs/service_external_tasks.go`
- 相关单测 / 集成测试

必须完成：

- 只在可信阶段展示和持久化有效进度
- 增加节流，避免每秒重事务落库
- 保证一级任务聚合与二级任务指标口径一致

验收标准：

- 下载未完成时不再直接 100%
- 上传在 `Transfer` 前不再推进进度
- 高频回调下落库次数明显降低

### 任务 2：修复 Aria2 错误态恢复边界

目标：

- 不再复用错误态或已移除的旧下载任务

范围：

- `services/center/internal/integration/downloader_aria2.go`
- `services/center/internal/assets/heavy_ops_cloud.go`
- `services/center/internal/integration/downloader_aria2_test.go`

必须完成：

- 明确可复用状态集合
- 错误态旧任务不得直接进入等待逻辑
- 为后续 URL 刷新 / 重新入队保留清晰边界

验收标准：

- 恢复流程不再直接卡死在旧错误任务上
- 单测覆盖 `active / waiting / paused / complete / error / removed`

### 任务 3：修复 CD2 上传通道断线容错

目标：

- `RemoteUploadChannel` 断线时不再本地全量失败

范围：

- `services/center/internal/integration/provider_cd2_115.go`
- `services/center/internal/integration/provider_cd2_115_test.go`

必须完成：

- gRPC 流断开时重置通道
- 保留本地上传会话
- 等待侧允许尝试重建上传通道

验收标准：

- 通道断开后上传会话不会立即进入本地失败终态
- 单测覆盖“通道断开但会话仍保持等待”

### 任务 4：补齐客户端验证口径与真实客户端测试清单

目标：

- 收口任务中心、文件中心和测试文档中的展示与验证口径

范围：

- `docs/实现路径/统一文件管理系统-网盘同步任务真实客户端测试用例清单.md`
- 任务中心相关前端测试

必须完成：

- 文档覆盖本轮新增的恢复与韧性问题
- 自动化测试与人工联调用例口径一致
- 真实客户端测试清单能直接指导 20MB 文件上传/下载、暂停/恢复、重启恢复验证

验收标准：

- 文档与代码口径一致
- 自动化测试和手工验证步骤都可执行

## 7. 实施顺序

严格按以下顺序实施：

1. 任务 1
2. 任务 2
3. 任务 3
4. 任务 4

原因：

- 任务 1 定义指标边界，是后续任务的基础
- 任务 2 与任务 3 分别收口下载恢复和上传韧性，耦合较低
- 任务 4 最后统一验证口径

## 8. 完成标志

当以下条件同时成立时，本轮规划视为完成：

- 下载任务不再出现“未下完但进度 100%”
- 上传任务不再在前置阶段显示错误的传输进度
- CD2 通道断线不会立刻本地全量失败
- Aria2 恢复不再复用错误态旧任务
- 传输进度回写已具备节流能力
- 文件中心与任务中心对同一任务的状态解释一致
- 测试用例文档已覆盖 20MB 真实客户端验证、暂停/继续、取消、重启恢复与回归检查
