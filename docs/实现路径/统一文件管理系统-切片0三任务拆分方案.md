## 1. 切片 0 的重新定义

切片 0 不再按“先把客户端接一个中心服务”理解，而是按“三段式系统同时起盘”理解。

本切片的目标是把以下 4 个运行单元真正建立起来并跑通最小真实联通链路：

1. PostgreSQL
2. Go 中心服务
3. Go 本地执行器
4. Tauri 客户端

切片 0 完成后，项目应从“只有桌面原型客户端”升级为“客户端 + 中心服务 + 本地执行器 + PostgreSQL”的最小运行系统。

## 2. 切片 0 的硬性要求

### 2.1 必做项

- 建立三段式工程目录与启动骨架
- 中心服务具备最小 HTTP 服务能力
- 中心服务具备配置读取、基础日志、统一错误模型、统一响应约定
- 中心服务可连接 PostgreSQL，并具备最小 migration 能力
- 本地执行器具备独立启动、注册、心跳能力
- 客户端具备最小系统状态接线能力
- 冻结切片 0 需要的最小共享 contracts / DTO / 事件信封
- 具备开发态最小运行编排能力

### 2.2 明确不做

- 不做历史数据迁移
- 不做旧数据兼容
- 不做 backfill、双写、导入旧库、兼容层
- 不做存储节点、文件中心、任务中心、异常中心、通知中心、导入中心的真实业务迁移
- 不做 115 / CloudDrive2 / aria2 集成
- 不做本地执行器的设备探测、扫描、导入执行、传输执行

### 2.3 当前开发环境数据处理原则

当前仓库中的以下数据都视为无保留价值的开发态原型数据：

- `sql.js` 文件中心本地库
- `localStorage` 中的客户端状态
- Tauri Rust 侧的存储节点过渡 SQLite

切片 0 不为这些数据编写任何迁移或兼容逻辑。后续进入真实后端切换时，允许直接重置。

## 3. 切片 0 的成立标准

切片 0 必须同时满足以下条件，才算完成：

- 中心服务是独立可运行的 Go 服务
- 本地执行器是独立可运行的 Go 进程
- PostgreSQL 可被中心服务连接，baseline migration 可执行
- 中心服务暴露健康检查接口与系统运行状态接口
- 本地执行器可向中心服务注册并持续发送心跳
- 客户端可展示中心服务与本地执行器状态
- 仓库中已有最小共享 contracts 的归属地
- 现有业务页仍保持原型运行，不提前服务化迁移

## 5. 三个任务的拆分方式

本切片按下面 3 个任务拆分，每个任务对应一个主要实现主体。理想情况下，后续只需围绕这 3 个任务分别推进一次对话即可完成切片 0。

### 任务 1：中心服务任务

**任务目标**

搭起 Go 中心服务工程骨架，并提供切片 0 所需的最小权威后端能力。

**任务范围**

- 建立 `services/center` Go 工程
- 建立服务启动入口与内部基础分层
- 建立配置读取
- 建立结构化日志
- 建立统一错误模型与统一响应 envelope
- 建立 PostgreSQL 连接能力
- 建立 baseline migration 机制
- 建立最小执行器注册与心跳接收能力
- 建立系统运行状态查询能力
- 冻结切片 0 所需最小共享 contracts，作为其他两个任务的前置输入

**建议最小接口**

- `GET /healthz`
- `GET /readyz`
- `POST /agent/register`
- `POST /agent/heartbeat`
- `GET /api/runtime/status`

**本任务必须产出的共享契约**

- 通用成功响应 DTO
- 通用错误响应 DTO
- 健康检查 DTO
- 就绪检查 DTO
- 系统运行状态 DTO
- 执行器注册请求/响应 DTO
- 执行器心跳请求/响应 DTO
- 最小事件信封 DTO

**数据库最小要求**

- 选定并固定 migration 机制
- 支持 baseline migration
- 启动时可校验 migration 是否就绪
- 开发环境支持 reset / recreate
- 不承担历史数据迁移

**数据库对象最小范围**

- migration 版本表
- 执行器最小注册/运行状态表，或等价持久化对象

**本任务完成定义**

- 中心服务独立启动成功
- 成功连接 PostgreSQL
- baseline migration 可执行
- `/healthz` 与 `/readyz` 可用
- 可接收执行器注册与心跳
- `/api/runtime/status` 可返回中心服务、数据库、迁移、执行器摘要状态

**当前已实现内容**

- 已实现最小目录骨架与启动入口
- 已实现 PostgreSQL 连接与 baseline migration
- 已实现最小接口：
  - `GET /healthz`
  - `GET /readyz`
  - `GET /api/runtime/status`
  - `POST /agent/register`
  - `POST /agent/heartbeat`
- 已实现最小 CORS 支持，客户端可直接请求运行态接口

**本任务明确不做**

- 资产域、存储域、作业域、异常域、导入域、通知域业务 API
- SSE 业务事件流
- 文件中心、任务中心、存储节点等业务页面的数据服务

### 任务 2：本地执行器任务

**任务目标**

搭起 Go 本地执行器工程骨架，使其成为一个真实可运行、可被中心服务纳管的独立进程。

**任务范围**

- 建立 `services/agent` Go 工程
- 建立执行器启动入口与内部基础分层
- 建立配置读取
- 建立结构化日志
- 建立执行器身份生成或读取机制
- 实现启动注册
- 实现周期性心跳
- 实现基础重试、退避、重复注册处理
- 上报最小运行摘要信息

**建议最小运行摘要**

- `agent_id`
- `version`
- `hostname`
- `platform`
- `pid`
- `started_at`
- `mode`

**本任务完成定义**

- 执行器独立启动成功
- 可调用中心服务注册接口
- 可持续发送心跳
- 中心服务可识别执行器在线状态
- 中心服务中断时，执行器可进入合理的重试状态

**当前已实现内容**

- 已实现 `services/agent` 独立 Go 工程
- 已实现配置读取与校验
- 已实现本地身份持久化
- 已实现最小运行摘要
- 已实现启动注册、周期性心跳与基础重试
- 已提供开发启动脚本 `scripts/dev/start-agent.ps1`

**本任务明确不做**

- 设备探测
- 文件扫描
- 导入执行
- 本机文件传输
- 作业领取
- 结果回写
- 外设插拔监听

### 任务 3：客户端任务

**任务目标**

在不提前迁移业务页面的前提下，为现有 Tauri 客户端增加切片 0 所需的最小系统状态接线。

**任务范围**

- 新增客户端运行时配置模块
- 新增系统状态请求模块
- 拉取中心服务就绪状态与系统运行状态
- 在顶层壳层展示系统状态
- 仅接线，不重构现有业务编排

**客户端应展示的最小状态**

- 中心服务：不可达 / 已启动但未就绪 / 可用
- 本地执行器：未注册 / 在线 / 心跳超时

**当前最终落地形态**

- 状态灯位于页头右上区域、导入入口左侧
- 常驻 UI 仅为一个状态灯，不展示常驻文字
- hover 时显示状态描述
- 判定规则固定为：
  - 中心服务异常：红灯
  - 中心服务正常、本地执行器异常：黄灯
  - 中心服务和本地执行器都正常：绿灯

**推荐落点**

- `client/src/lib/runtimeConfig.ts`
- `client/src/lib/systemRuntimeApi.ts`
- `client/src/App.tsx`

**本任务完成定义**

- 客户端可探测中心服务状态
- 客户端可读取执行器摘要状态
- 顶层壳层可稳定展示系统状态
- 即使中心服务或执行器中断，客户端也能明确展示退化状态

**当前已实现内容**

- 已实现运行时配置模块
- 已实现系统状态请求模块
- 已在 `App.tsx` 顶层壳层完成状态灯接线
- 已补齐客户端对应测试与构建验证

**本任务明确不做**

- 不迁移 [App.tsx](/B:/new_project/mare/client/src/App.tsx) 现有业务编排逻辑
- 不迁移 [clientState.ts](/B:/new_project/mare/client/src/lib/clientState.ts)
- 不迁移 `fileCenterApi.ts`
- 不迁移 [storageNodesApi.ts](/B:/new_project/mare/client/src/lib/storageNodesApi.ts)
- 不迁移文件中心、任务中心、异常中心、通知中心、导入中心、存储节点页面的数据源
- 不改 [lib.rs](/B:/new_project/mare/client/src-tauri/src/lib.rs) 现有存储节点桥接链路

## 6. 三个任务的依赖关系

### 6.1 推荐顺序

推荐按下面顺序推进：

1. 中心服务任务
2. 本地执行器任务
3. 客户端任务

### 6.2 原因

- 中心服务任务先完成，才能冻结最小 contracts、接口和数据库 baseline
- 本地执行器任务依赖中心服务的注册与心跳接口
- 客户端任务虽然也依赖中心服务接口，但最好在执行器在线状态可被中心服务汇总后再接，能一次完成端到端可见闭环

### 6.3 可并行部分

- 客户端的运行时配置模块可以在中心服务接口 shape 基本冻结后并行准备
- 本地执行器骨架可以与中心服务脚手架并行创建，但注册/心跳落地必须依赖中心服务接口

## 7. 三个任务的统一验收路径

切片 0 的人工验收路径统一如下：

1. 启动 PostgreSQL
2. 执行 baseline migration
3. 启动中心服务
4. 启动本地执行器
5. 启动 Tauri 客户端
6. 客户端显示中心服务可用、本地执行器在线
7. 停掉本地执行器，客户端显示执行器离线或超时
8. 停掉中心服务，客户端显示服务不可用

当前推荐优先使用以下开发入口完成切片 0 联调：

1. `cnpm run one-link-start`
2. `cnpm run one-link-start:with-db`

前者保留数据库状态，后者会先重启开发态 PostgreSQL，再按顺序拉起中心服务、本地执行器和客户端。

## 8. 三个任务的统一自动化验证要求

### 中心服务侧

- 配置缺失启动失败测试
- 数据库连接失败测试
- baseline migration 成功与未就绪测试
- `/healthz`、`/readyz`、`/api/runtime/status` 响应测试
- 注册与心跳接口测试

### 本地执行器侧

- 注册成功测试
- 心跳成功测试
- 中心服务不可达重试测试
- 重复注册测试
- 心跳超时相关状态测试

### 客户端侧

- 系统状态 API 请求成功测试
- 超时、失败、非 200 响应测试
- 顶层壳层状态展示测试

## 9. 三个任务都必须遵守的边界

- 不为开发态旧数据编写迁移
- 不增加新的 mock、fallback 或降级实现作为正式路径
- 不在切片 0 提前实现切片 1 及以后的业务能力
- 不把本地执行器做成“空壳但假装支持业务任务”的半成品
- 不把客户端任务扩大成页面级服务化改造

## 10. 后续对话建议

为了尽量在 3 次对话内完成切片 0，建议后续按下面方式推进：

1. 第一次对话：只做中心服务任务
2. 第二次对话：只做本地执行器任务
3. 第三次对话：只做客户端任务，并完成三者联调验证

这样拆分后，每次实现都有明确边界、清晰输入和独立验收点，也更符合切片 0“先立底座、再跑联通”的目标。
